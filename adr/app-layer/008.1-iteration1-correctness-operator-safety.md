## I1.1 Pool backpressure

**Problem.** `submit()` and `route()` discard `pool.add()`'s boolean; a full
pool silently drops the message while the REST caller gets `202` + id. Pool
capacity is hardcoded (`new AppMsgPool(10_000)`), and local submit relays to
peers *before* the local pool add.

**Design.**
- Config: `yano.app-chain.pool.max-messages` (suffix `pool.max-messages`,
  builder field `poolMaxMessages`, default 10_000, must be ≥ `block.max-messages`).
- New `PoolFullException extends IllegalStateException` in
  `core-api api.appchain` (message includes chain id + capacity). REST maps it
  to **429** with the standard `{"error": ...}` body *before* the generic
  `IllegalStateException → 503` mapping.
- `submit()` order becomes: build + sign → **pool.add** → (throw
  `PoolFullException` if rejected) → relay to peers. A message the node cannot
  hold is never diffused half-way.
- `route()` (inbound gossip): rejected `pool.add` increments a per-chain
  `droppedPoolFull` counter (no exception — gossip is best-effort by design;
  the sender's own node already accepted it).
- Drop accounting by reason, exposed in `status().drops` and metrics (I1.8):
  `pool_full`, `expired`, `bad_auth`, `not_member`, `stale_seq` (I1.2).
  (`duplicate` remains its own existing counter.)

**Acceptance.** Full-pool submit → 429, message not relayed; inbound overflow
visible in `status()` and `/q/metrics`; capacity configurable per chain.

## I1.2 Sender-seq light enforcement

**Problem.** `sender-seq` is signed and carried but never validated; replay
protection rests on a 100k sliding message-id window (permanent dedup covers
only *finalized* ids). The in-process seq counter resets to 0 on restart.

**Semantics decided (parent §6.2).** `sender-seq` = **replay rejection + gap
observability**, not ordering. Enforced rule: a message is stale if
`senderSeq ≤ lastFinalizedSeq(sender)`. Gaps are allowed (out-of-order
diffusion is normal; unsequenced messages may expire).

**Design.**
- Ledger: `commitBlock()` additionally writes `sender_seq/<senderHex> → uint`
  (max seq per sender in the block) inside the same atomic WriteBatch —
  rollback-safe by construction (app ledger never rolls back after APP_FINAL).
- Producer side: the subsystem's own seq counter initializes from
  `sender_seq/<own key>` at start (fixes restart reset).
- Admission (both `submit()` local and `route()` inbound): reject stale
  messages (submit → `IllegalArgumentException` → 400; inbound → drop with
  reason `stale_seq`).
- Proposer: `selectMessages()` filters stale messages (pool may hold a message
  finalized-by-seq elsewhere between admission and proposal).
- Follower in-block enforcement (consensus-visible!) behind
  `yano.app-chain.message.enforce-sender-seq` (default **false** this
  release): when true, a proposal containing a message with
  `seq ≤ lastFinalizedSeq(sender)` as of the parent block is rejected (no
  vote), and catch-up applies the same check per block using the seq state it
  re-derives. Default-off keeps old ledgers/catch-up compatible; testkit and
  devnet regressions run with it on; flip the default one release later.
- Docs state exactly what is and isn't guaranteed.

**Acceptance.** Replayed finalized message rejected at admission on every node;
restart does not reset the node's own seq; enforcement flag on → follower
rejects a stale-seq proposal (adversarial test); flag off → legacy behavior.

## I1.3 Follower verification of `l1-ref`

**Problem.** Followers co-sign whatever `l1Slot/l1BlockHash` the proposer puts
in a block; catch-up doesn't check it either. Additionally the L1 point feed is
silent-degrading: `eventBus == null` or `l1.stability-depth = 0` → every block
carries `l1Slot = 0` with no warning.

**Design.**
- Follower check in `handleProposal()` when `stability-depth > 0` and the
  proposal carries `l1Slot > 0`:
  1. **Monotonicity**: `l1Slot ≥ previous block's l1Slot` (previous from
     ledger tip), else reject.
  2. **Own-view match**: the `(slot, hash)` pair must appear in the follower's
     own observed stable window (`recentL1Points`, capacity ≥ stability window)
     — i.e. the follower has that exact point at depth ≥ `stability-depth`.
     Known slot with a different hash → **reject immediately** (fail-closed).
  3. **Deferral**: point beyond the follower's current L1 view (follower lags)
     → don't vote yet; re-evaluate the held proposal for up to
     `l1RefWaitMs = max(2 × block.interval-ms, 10s)`, then give up (no vote;
     the proposer re-proposes). Deferrals are counted
     (`status().l1RefDeferrals`).
  4. Fallback when the slot is older than the follower's window (e.g. fresh
     restart, empty deque): consult an optional `L1PointLookup` (chain-state
     backed, wired by the runtime); if unavailable, accept on monotonicity
     alone and log at WARN (documented as the restart-window caveat).
- Same checks in `applyCertifiedBlock()` (catch-up), with the fallback rule —
  historical blocks may predate the local window; monotonicity is always
  enforceable, own-view match is enforced when local L1 history covers the ref.
- **Fail-fast startup** (parent finding 7): if `anchor.enabled` or
  `stability-depth > 0` but no L1 point feed is wired (no event bus), the
  subsystem refuses to start with a clear message. Library mode without refs
  logs one explicit WARN ("blocks will carry l1Slot=0").
- A proposal with `l1Slot == 0` while `stability-depth > 0` is rejected
  (a configured chain must not silently lose its L1 linkage).

**Acceptance.** Adversarial tests: fabricated hash → no vote; backwards slot →
no vote; follower briefly lagging → vote after deferral; misconfigured
chain (depth>0, no feed) → startup failure.

## I1.4 stdlib config wiring + balances fix

**Problem.** stdlib providers only implement no-arg `create()` so no stdlib
machine can receive settings; the `balances` minter constructor is dead code —
**any member can mint**. `removeMember` normalizes keys inconsistently.

**Design.**
- Settings namespace for machines (flows through the existing
  `pluginSettings`/`AppStateMachineContext` seam, same as `zk.*`):
  `yano.app-chain.machines.<machine-id>.<key>`.
- All four stdlib providers implement `create(AppStateMachineContext)`:
  - `balances`: `machines.balances.minter` (64-hex member key). Set → only that
    sender's mint commands apply; unset → current behavior (any member),
    explicitly documented as the open mode.
  - `kv-registry`: `machines.kv-registry.value-format = raw | cbor | utf8`
    (default `raw`). Non-conforming values are deterministic no-ops (same
    posture as ownership violations). Covers ADR-006's pending "optional
    schema check" without new dependencies.
  - `approvals`, `doc-trail`: context-aware constructors, no settings yet.
- `removeMember` uses the same `normalizeMemberKeys` path as `addMember`.

**Acceptance.** Minter-restricted chain: non-minter mint = no-op, minter mint
applies (root-visible); value-format=cbor rejects malformed CBOR values;
existing stdlib tests still green.

## I1.5 Anchor tx production-grade construction

**Problem.** Fixed 300k fee, first-fit single input, `change = input − fee`
with no min-UTxO guard, no validity interval, rollback rewind by a fixed
`everyBlocks` step.

**Design.**
- **Fee**: linear fee from current protocol parameters
  (`minFeeA × size + minFeeB`) via a `Supplier<ProtocolParams>` the runtime
  wires from the node's ledger state; two-pass build (draft → size → fee →
  re-sign). Fallback when params unavailable: configured
  `anchor.fallback-fee-lovelace` (default 300_000) — devnet stays simple.
- **Inputs/change**: accumulate pure-ADA UTxOs (cap 10) until
  `sum ≥ fee + minUtxo(change)`; if change would fall below min-UTxO, fold it
  into the fee (change-to-fee) rather than emitting an invalid output.
- **Validity interval**: `ttl = currentL1Slot + anchor.validity-slots`
  (default 7_200). A timed-out anchor tx is definitively expired before
  resubmission, so a resubmit can never race a late-landing original.
- **Rollback rewind**: persist `anchor_last_confirmed_height`; on L1 rollback
  of an anchor tx, rewind `anchor_last_height` to the last *confirmed* value
  (not `− everyBlocks`), so multi-interval ranges re-anchor completely.

**Acceptance.** Devnet e2e: anchor confirms with computed fee; forced rollback
re-anchors the full range; unit tests for input selection edge cases
(exact-fee input, dust change, multi-input).

## I1.6 Determinism conformance harness

**Problem.** Determinism is javadoc-only; a nondeterministic machine stalls the
chain in production with no pre-flight signal.

**Design (in `appchain-testkit`).**
- `StateMachineConformance` builder API:
  `provider(...)`, `settings(map)`, `corpus(...)`, `runs(n)` (default 3),
  `restartAtHeight(h)` (kill-and-reopen mid-corpus), `run()` →
  `ConformanceResult` (per-height roots per run, first divergence if any).
  `assertDeterministic()` convenience throws with a precise diff (height, run,
  roots).
- `Corpus`: deterministic pseudo-random block/message generator from a seed
  (`Corpus.random(seed, blocks, msgsPerBlock, bodyGenerator)`) or explicit
  (`Corpus.of(blocks)`); bodies supplied by the app under test so commands are
  realistic.
- Mechanics: each run applies the same corpus through the real
  `AppLedgerStore` commit path (same MPF batching as production) in a fresh
  temp dir; restart run closes and reopens the store mid-corpus and continues;
  results compared per height.
- Scaffold plugin template gains a pre-wired conformance test; user guide §6
  gains the forbidden-API list (wall clock, randomness, network/file I/O,
  environment, unordered iteration, locale/charset-dependent encoding,
  library-default serialization) — mirrored in the `AppStateMachine` javadoc.
- Runtime bytecode enforcement is explicitly out of scope (parent §I1.6).

**Acceptance.** stdlib machines all pass; a deliberately nondeterministic
sample (HashMap-iteration machine) fails with a useful diff; scaffold test
compiles and passes out of the box.

## I1.7 Snapshot signed manifests

**Problem.** `verifyIntegrity()` compares two stored values from the same
untrusted directory; snapshots carry no attestation binding them to a
certified tip.

**Design.**
- `createSnapshot()` writes, next to the checkpoint:
  - `snapshot-manifest.json`:
    `{version: 1, chainId, height, blockHash, stateRoot, memberEpochsHash,`
    `lastAnchor: {txHash, toHeight} | null, files: {relPath → sha256},`
    `createdAtMillis}`
  - `snapshot-manifest.sig`: Ed25519 signature over the exact manifest bytes
    by the node's member key (via the existing `SignerProvider` seam), plus
    the signer's public key hex.
- **Restore verification is pre-open** (RocksDB mutates files on open):
  at subsystem start, if `snapshot-manifest.json` is present and no
  `.manifest-verified` marker exists: verify the signature against the chain's
  configured member set (any current member key accepted; operator can pin
  `snapshot.trusted-keys`), verify every file hash, then write the marker.
  Failure → refuse to start.
- **Post-open binding**: tip block hash, state root, and member-epochs hash
  must equal the manifest values (catches a swapped-out manifest).
- `verifyIntegrity()` strengthened for all starts (manifest or not): recompute
  the tip block hash from stored bytes and verify its finality cert against
  membership-at-height — not just root == root.

**Acceptance.** Tamper tests: flipped byte in any sst file → refuse; manifest
signed by non-member → refuse; legit snapshot restores and serves; existing
no-manifest ledgers keep starting (manifest optional this release).

### I1.5 — delivered 2026-07-10

- Anchor tx fee is now **linear from the node's protocol params** (two-pass:
  sign a draft with a same-CBOR-width placeholder fee, measure, price
  `minFeeA×size + minFeeB`); params wired RuntimeNode →
  `AppChainSubsystem.wireAnchorFees` (current-epoch ledger params, falling
  back to static/genesis params, then to `anchor.fallback-fee-lovelace`).
- **Multi-input selection** (up to 10 pure-ADA UTxOs) grows until
  `fee + min-change` is covered; the change output is min-UTxO-guarded — a
  dust wallet fails with a clear "fund the anchor wallet" error instead of
  building an invalid tx.
- **Validity interval**: `ttl = newest observed L1 slot + anchor.validity-slots`
  (default 7200); no observed slot (cold start) → no TTL rather than blocking.
- **Precise rollback rewind**: the confirmed anchor's range start is persisted
  (`anchor_last_from_height`); an L1 rollback rewinds `anchor_last_height` to
  exactly `from − 1` — the old fixed `− everyBlocks` step under-rewound
  multi-interval ranges. *Known limit (documented):* only the last confirmed
  anchor is tracked; a deep rollback un-confirming several anchors at once
  remains a runbook case.
- Tests (`AnchorServiceTest`, 4, pure unit with a fake UtxoState — Mockito
  cannot mock `UtxoState` on this JVM): param-fee + TTL + two-input
  selection with exact fee/change assertions on the deserialized tx; fallback
  fee without params; dust-wallet clear failure; multi-interval rollback
  rewind precision (old code would rewind 8→0, new rewinds 8→3).

### I1.3 — delivered 2026-07-10

- Engine gains an `L1RefValidator` (verdicts `OK | MISMATCH | AHEAD | UNKNOWN`)
  wired from the subsystem's own `recentL1Points` window (depth + 64 blocks).
  In-window slot absent or hash mismatch = **MISMATCH** (fabricated or
  rolled-back ref) → proposal rejected before any re-execution.
- `handleProposal`: with `stability-depth > 0`, a missing L1 ref is rejected;
  monotonicity vs the previous block enforced; **AHEAD defers** — the engine
  executor became a ScheduledExecutorService and re-handles the held proposal
  every 500 ms for up to `max(2×block-interval, 5s)`, counted in
  `status().l1RefDeferrals`; deferrals clear on commit. **UNKNOWN** (ref older
  than the local window, e.g. right after restart) accepts on monotonicity
  with a WARN — the documented restart-window caveat.
- Catch-up: monotonicity always; MISMATCH rejects; AHEAD pauses the batch
  (next catch-up tick naturally retries after local L1 advances); blocks with
  `l1Slot = 0` from a pre-ref era are tolerated.
- **Fail-fast startup**: `stability-depth > 0` or anchoring with a null
  EventBus now refuses to start (was: silent `l1Slot = 0` blocks / anchors
  that never confirm).
- Tests (`AppChainL1RefTest`, 4, with per-node `SimpleEventBus` synthetic L1
  feeds): matching views finalize with a verified depth-correct ref;
  fabricated ref (same slots, different hashes) never finalizes; lagging
  follower defers then votes once its L1 catches up; missing event feed fails
  startup.

### I1.2 — delivered 2026-07-10

- Ledger: `commitBlock` writes `sender_seq_<key> → max(committed, block max)`
  per sender in the same atomic batch (one write per sender per block — avoids
  the WriteBatch read-visibility trap; floor never regresses).
- Own seq is **wall-clock-seeded** (`AtomicLong(currentTimeMillis)`) so any
  node — including gossip-only nodes with no ledger — never reuses a lower seq
  after restart; ledger nodes additionally floor it with the persisted value
  at start. *Learning:* the original design (persisted floor only) would have
  broken gossip-only submitters, whose last finalized seq lives in other
  nodes' ledgers.
- Admission (always on, ledger nodes): inbound seq ≤ finalized floor →
  dropped as `stale_seq`. Proposer selection filters the same and, with
  enforcement on, keeps per-sender seqs strictly increasing within the block
  (an honest enforcing proposer never builds a block enforcing followers
  reject).
- Consensus rule behind `message.enforce-sender-seq` (default **false** this
  release): `verifySenderSeqs` runs in both `handleProposal` and
  `applyCertifiedBlock` — per-sender strictly increasing above the finalized
  floor, missing seq (=0) also rejected.
- `AppChainConfig` gains `enforceSenderSeq` (canonical constructor extended
  again; 19-arg compat constructor keeps defaults). Full flat + `chains[i]` +
  node-app plumbing.
- Tests (`AppChainSenderSeqTest`, 5): admission replay drop, restart
  monotonicity, honest path with enforcement on, **adversarial two-node
  same-block duplicate-seq rejection** (proposer flag off + follower flag on →
  nothing finalizes) and its control (both flags off → finalizes, documenting
  default-off behavior). Full appchain runtime suite green.

### I1.4 — delivered 2026-07-10

- Settings namespace `yano.app-chain.machines.<machine-id>.*` flows through the
  existing `pluginSettings` seam: prefix added to the RuntimeNode union
  collector (single + multi-chain) and `forwardDynamicKeys` in node-app.
- `BalancesProvider`/`KvRegistryProvider` override `create(context)`;
  `approvals`/`doc-trail` keep the default no-arg delegation (no settings yet —

  (empty or 64-hex).
- kv-registry `ValueFormat` (`raw|cbor|utf8`): rejected at admission AND a
  deterministic no-op in `apply()` (consensus-enforced, the Wave-3 lesson).
  *Learning:* the CBOR check is "whatever `deserializeOne` accepts",
  deterministically — the lenient decoder accepts a lone `0xff` break byte, so
  the format check is structural, not strict-canonical; documented as such.
- `removeMember` now uses `normalizeMemberKeys` (same validation as
  `addMember`); rotation test extended with case/whitespace + malformed-key
  guards.
- Tests: `StdlibConfigWiringTest` (minter restriction end-to-end through
  ServiceLoader + context, fail-fast config, value-format no-op, parse);
  user guide §9 documents the settings and the all-members-identical rule.
