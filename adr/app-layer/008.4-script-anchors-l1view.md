## 0. In plain words

**Script anchors (I3.1).** Today an anchor is a note pinned to Cardano's
bulletin board (tx metadata): great for timestamps, but nothing *enforces*
what the next note says. A script anchor turns that into an **official
ledger slot on Cardano guarded by a machine**: one UTxO per chain whose
validator only lets the record advance — never rewind — and only when
**enough members have signed** the update. Other Cardano contracts can now
*trust and use* the app chain's state root (verify an MPF proof against it
on-chain), which is the doorway to enforceable bridges and settlement.

**L1View / observations (I3.2).** App chains sometimes need to react to
Cardano ("a deposit arrived at address X"). The trap is every member seeing
Cardano at a slightly different moment. The design: the proposer writes the
observation *into the app chain* ("tx T at slot S carried label L"), pinned
at or below the block's already-verified `l1-ref` — and **every follower
checks the claim against its own copy of Cardano before voting**. An
unconfirmable observation kills the proposal. L1 facts become ordinary,
provable, sequenced messages.

## 1. Goals and non-goals

**Goals:** on-chain-enforced anchor lineage (monotonic height, threshold
signatures, membership carried on-chain); the anchor artifact consumable by
other Plutus validators; deterministic, follower-verified L1 observations.

**Non-goals (this iteration):** validity/fraud proofs (E7.5 extends this
validator later); arbitrary historical UTxO queries in `L1View` (needs a
versioned UTxO store — explicitly deferred, see §3.1); bridge products
themselves (this ships the primitive they need).

## 2. I3.1 — Script anchor (mode A2)

### 2.1 On-chain ABI — one spec, two implementations

The datum/redeemer shapes are the *contract of the contract*, specified as
CDDL in `core-api/src/main/cddl/appchain/anchor-v1.cddl`. Both validator
implementations (julc default, Aiken opt-in) must satisfy it and are
drop-in interchangeable; everything off-chain derives script hash/address
from the **configured compiled UPLC artifact**.

```cddl
anchor-datum = [
    version      : uint,             ; 1
    chain-id     : bstr .size (1..128), ; app-chain identity (UTF-8 bytes)
    height       : uint,             ; app-block height this anchor attests
    block-hash   : bstr .size 32,
    state-root   : bstr .size 32,
    member-keys  : [ 1*32 bstr .size 32 ], ; Ed25519 member keys (sorted)
    threshold    : uint,
]
anchor-redeemer = 0                  ; "Advance" (reserved, not inspected)
```

On the wire the datum is Plutus Data wrapped in constructor 0 —
`Constr(0, [fields])` — which is what both the julc record codec and Aiken
types produce. The redeemer slot is a *reserved* action discriminant: v1
has a single action and the validator does not inspect the value (all
safety comes from the datum rules + signature threshold); off-chain code
passes int `0`. A future ABI version introducing a second action (e.g. a
ZK-verified advance, E7.5) assigns it a new redeemer value and gates
interpretation on the datum `version` field.

The 128-byte chain-id and 32-member limits are v1 framework invariants, not
decoder-only limits. The member cap is deliberately below the mathematical
validator limit so a worst-case 32-of-32 advance fits both bundled validators'
mainnet execution budgets with material headroom. `AppChainConfig` rejects
larger profiles at startup and governed
membership treats an addition beyond the limit as a deterministic no-op, so a
chain cannot finalize normally and discover the mismatch only when anchoring.

### 2.2 Validator rules (spending, Plutus V3)

A spend is valid iff:
1. **Both datums are exact v1 values**: input and continuing-output datums
   are `Constr(0, [version, chain-id, height, block-hash, state-root,
   member-keys, threshold])` with exactly seven fields, `version == 1`, a
   1--128-byte chain id, non-negative height, 32-byte block hash and state
   root, 1--32 strictly sorted unique 32-byte member keys, and
   `threshold ∈ [1, len(member-keys)]`. The validators can enforce the byte
   profile; the public `AnchorDatumV1` codec additionally requires canonical
   UTF-8 with no NUL in the chain id.
2. **Thread token continues**: exactly one continuing output to the script
   address carrying the state-thread NFT (§2.3), with an inline datum.
3. **Monotonic**: `out.height > in.height`, `out.chain-id == in.chain-id`,
   `out.version == in.version`.
4. **Threshold-signed**: at least `in.threshold` of `in.member-keys` appear
   in the transaction's required signers (blake2b-224 key hashes of the
   member Ed25519 keys — app-chain member keys double as L1 signing keys).
5. **Membership evolves on-chain**: `out.member-keys`/`out.threshold` MAY
   differ from the input's (rule 4 is checked against the INPUT datum —
   the current members authorize their successors, mirroring 008.3).
6. Value at the script is preserved or increased (no draining).

### 2.3 Identity: one-shot state-thread NFT

A one-shot minting policy (parameterized by the bootstrap UTxO outref)
mints exactly one thread token at bootstrap; the validator requires it to
continue (rule 2). Consumers identify THE anchor as "the UTxO holding
policy-id X" — no lineage walking, no trust in the operator's node. The
anchor's public identity = `(validator script hash, thread policy id)`,
printed at bootstrap and exposed via REST/status.

### 2.4 Implementations and artifacts

- **julc (default)**: validator + one-shot policy written in Java, compiled
  to UPLC at build time (`julc 0.1.0-pre14`, already in the catalog).
  julc is a **build/test-time dependency only** — the distribution bundles
  the compiled UPLC (hex) + blueprint (script hash) as resources.
- **Aiken (opt-in)**: the same two scripts in Aiken, source in-repo under
  `onchain/aiken/appchain-anchor/`, compiled artifact checked in beside it
  (building requires the Aiken toolchain; the artifact is what ships).
- Config selects the artifact; hash/address always derived from it:
  ```yaml
  yano.app-chain.anchor.mode: metadata | script      # default: metadata
  yano.app-chain.anchor.script.validator: builtin:julc   # or file:/path.uplc | hex:...
  yano.app-chain.anchor.script.thread-policy: builtin:julc
```

**Pre-release identity migration (2026-07-16).** Enforcing the complete v1
profile in both validators changed both compiled spending-validator artifacts
and therefore their script hashes. An anchor bootstrapped with an earlier
preview artifact is a different immutable on-chain identity and cannot be
advanced by the hardened validator. Preview databases/history are disposable,
so operators must clean or migrate that preview deployment and bootstrap a new
anchor after upgrading. Thread-policy artifacts are unchanged. This is the
expected consequence of correcting the ABI before a v1 release; future
released validator identities require an explicit versioned migration plan.

- **Conformance + budget gate**: one test-vector suite (happy path, height
  regression, sub-threshold sigs, missing thread token, membership change,
  chain-id swap, drain attempt) executed against BOTH artifacts with
  `julc-vm-java`; CI records script size + ex-units for both and fails if
  either exceeds the configured budget (mainnet limits with margin). If
  julc codegen busts budgets, Aiken becomes the default and the ADR gets a
  delivery note — measured, not assumed.

### 2.5 Off-chain: threshold co-signing over the app channel

Rule 4 needs `threshold` member signatures **on the L1 transaction** — a
single anchor wallet is no longer enough (and that's the point: a
compromised operator cannot advance a bogus root). The members already
share an authenticated channel, so anchor txs are co-signed over it:

1. The **anchor leader** (v1: the node with `anchor.enabled`, as today)
   builds the unsigned spend tx: input = current anchor UTxO, output datum
   = `{tip height, block-hash, state-root, current members, threshold}`
   from its ledger, fee/TTL per I1.5 machinery.
2. Leader broadcasts `[tx-body-cbor, anchor-outref]` on the reserved topic
   `~anchor/sign` (diffusion-only, like `~consensus/*` — never pooled).
3. Each member **independently verifies against its own ledger** —
   deterministic checks: datum height ≤ my tip; `block-hash`/`state-root`
   equal MY `block(height)`; members/threshold equal MY current epoch; the
   tx spends the expected anchor outref; outputs don't leak value. Only
   then it signs the tx body with its member key and replies on
   `~anchor/sig` with `[tx-id, witness]`.
4. Leader assembles ≥ threshold witnesses (its own included), submits via
   the node's own tx path, confirms via `BlockAppliedEvent` exactly like
   metadata anchors (pending/rollback semantics from I1.5 carry over:
   rollback → the anchor UTxO spend reappears in the mempool or is
   resubmitted; monotonicity makes duplicates harmless — first-in wins).
5. Members that can't verify (behind tip) simply don't sign; the leader
   retries next tick. Sub-threshold liveness = anchoring pauses (chain
   itself unaffected) — surfaced via `anchor.lastError` + health group.

Bootstrap is an admin action: `POST /chains/{id}/anchor/bootstrap` funds
and creates the initial UTxO + mints the thread token from the anchor
wallet (bootstrap needs no member signatures — identity starts at the
minted token).  The bootstrap datum is deliberately app height `0` with
zero block/state roots: it establishes identity only.  On a non-empty app
chain the normal scheduler immediately proposes the first real
threshold-co-signed advance to the current tip, independent of the regular
anchor cadence.  Followers therefore learn and promote the identity through
the existing verified-advance path rather than waiting for another user
transaction or the maximum anchor interval.  The leader records `(script
hash, policy id, outref)` in ledger meta and status.

### 2.6 What consumers get

- REST/status: anchor identity, current on-chain datum, last advance tx.
- Evidence bundles (E3.4) gain the script-anchor reference — an offline
  auditor can now check the anchor UTxO on ANY Cardano node/explorer.
- Other validators reference the anchor UTxO (by thread policy) and verify
  MPF inclusion proofs against `state-root` with the existing ccl Aiken MPF
  verifier — the enforceable-bridge primitive.

Each member derives that durable frontier independently from the unique
thread-NFT UTxO in its own committed L1 view and verifies the datum against
its own app-block hash and state root.  Only construction/submission remains
leader-only.  Therefore follower status, evidence, snapshots and
`L1_ANCHORED` effect gates converge without trusting a leader notification;
the UI distinguishes the leader's session-local confirmed-submission count
from a follower's session-local observed-anchor count.

Reconciliation is fork-safe: the thread output and the UTxO store's exact
committed `(slot, block-hash)` are read atomically and accepted only when that
point equals the member's canonical L1 tip.  This also permits a restarted or
restored member to correct stale anchor metadata downward without replaying a
historical rollback event.  A zero-config follower treats the first verified
sign request only as an identity candidate; it promotes that identity and
opens finality/effect gates only when one of the exact transactions it
verified is threshold-accepted into the committed thread UTxO.

## 3. I3.2 — L1View + `~l1` observations (scoped v1)

### 3.1 Scope decision

Full "read UTxOs at address as of slot S" needs a versioned UTxO store the
node doesn't have (UtxoState is current-tip) — **deferred, documented**.

**Honest residuals of the v1 design** (what pointer observations cannot
express): *negative facts* ("address X received nothing in [A,B]" — you
cannot point at an absence) and *cold-start aggregate scans* (facts from
before an observer was watching). Aggregates going FORWARD are covered:
the state machine accumulates individually-verified observations into
provable state. Observation liveness under a censoring proposer is
mitigated by rotation (008.2); under fixed mode it rides on the proposer
like everything else.

**Designated future architecture (recorded 2026-07-10, proposed by the
project owner): commitment-history + archival proof-serving.** Every
member incrementally maintains a UTxO-set commitment during L1 sync and
stores only the 32-byte ROOT per block (trivial storage, deterministically
recomputed by catch-up nodes); only archival/proposer nodes keep the full
**versioned JMT** (cardano-client-lib `verified-structures`, currently
preview) and attach JMT proofs — including exclusion proofs, which cover
the negative-fact residual — to observations; followers verify against
their OWN stored root for that slot. JMT's on-chain-verification cost is
irrelevant here (verification is follower-side, off-chain). Prerequisites
before this graduates from deferred: (a) benchmark incremental commitment
maintenance on the L1 sync hot path, (b) JMT implementation out of
preview, (c) an archival-role/liveness design compatible with rotation
(every eligible proposer needs proof-serving, or it becomes a role).
The `L1View` API ships extensible so this adds without breaking machines.

What ships NOW is what ADR-005 D5 actually needs for deterministic
L1-derived app logic:

- **Observation injection** (the primary mechanism): configured observers
  on the proposer emit messages on `~l1/<observer-id>`; each observation is
  a *pointer + claim*: `[tx-hash, slot, block-hash, claim]` where claim is
  observer-specific CBOR (e.g. metadata-label content, output-to-address
  amount). `slot ≤ block.l1-ref.slot` is REQUIRED.
- **Follower verification is consensus-critical**: before voting, followers
  verify the observation against their OWN L1 view and recomputed claim;
  mismatch → reject proposal (fail-closed); local view not yet at `slot` →
  the existing defer-retry machinery. Catch-up applies the same check when
  local history covers the slot.

  *Implementation refinement (2026-07-10)*: verification is **event-fed**,
  mirroring the I1.3 l1-ref design exactly — EVERY member runs the
  configured observers over its own streamed L1 blocks (the subsystem
  already receives fully-parsed blocks via `BlockAppliedEvent`) and keeps a
  bounded window of self-computed observations; follower verification is a
  lookup in that window. Same verdict semantics as l1-refs: match → OK,
  differs or absent-but-in-window → MISMATCH (reject), newer than our view
  → AHEAD (defer-retry), older than the window → UNKNOWN (accept — the
  certified chain vouches, exactly like old l1-refs during catch-up).
  Observers are pure functions `L1 block → observations`, so every member
  computes identical claims from its own stream — no chain-state random
  access, no re-serialization pitfalls. Chain-state pointer resolution
  (`blockAt(slot)` etc.) remains the L1View extension path for state
  machines that need reads outside the observation flow.
- **`L1View` (core-api)**, pointer-based reads for state machines and
  observers — `blockAt(slot)`, `tx(txHash, slot)`, `metadata(txHash, slot)`
  — all resolvable from chain state by concrete pointer, no scans.
- Built-in observers (config-selected): `metadata-label` (watch label N)
  and `address-deposit` (watch payments to address) — the two ADR-005 D5
  use cases (registries, bridge deposits). Observer SPI follows the same
  ServiceLoader pattern as everything else.

### 3.2 Consumption

Observations arrive at state machines as ordinary messages (blob-first,
topic `~l1/...` — machines opt in by decoding them; a CDDL + decode helper
ships in core-api). They are provable like any message, so "the group
agreed deposit T happened" carries an MPF proof against an anchored root —
closing the loop with §2.6.

### 3.3 Designed extension: checkpoint observations (owner-proposed 2026-07-10)

At anchor cadence (anchors are periodic, not per-block), the proposer
injects a **complete snapshot of each watched address's UTxO set** as a
`~l1/checkpoint/<observer-id>` observation: `[slot, address, utxo-set]`
(or `[slot, address, set-hash, chunk-refs]` when the 64 KB body cap forces
chunking). Why this is stronger than it looks:

- **Verification is near-live, so no historical store is needed**: the
  snapshot's slot sits within `l1.stability-depth` of the followers' tips.
  A follower verifies the COMPLETENESS claim by reconstructing the set at
  that slot from its current UTxO state minus/plus the retained delta log
  (`UtxoState.forEachUtxoDeltaInSlotRange` — feasibility check required:
  delta retention window must be ≥ stability depth, a config constraint to
  validate before implementation).
- **Once finalized, it is certified history**: catch-up nodes get the L1
  facts as ordinary verified app messages — no L1 history required, same
  trust model as everything else in the ledger (threshold-verified when
  fresh, cert-attested forever after).
- **It closes most §3.1 residuals for watched addresses at checkpoint
  granularity**: negative facts ("nothing arrived between checkpoints") =
  two equal snapshots with no event observations between; cold-start =
  covered from the first checkpoint onward. Every snapshot is provable
  state under the (immediately following) anchored root — auditor-friendly
  by construction.
- **Bounded size**: pre-declared watched addresses only, anchor cadence
  only; typical bridge/registry addresses hold small live sets. Hot
  addresses fall back to hash+chunks.

Not in I3.2 v1 scope (event observations ship first); tracked in
`pending-tasks.md` as a late-Iteration-3 / Iteration-4 candidate gated on
the delta-window feasibility check. Together with the JMT
commitment-history architecture (§3.1) this gives a two-tier future: cheap
checkpoints for watched state, archival proofs for arbitrary state.

## 4. Test plan (definition of done)

**On-chain (in-JVM via julc-vm-java, both artifacts):** the §2.4 vector
suite; budget report in CI.
**Co-signing:** 2-of-3 devnet-less integration — leader builds, members
verify+sign over sockets, sub-threshold pause when a member lags, bogus
datum (wrong root) refused by members.
**Observations:** proposer injects a metadata observation; followers verify
against own chain state; fabricated claim rejected fail-closed; lagging
follower defers then votes; catch-up re-verifies.
**Devnet e2e (merge gate):** bootstrap → several script-anchor advances
(threshold-signed) confirmed on devnet L1 → governed membership change →
next advance carries the new member set on-chain → evidence bundle
references the script anchor; regression skills for metadata mode + new
`test-app-chain-script-anchor` skill.

### I3.2 — L1 observations — DONE (2026-07-10)

- Event-fed design per the §3.1 implementation refinement: EVERY member
  runs the configured observers over its own streamed L1 blocks
  (`L1ObservationService`, window = stability-depth + 64 blocks, mirrors
  recent-L1-points) and the currently-scheduled proposer injects the
  results as `~l1/<observer-id>` messages (internal pool path — `submit()`
  still rejects `~` topics externally).
- **Rollback safety (three layers)**: (1) injection is STABILITY-GATED —
  observed facts wait in a pending queue and drain only once
  `slot ≤ stable-l1-ref` (a fact must be `l1.stability-depth`
  confirmations old before it may be sequenced, because the app chain
  never rolls back); (2) the engine enforces the §3.1 REQUIRED rule
  `observation.slot ≤ block.l1Slot` fail-closed on proposals AND catch-up,
  so a rushing/malicious proposer cannot finalize a reorg-able fact —
  observers therefore REQUIRE `l1.stability-depth > 0` (start fails
  otherwise); (3) an L1 rollback prunes both the verification window and
  the pending queue — pre-stability facts reorged away are simply
  forgotten and re-observed at their new slot on the new chain.
  Residual: the drain is a single deterministic attempt per fact (all
  members drain at the same L1 block; the scheduled proposer injects, and
  pool replication carries the message across proposer changes) — if the
  scheduled proposer is OFFLINE at exactly the drain block, that fact is
  not sequenced (recorded in pending-tasks; rotation makes the window
  short, and a down proposer stalls the chain anyway).
- Consensus-critical follower verification grafted into the engine beside
  the I1.3 l1-ref check (`verifyProposalObservations` /
  `verifyCatchUpObservations`, shared verdict enum + defer machinery):
  MISMATCH (differing claim, wrong block hash, fabricated in-window,
  undecodable body, topic/body disagreement) rejects fail-closed; AHEAD
  defers with the existing bounded retry; below-window is UNKNOWN
  (certified chain vouches — catch-up's common case). A member with NO
  observers configured rejects any proposed observation — observers must
  be configured identically on every member (misconfig fails start).
- core-api: `l1view/L1Observation` (+ CBOR codec,
  `l1-observation-v1.cddl`), `L1Observer` SPI, `L1ObserverProvider`
  (ServiceLoader, same pattern as sequencer modes). Built-ins:
  `metadata-label` (claims the label's CBOR value) and `address-deposit`
  (claims `[address, lovelace]` per paying tx) — the two ADR-005 D5 use
  cases. Config: `observers.<id>.type/...` (new `observers.` plugin-
  settings prefix); L1 rollback prunes the window.
- Tests: full verdict matrix (incl. tampered claim, fabricated pointer,
  wrong block hash, topic mismatch), two-member determinism, rollback
  semantics, fail-fast misconfig. Devnet e2e rides the I3.3 gate.
