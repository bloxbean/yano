## 0. In plain words (the notebook picture)

The app chain is a shared notebook kept by a small group. One member (the
*proposer*) writes the next page; the others check and sign it. A page is
final once *threshold*-many members signed (e.g. 2-of-3).

**Problem:** today one fixed member writes every page. If Alice is offline,
the notebook freezes. **Rotation** fixes this: members take turns by a public
schedule — miss your turn, the next member writes. The hard case is a writer
dying *mid-page*. The iron safety rule is: **each member signs at most ONE
version of a page, ever** — that's what makes two conflicting "final" page-10s
impossible. So if Bob already signed Alice's half-finished version X, he can
never sign Carol's fresh version Y. Two habits solve it without a heavyweight
protocol: **keep showing your copy** (Bob re-gossips X until page 10 is done)
and **finish what's started** (Carol pushes X instead of writing Y).

**Honest residual case:** with 2-of-2, a precise double-accident (Alice signs
X, Bob never saw X and signs his own Y) freezes that page — each spent their
one signature on a different version. The node detects it and raises a loud
alarm; an operator runbook resolves it. With 2-of-3, a third member breaks
the tie automatically ("oldest proposal wins"), so prefer thresholds like
2-of-3 / 3-of-4 over all-of-n. We deliberately chose this rare-freeze +
alarm + runbook trade over a full BFT view-change protocol (~3× consensus
complexity, where real-world consensus bugs live). A BFT mode remains
addable later — see §2.8.

## 1. Goal and non-goals

**Goal:** remove the single-proposer liveness dependency. Today (S1) a crashed,
partitioned, or censoring proposer halts the chain until an operator acts. With
S2, proposership rotates over deterministic time windows; a missed window costs
one window of latency, not an outage.

**Non-goals:** full BFT view-change (PBFT/HotStuff-class); proposer slashing;
changing the finality model. The threshold-cert safety design of ADR-005 D2 is
kept intact — S2 changes *who may propose when*, not *what makes a block final*.

## 2. Design

### 2.1 Window schedule (L1 as the shared clock)

Every member already observes the same L1 chain, so L1 slots are the shared
clock (ADR-005's "why L1 stays on" rationale — no NTP trust added):

```
window(slot)      = floor(slot / windowSlots)           ; windowSlots per chain config
proposerFor(w, h) = sorted(membersAt(h))[ blake2b256(chainId ‖ w)[0..7] mod n ]
```

- `membersAt(h)` is the existing epoch-resolved member set (rotation composes
  with membership changes, including 008.3 governed membership).
- The hash-shuffle (vs plain round-robin) avoids a fixed victim ordering;
  the ADR-005 `epochNonce` ingredient is dropped for now — unpredictability
  matters for grinding resistance in open networks, not in a permissioned
  group where the schedule being public is fine. It can be added inside the
  hash later without changing the structure.
- A member's *current* slot = newest entry of its `recentL1Points` (already
  maintained since I1.3). Rotating mode therefore **requires an L1 feed**
  (same fail-fast as I1.3) and `windowSlots` large enough to absorb L1 view
  skew between members (default 60 slots; must be ≥ 2× `block.interval-ms`
  worth of slots).

### 2.2 Proposal rules

- Every member runs the proposer tick; it proposes only when
  `proposerFor(currentWindow, h) == self` and no round is pending.
- **Grace acceptance:** a follower accepts a proposal if its proposer was
  eligible for the follower's *current or immediately previous* window —
  absorbing boundary skew between members' L1 views.
- **Adoption-first rule:** at the start of its window, before building a fresh
  block at height `h`, a proposer waits a short adoption grace
  (`min(blockIntervalMs, windowSlots·slotLen/4)`) for any re-gossiped locked
  proposal at `h` (see 2.3) and, if one arrives, re-broadcasts *that block
  verbatim* instead of proposing its own. This is what makes partial rounds
  converge instead of splitting votes.

### 2.3 Safety: vote locks are kept; liveness: locked proposals re-gossip

The S1 invariant — **a member votes for at most one block per height, ever**
(persisted vote lock) — is retained unchanged. This is the safety anchor: two
conflicting certs at one height would require `threshold` members to
double-vote, which the lock forbids across restarts.

The consequence is the classic partial-round problem: proposer A's block at
height `h` collected some votes but no cert, then A died. Locked members can
never vote for a different block at `h`. S2 handles it without a view-change
protocol:

1. **Locks persist the block, not just the hash.** The vote lock gains the
   locked block's CBOR (meta `vote_lock_block_<h>`), so a locked member can
   re-broadcast the full original, proposer-signed proposal after a restart.
2. **Locked members re-gossip** their locked proposal on every round-timeout
   tick until the height is finalized. Re-gossip preserves the original
   proposer signature — receivers verify it exactly like a live proposal.
3. **Elapsed-window eligibility:** a proposal at height `h` is acceptable if
   its proposer was eligible for *some* window since the member committed
   `h-1` (bounded search over elapsed windows, capped at `maxElapsedWindows`,
   default 64). An old-window proposal re-gossiped later is therefore still
   verifiable.
4. **Earliest-window preference:** when competing acceptable proposals exist
   at one height, an unlocked member votes for the one whose proposer's
   earliest eligible window is lowest. Combined with the adoption-first rule,
   honest members converge on the oldest partially-voted block.

**Residual risk (explicit trade-off):** one-vote-per-height means split votes
can still exhaust the vote budget — worst case `2-of-2` where each member
voted a different block before seeing the other's. The adoption grace and
earliest-window preference make this a boundary-race rarity rather than a
crash consequence, but it is not eliminated without real view-changes
(deferred to a future BFT option per ADR-005 S4). Mitigations shipped with
this ADR: a `SPLIT_VOTES` detector (locked members observing a competing
locked proposal at the same height emit `AppChainStalledEvent` with reason)
and a documented runbook (the split is visible, deterministic, and resolvable
by a coordinated operator action). For `threshold ≤ n−1` configurations a
single split vote does not stall (the remaining members still reach
threshold on one side).

### 2.4 Certified history: certs are the authority

Catch-up (`applyCertifiedBlock`) under rotating mode verifies
`proposer ∈ membersAt(height)` **plus the threshold cert** — it does *not*
try to reconstruct which window the block was proposed in (window timing is
not part of the block, deliberately: no wire/CDDL change). Rationale: for
finalized history the cert *is* the legitimacy proof — `threshold` honest
members only voted after applying the live window rules. This mirrors how
membership epochs already trust finalized history.

### 2.5 Configuration and migration

```yaml
yano.app-chain.sequencer.mode: fixed | rotating     # default: fixed (S1)
yano.app-chain.sequencer.window-slots: 60           # rotating only
yano.app-chain.sequencer.proposer: <hex>            # fixed only (as today)
```

- All members of a chain must agree on the mode (like the state-machine id);
  a mixed config fails closed — wrong-proposer proposals are rejected and the
  chain stalls rather than diverging.
- `rotating` requires an L1 feed and `l1.stability-depth ≥ 0` is unchanged;
  the window clock uses the *newest observed* slot (not the stable ref).
- S1 remains the default until S2 has soaked (flip considered in a later
  release). Existing chains can switch modes with a coordinated restart —
  no ledger migration is required (nothing about blocks changes).

### 2.6 Engine changes (seams identified in the I-1 audit)

| Site | Change |
|---|---|
| `AppChainEngine.isProposer` (final boolean) | → `proposerFor(window, height)`; tick self-gates |
| `handleProposal` proposer check | fixed-key equality → grace/elapsed-window eligibility |
| `applyCertifiedBlock` proposer check | fixed-key equality → membership-at-height (cert authority) |
| `doProposeTick` | adoption grace; re-gossip of locked proposal when locked |
| `AppLedgerStore` vote lock | persist locked block CBOR alongside the hash |
| `AppChainSubsystem` scheduling | proposer tick scheduled on **all** members in rotating mode |
| status/metrics | `currentWindow`, `currentProposer`, `missedWindows` counter, `yano.appchain.proposer.missed_windows_total{chain,member}` |

### 2.7 Extensibility: sequencer modes are a small SPI (pluggable consensus)

`fixed` and `rotating` become two built-in implementations of one small
interface, so future modes (BFT view-change, Raft-backed, leaderless merge —
ADR-005 S3/S4/S5) are config choices, and applications can supply their own
as a plugin jar or in library mode — exactly the `AppStateMachineProvider`
pattern:

```java
public interface SequencerMode {
    String id();                                       // "fixed", "rotating", custom id
    void init(SequencerContext ctx);                   // self key, membersAt, L1 slot supplier,
                                                       //   mode settings (sequencer.* config)
    boolean shouldProposeNow(long height);             // self-gate for the proposer tick
    ProposalEligibility checkProposal(byte[] proposer, long height);
                                                       // ACCEPT | REJECT | DEFER (live path only)
}

public interface SequencerModeProvider {               // ServiceLoader-discovered
    String id();
    SequencerMode create(SequencerContext ctx);
}
```

- Selection: `yano.app-chain.sequencer.mode` — built-ins first, then
  ServiceLoader lookup (plugin jars in `plugins/`, or programmatic in library
  mode). Unknown id = fail-fast at startup.
- The SPI governs the **live path only** (who proposes, whose proposals are
  acceptable). Finalized history stays mode-independent: catch-up always
  verifies membership-at-height + threshold cert (§2.4) — so chains can even
  change mode over their lifetime without re-verifying old blocks, and a
  custom mode cannot weaken finality (certs and one-vote-per-height remain
  framework-enforced outside the SPI).
- All members must run the same mode id (fail-closed, like the state-machine
  id); documented in the same "must match on every member" table.

### 2.8 Future modes (explicitly kept open)

The SPI slots in ADR-005's deferred options without redesign: **S4 BFT
view-change** (removes the §0 residual freeze; signatures bound to rounds),
**S3 Raft/CFT**, **S5 leaderless deterministic merge**. Each is "implement
`SequencerModeProvider`, drop the jar, set `sequencer.mode`".

### 2.9 I2.3 — message-id dedup hardening (rides this iteration)

Replace the count-bounded (100k) `seenMessageIds` set with TTL-bounded
eviction: `id → expiresAt` (the envelope's own expiry), swept with the
existing pool sweep tick, plus a hard cap (per-chain
`dedup.max-entries`, default 200k) as a memory backstop. An id can no longer
age out of the window while the message is still admissible — closing the
re-diffusion loophole recorded in ADR-008 finding 6.

## 3. Test plan (definition of done)

Adversarial, multi-node (extends the I-1 harness patterns):

1. **Proposer crash, clean round** — kill the window proposer before it
   proposes; chain resumes within ≤ 1 window; missed-window metric increments.
2. **Proposer crash, partial round** — kill after votes but before cert;
   locked members re-gossip; next proposer adopts; the SAME block finalizes.
3. **Boundary skew** — follower's L1 view one window behind: grace acceptance
   still votes; no stall.
4. **Rogue out-of-window proposer** — proposals from a member not eligible in
   any elapsed window are rejected (fail-closed), chain unaffected.
5. **Split-vote detection** — engineered 2-of-2 split raises the stall event
   with the split reason (documented runbook path).
6. **Rotation × membership epochs** — schedule follows `membersAt(h)` across
   an admin (and later 008.3-governed) rotation.
7. **Catch-up** — late joiner verifies rotating-era history by certs.
8. **Dedup TTL** — an unfinalized id re-diffused after expiry is dropped;
   before expiry it stays deduplicated; memory stays under the cap.

Plus: S1 regression suite unchanged (mode default), devnet cluster skill run
with `mode: rotating`, and a soak (≥ 30 min devnet, random proposer kills).

## Delivery notes

### I2.1 (rotating sequencer) — delivered 2026-07-10

- **SPI shipped as designed** (§2.7): `SequencerMode`/`SequencerContext`/
  `SequencerModeProvider` in core-api; built-ins `FixedSequencerMode` (S1,
  reads `sequencer.proposer`) and `RotatingSequencerMode` (window schedule,
  lookback acceptance, DEFER without a clock); ServiceLoader for custom modes;
  selection via `sequencer.mode` (bare `sequencer.proposer` still means fixed
  — full v1 compat, zero `AppChainConfig` constructor changes: the
  `sequencer.*` settings ride the pluginSettings prefix collector).
- Engine: mode-gated proposing on EVERY member's tick; proposal acceptance =
  envelope-sender-is-block-proposer (authenticity) + mode verdict
  (ACCEPT/REJECT/DEFER with bounded retry); catch-up = membership-at-height +
  cert (mode-independent, §2.4); **any member holding the round aggregates
  votes and can form the cert** (a dead proposer can no longer sink collected
  votes); followers record their own vote locally; split-vote observation
  counted + logged (`status().sequencer.splitVotesObserved`).
- Partial-round recovery (§2.3): vote locks persist the ORIGINAL
  proposer-signed envelope (`ConsensusCodec.encodeEnvelope`); locked members
  re-gossip it (+ re-vote) every round timeout until the height finalizes;
  proposers holding a lock re-gossip instead of proposing.
- **Three real liveness bugs found & fixed by the partial-round test** (all
  pre-existing in the transport layer, exposed by "peer down" scenarios):
  1. `AppPeerClient.enqueue` shared a monitor with `ensureConnected`, which
     blocks through an infinite dead-peer connect-retry loop — submitters
     stalled behind it. Enqueue now only locks the replay queue.
  2. The blocking connect ran ON the shared subsystem scheduler — one dead
     peer silently wedged proposer/anchor/catch-up ticks entirely.
     Connect attempts now run on per-peer daemon connector threads
     (`ensureConnectedAsync`).
  3. `shutdown()` (same monitor) deadlocked behind the retrying connector;
     it is now monitor-free and closes the pre-published in-progress client,
     which breaks the retry loop.
  Plus one race: consensus messages arriving between server bind and engine
  wiring were **permanently lost** (wire dedup + the transport never
  re-delivers an acked id in a session) — system-topic messages now always
  route (relay still deduped for loop control) and an `earlyConsensus`
  buffer drains into the engine at start.
- Tests: `RotatingSequencerModeTest` (5 — schedule determinism, self-gating,
  lookback acceptance/reject, defer, fail-fast settings) and
  `AppChainRotationIntegrationTest` (2 — **killing the block-1 proposer no
  longer halts the chain** with distinct proposer on block 2 and equal roots;
  partial round finishes the SAME original block across a proposer restart,
  provably via re-gossip since the pool is memory-only). Full appchain suite
  (20+ classes) green.
- *Deviations from design:* the earliest-window preference for competing
  proposals was dropped — first-acceptable-wins voting plus proposer-side
  lock-re-gossip converges in practice, and the split case is detected +
  counted (runbook); the missed-window accountability metric is deferred to
  the wrap-up item (needs per-member attribution design).

### I2.3 (dedup TTL) — delivered 2026-07-10

- `SeenMessageIds` (runtime): `id → expiresAt` map; duplicates recognized for
  the message's whole lifetime; swept on the existing 30s pool tick; expired
  ids become markable again (safe — the agents reject the expired message
  itself). `status()` gains `seenIds`.
- *Deviation from design:* the hard cap is a fixed constant (200k,
  `SEEN_IDS_HARD_CAP`) rather than per-chain `dedup.max-entries` config — a
  pure memory backstop didn't justify another `AppChainConfig` constructor
  extension; cap hits log a WARN with the eviction count. Revisit only if a
  real deployment hits the cap.
- Tests: `SeenMessageIdsTest` (dedup within TTL, sweep + re-markable after
  expiry, cap evicts only the excess); backpressure + two-node integration
  suites re-run green.

### I4.2 (stale vote-lock recovery) — delivered 2026-07-11

The Iteration-3 devnet gate found a permanent liveness wedge in the vote-lock
design: a member that voted at height H, crashed, and restarted after the
locked proposal's messages EXPIRED could neither re-gossip the round (every
honest member rejects expired messages at proposal verification) nor propose
a competing block — the height was dead, warning forever.

Two-part fix, safety-first:

- **Propose-around (automatic).** When the lock's stored envelope is missing
  or expired, the locked block is permanently un-certifiable — so the
  proposer may build a FRESH block at the height while **never self-voting**
  (its lock and the at-most-one-vote guarantee stand untouched, and the
  stored envelope is not clobbered). The round then needs `threshold` votes
  from OTHER members: any config with threshold ≤ n−1 heals automatically.
  The wedge is surfaced as `sequencer.staleLockedHeight` in status while
  unresolved.
- **Operator unlock (runbook escape hatch).** For threshold-unreachable
  configs (e.g. 2-of-2), `POST /app-chain/admin/unlock-stale-round` clears
  THIS member's lock at tip+1 so it may vote once more. The engine REFUSES
  while the locked round is still recoverable (proposal not expired).
  Runbook: before unlocking, confirm on every member that height tip+1 has
  no certificate (`status.tipHeight` equal everywhere, no cert for the
  locked hash) — the unlock consciously overrides the one-vote guarantee
  under operator supervision, which is safe exactly when the old block can
  no longer certify.

Tests (`AppChainStaleLockTest`): 2-of-3 propose-around finalizes with a cert
that verifiably EXCLUDES the stale-locked member's signature; 2-of-2 unlock
is refused while recoverable, then restores liveness with the member voting
once more. The recoverable path (`partialRound_finishesSameBlockAfterRestart`)
is unchanged and re-run green.
