## 0. In plain words

Yano can already load plugin JARs and discover typed providers, but each SPI is
currently scanned independently. The node cannot reliably answer which
providers belong to one product, whether that product is compatible with this
Yano plugin API, whether its dependencies are present, or whether operator
allow/deny policy applies to all of its contributions.

This ADR adds one deterministic catalog in front of provider activation. A
plugin artifact supplies a bundle-id-qualified manifest under
`META-INF/yano/plugins/`. The manifest gives the
artifact one bundle identity and declares the typed ServiceLoader providers it
contains. The manifest does not instantiate arbitrary classes and does not
replace any typed SPI. A declared provider must also be registered in the
corresponding Java ServiceLoader file; only the existing SPI creates behavior.

For manifested bundles, catalog construction completes manifest parsing,
policy, compatibility, dependency, conflict and provider-correlation checks
before selected provider constructors or lifecycle callbacks run. The
resulting registry is then the only provider source used by `PluginManager`
and app-chain extension lookup. Existing unmanifested providers continue
through explicit synthetic legacy descriptors for JVM directory JARs and
library-compatibility embedding during the migration window. That adapter must
construct a legacy provider before policy so it can snapshot the selector and
`NodePlugin` dependency metadata that have no declarative source; callbacks
are bounded, TCCL-scoped and converted to secret-safe platform diagnostics.
Packaged JVM/native build-time inclusion requires a manifest so its executable
closure has a stable owner and never needs this exception.

The first class-loader model remains deliberately simple: directory artifacts
are captured and validated one at a time, then one deterministic parent-first
loader is created from policy-selected snapshot JARs only. Policy-filtered JARs
remain visible in immutable inventory but are never lookup URLs on the
executable loader. Native images retain build-time inclusion. This ADR adds
ownership, validation and diagnostics; it does not claim sandboxing or hot
reload.

## 1. Context and verified gaps

The repository already has useful pieces:

1. `PluginClassLoaderProducer` builds a parent-first `URLClassLoader` from
   `yaci.plugins.directory` for packaged JVM nodes and falls back to the
   build-time classpath in native images.
2. ADR-011.1 makes `NodePlugin` discovery, policy, dependency ordering and
   lifecycle transactional.
3. Seven typed app-chain SPIs ship today:
   `AppStateMachineProvider`, `SequencerModeProvider`, `L1ObserverProvider`,
   `SignerProviderFactory`, `AppEffectExecutorFactory` and
   `FinalizedStreamSinkFactory`, plus ADR-011.3's `DomainApiProvider`.
4. First-party extension artifacts already use ServiceLoader files, including
   stdlib state machines, ZK state machines, the Kafka sink and the Cardano
   effect executor.

The missing control plane is concrete:

- typed SPIs still call `ServiceLoader.load()` independently and can bypass
  `PluginsOptions` policy;
- provider iteration and plugin-directory JAR order are not a platform
  contract;
- duplicate provider names are generally first-discovery-wins;
- an artifact has no shared id/version/dependency or Yano API compatibility;
- providers from a manifested artifact cannot be checked for undeclared or
  missing ServiceLoader entries;
- the existing plugin class loader is not explicitly owned/closed and does not
  reject plugin artifacts that package their own Yano API classes;
- there is no deterministic inventory or deployment-comparison fingerprint;
- native inclusion has no catalog conformance test.

## 2. Scope

### 2.1 Goals

1. Define a strict versioned JSON manifest for one bundle identity and its
   typed provider contributions.
2. Discover all eight supported provider kinds through one catalog:
   `NodePlugin` plus the seven app-chain SPIs.
3. Validate the full selected bundle graph before manifested provider
   construction, every `NodePlugin.init()`, or app-chain contribution creation;
   legacy provider construction is the documented metadata-snapshot exception.
4. Apply global enablement and bundle allow/deny policy to every contribution.
5. Reject incompatible API ranges, duplicate ids/names/providers, missing
   dependencies, dependency-version mismatches, cycles and manifest ↔
   ServiceLoader drift deterministically.
6. Preserve unmanifested ServiceLoader providers through visible synthetic
   legacy descriptors in JVM directory and explicit library-compatibility
   modes; require manifests for packaged JVM/native build-time inclusion.
7. Provide one immutable provider registry to all consumers so no supported
   SPI performs an independent scan after catalog construction.
8. Calculate runtime-owned artifact/closure evidence and a canonical
   selected-catalog fingerprint.
9. Expose a read-only inventory suitable for later REST, health and CLI
   surfaces without leaking local filesystem paths or configuration values.
10. Make JVM plugin-directory loading deterministic and lifecycle-owned, while
    retaining native build-time inclusion.

### 2.2 Non-goals

This ADR does not add:

- arbitrary provider classes named only in JSON;
- child-first/per-bundle class-loader isolation or Java module layers;
- sandboxing, permission enforcement, hot reload or runtime install/uninstall;
- signed JAR/manifest policy or external supply-chain attestation;
- generic machine query/domain API routes (ADR-011.3);
- plugin health, metrics, CLI or dashboard surfaces (ADR-011.4);
- consensus-visible executor installation or any change to ADR-010 effect
  records;
- automatic app-chain state-machine upgrade coordination. Manifest
  compatibility cannot replace height-gated deterministic upgrades.

## 3. Terms and compatibility version

- **Bundle** — one manifested plugin product, identified by stable reverse-DNS
  id and semantic version.
- **Contribution** — one typed ServiceLoader provider declared by kind, public
  selector name and provider class.
- **Catalog** — immutable validated bundle/provider graph for one runtime.
- **Selected bundle** — discovered bundle that survives global and
  allow/deny policy.
- **Legacy bundle** — synthetic descriptor for an unmanifested ServiceLoader
  provider.
- **Plugin API major** — an integer compatibility contract independent of the
  Yano product release version. This ADR starts at major `1`.

Product version `0.1.0-pre9`, Cardano protocol version and manifest schema
version are not plugin API compatibility versions. Incrementing the plugin API
major requires an explicit compatibility ADR and migration window.

## 4. Manifest v1

### 4.1 Authoring location, packaged index and example

Each manifested artifact contains exactly one bundle-id-qualified resource:

```text
META-INF/yano/plugins/<bundle-id>.json
```

For example, bundle `com.example.product-passport` uses
`META-INF/yano/plugins/com.example.product-passport.json`. The filename must
exactly equal the manifest `id`. Unique resource paths let multiple manifests
survive an uber-JAR merge without relying on duplicate-resource behavior.

Discovery never assumes a class loader can list a resource directory. The
packaged application contains a build-generated canonical index:

```text
META-INF/yano-plugin-index-v1.json
```

The strict JSON index contains canonical manifest snapshots, runtime-owned
byte evidence, digest mode, and each correlated ServiceLoader provider
declaration. Entries are sorted by bundle id and contribution key.
The application build generates it by inspecting resolved runtime artifacts
and project outputs *before* Quarkus uber-JAR or native-image packaging,
requires an explicit root-plus-executable-dependencies closure for every
manifested build-time bundle, validates that every provider/resource is
retained, and includes the index as a native resource. A plugin-directory JAR
is inspected and hashed directly and merged into the runtime catalog; it does
not mutate the embedded application index. Tests and embedders supply explicit
directory artifact sources through the public `PluginLoaderHandle` +
`PluginRuntimeEnvironment` boundary; the internal catalog builder is not
exposed as a second activation API.

An extension artifact still carries its qualified authoring manifest; the
application-generated index is authoritative after flattening. Index
generation rejects conflicting paths, ids, provider declarations or service
entries instead of letting a packaging tool choose a duplicate-resource
winner. An input dependency already containing
`META-INF/yano-plugin-index-v1.json` is also rejected: an application has one
authoritative generated index, never a merge of nested aggregate indexes.

Discovery mode is explicit. When plugins are enabled, packaged JVM and native
applications require exactly one embedded aggregate index; zero or multiple
copies are fatal. Disabling plugins intentionally bypasses provider discovery
and publishes an empty selected catalog, so no index is required in that mode.
Apart from that intentionally disabled mode, only the public
library-compatibility mode may run without an index. When an index is present,
every supported ServiceLoader provider must be represented
by either a manifested bundle entry or an indexed legacy entry: the runtime
never silently downgrades an omitted provider to a synthetic legacy bundle.

Example:

```json
{
  "schemaVersion": 1,
  "id": "com.example.product-passport",
  "version": "1.2.0",
  "yanoApi": {
    "min": 1,
    "max": 1
  },
  "dependencies": [
    {
      "id": "com.example.shared-claims",
      "minVersion": "1.0.0",
      "maxVersionExclusive": "2.0.0"
    }
  ],
  "contributions": [
    {
      "kind": "app-state-machine",
      "name": "product-passport",
      "provider": "com.example.passport.PassportStateMachineProvider"
    }
  ]
}
```

### 4.2 Strict parsing limits

- UTF-8 JSON only;
- maximum manifest size: 64 KiB;
- duplicate JSON keys and unknown fields are rejected;
- `schemaVersion` must equal `1`;
- every array is present or treated as empty, then defensively copied;
- bundle/dependency ids are at most 160 characters, contribution names at
  most 128, provider binary class names at most 512, and each manifest has at
  most 256 dependencies and 256 contributions;
- the aggregate index is limited to 16 MiB, 4,096 bundles and 32,768 explicitly
  indexed legacy providers;
- error messages identify bundle/resource and field but never dump the full
  manifest or configuration.

### 4.3 Identity and version rules

- `id` is lowercase reverse-DNS form, 3–160 characters, with no whitespace;
- `version` is strict SemVer 2.0.0, including optional prerelease/build data;
- `yanoApi.min` and `.max` are positive inclusive API majors and
  `min <= max`;
- the current runtime API major must fall within the range;
- dependency ids use the same id grammar;
- dependency `minVersion` is inclusive and `maxVersionExclusive` is exclusive;
  either bound may be omitted, but an empty range is invalid;
- all dependencies are required in v1. Optional/degraded dependency semantics
  require a later decision because they affect lifecycle and health.

### 4.4 Contribution kinds

| Manifest kind | Java SPI | Selector | Trust tier |
|---|---|---|---|
| `node-plugin` | `NodePlugin` | `id()` | strongest/required compatibility lifecycle |
| `app-state-machine` | `AppStateMachineProvider` | `id()` | CONSENSUS |
| `sequencer-mode` | `SequencerModeProvider` | `id()` | CONSENSUS |
| `l1-observer` | `L1ObserverProvider` | `type()` | CONSENSUS |
| `signer-provider` | `SignerProviderFactory` | `scheme()` | PRIVILEGED_LOCAL |
| `effect-executor` | `AppEffectExecutorFactory` | `scheme()` | PRIVILEGED_LOCAL |
| `finalized-sink` | `FinalizedStreamSinkFactory` | `scheme()` | AUXILIARY_LOCAL |
| `domain-api` | `DomainApiProvider` | `id()` (equal to bundle id) | PRIVILEGED_LOCAL; manifest required (added by ADR-011.3) |

The platform derives trust from kind; a manifest cannot choose a weaker tier.
`node-plugin` remains required because the legacy SPI is open-ended and may
provide validation customization. A later typed replacement can narrow that
classification.

Contribution rules:

- `(kind, name)` is unique across the selected catalog;
- an aggregate index may retain multiple installed alternatives with the same
  `(kind, name)` when policy excludes all but one; selecting more than one is a
  deterministic startup error in packaged JVM, native and directory modes;
- `provider` is a fully qualified ASCII binary class name and must be unique
  for that kind;
- the provider class must also appear in the corresponding
  `META-INF/services/<SPI>` resource;
- the catalog obtains provider instances only through `ServiceLoader.Provider`;
  it never calls `Class.forName(manifestValue).newInstance()`;
- after lazy provider construction, its selector method must exactly match
  manifest `name`;
- an artifact with a manifest may not contain an undeclared supported provider;
- a manifest may declare at most one `node-plugin`; its bundle id, manifest
  version and dependency-id set must match `NodePlugin.id()`, `version()` and
  `dependsOn()` so the ADR-011.1 lifecycle graph cannot diverge. The provider-
  owned dependency iterator is independently TCCL-scoped and capped at 256
  entries before parity comparison;
- contribution names are 1–128-character ASCII selectors, exact and case-
  sensitive, with no surrounding whitespace; provider names are fully
  qualified binary Java class names of at most 512 characters;
- selector grammar is also addressable by its runtime configuration surface:
  sink/effect schemes may not contain `.` (the namespace delimiter), and
  signer schemes may not contain `:` (the key-reference delimiter); other
  typed selectors retain exact case and supported punctuation;
- built-in selector names are reserved: state machine `ordered-log`, sequencer
  modes `fixed`/`rotating`, L1 observers `metadata-label`/`address-deposit`,
  and effect executor scheme `webhook`.

## 5. Catalog construction boundary

Catalog construction is phased:

```text
resource discovery
  -> strict manifest parsing/snapshot
  -> isolated per-directory-artifact ServiceLoader type discovery (no provider get())
  -> full installed artifact/provider correlation
  -> bundle policy classification
  -> selected-only terminal loader creation
  -> terminal ServiceLoader provider-type discovery (no provider get())
  -> duplicate/schema/API validation
  -> dependency/version/cycle validation
  -> deterministic bundle order
  -> immutable provider registry + inventory publication
  -> lazy selected-provider construction at required activation point
```

No `NodePlugin` lifecycle callback or typed contribution factory method runs
before the immutable catalog is published. For manifested providers, normal
catalog validation also invokes no provider constructor. Loading a provider
class to obtain its `ServiceLoader.Provider.type()` is trusted linkage, not
activation; static initializers and constructors must still be side-effect
free, but only provider construction invokes plugin metadata methods.

Any structural catalog error is startup-fatal regardless of contribution
tier. Trust-tier failure policy applies after a structurally valid provider is
explicitly activated for a node/chain role.

## 6. Artifact correlation

The runtime correlates manifests, provider types and code sources rather than
trusting a manifest's provider strings in isolation.

For normal JVM plugin-directory JARs:

1. inspect every deterministically sorted JAR for qualified manifests under
   `META-INF/yano/plugins/`;
2. resolve the containing JAR/code-source identity;
3. enumerate supported ServiceLoader provider types in a short-lived loader
   containing that artifact alone (while retaining parent-first Yano API
   identity), including artifacts later filtered by policy;
4. resolve each provider type's code source;
5. require declared providers and ServiceLoader entries to agree within the
   same artifact, without invoking provider constructors;
6. classify whole artifacts by bundle policy and create the terminal shared
   loader from selected snapshot JARs only; an artifact that would mix selected
   and filtered bundles is rejected rather than partially loaded;
7. after provider types have loaded, hash each directory artifact again and
   require the indexed digest to remain unchanged before publication.

For application classpath, exploded development classes and native build-time
inclusion, discovery follows the generated index. The index is build-recorded
artifact-ownership evidence; after packaging flattens those inputs, runtime
validation requires an exact globally unique manifest-to-ServiceLoader
provider mapping but does not claim to re-prove the original artifact-local
code-source boundary. Its strict generation mode requires an explicit Gradle
dependency closure for every discovered bundle id, requires that closure to
include the manifest/provider root, and rejects missing or unknown mappings.
This keeps a future thin provider from silently receiving thin-JAR-only
evidence. Inventory exposes source category and digest mode so an operator can
distinguish this evidence from a directly inspected directory JAR.

The application index is a singleton packaging boundary. Index generation
rejects dependency artifacts that already contain the aggregate-index path,
and runtime discovery rejects multiple index resources rather than attempting
to merge them. Directory artifact indexes are scanned independently and then
validated in the same catalog graph against the one authoritative application
index.

Two manifests declaring the same provider, one artifact exposing multiple
bundle manifests, or a provider appearing under conflicting code sources is
fatal.

## 7. Operator policy and dependency graph

ADR-011.1 policy semantics now apply to bundle ids across all contribution
kinds:

```text
selected(id) = plugins.enabled
            && (allowList is empty || allowList contains id)
            && !denyList contains id
```

Rules:

1. `enabled=false` performs no plugin-directory/provider activation and
   publishes an empty selected plugin catalog. Built-in non-plugin behavior
   remains available as explicit `SYSTEM` contributions and is never governed
   by bundle allow/deny policy.
2. Deny wins over allow.
3. Every non-denied explicit allow id must be discovered or startup fails.
4. Unknown deny ids are harmless.
5. Dependencies never bypass policy. A selected bundle whose required
   dependency is absent or filtered fails validation.
6. Dependency versions are checked against the selected dependency manifest.
   An unbounded manifested dependency may target a legacy `NodePlugin` whose
   compatibility version is opaque; a SemVer-bounded dependency on such a
   version fails with an explicit migration diagnostic.
7. Cycles are fatal and report a stable lexicographic cycle path.
8. Topological ordering uses lexicographic bundle-id tie breaking.
9. Policy never selects individual contributions in v1; it selects the bundle
   as the auditable deployment unit. Per-chain configuration still determines
   which available typed contribution is activated.

## 8. Legacy provider migration

Unmanifested supported ServiceLoader providers remain usable for a documented
migration window in directory and explicit library-compatibility modes.
Strict packaged JVM/native index generation rejects them because no stable
bundle id exists with which to declare and audit their executable dependency
closure. Non-strict compatibility indexes may record them explicitly.
Implicit synthesis is restricted to the explicit library-compatibility mode
when no aggregate index is present; packaged JVM and native modes never infer
an omitted legacy provider.

- the runtime must construct a legacy provider to obtain its selector metadata;
- a legacy `NodePlugin` synthetic bundle id/version/dependencies are its
  existing `id()`, `version()` and `dependsOn()` values, preserving ADR-011.1
  behavior and operator policy;
- a legacy typed provider receives a deterministic synthetic id of the form
  `legacy.<kind>.<normalized-name>.<class-hash>` and version
  `0.0.0-legacy`;
- one typed provider is one legacy bundle; no cross-provider artifact identity
  or dependencies are inferred;
- inventory marks it `legacy=true`, reports provider class and runtime-derived
  digest, and emits one warning per bundle;
- `LEGACY_CLASS` hashes the defining provider `.class` bytes; the class-name
  hash is used only in a typed legacy bundle's stable synthetic id;
- `JAR`, `ARTIFACT_TREE`, and `LEGACY_CLASS` legacy evidence does not cover an
  external dependency closure. This is a visible migration weakness, not a
  build/native deployment guarantee; adopting a manifest is required there;
- duplicate selector names and conflicts with manifested contributions remain
  fatal;
- legacy construction/metadata failure is a catalog discovery failure because
  the runtime cannot identify it safely. Non-process-fatal plugin throwables
  are retained as causes under bounded platform diagnostics; only
  `VirtualMachineError`/`ThreadDeath` escape raw after best-effort cleanup;
- reverse cleanup selects failure precedence before converting a
  non-process-fatal plugin `Error` to its safe activation diagnostic. A later
  assertion/linkage cleanup failure therefore outranks an earlier ordinary
  catalog failure without promoting plugin-controlled text, and every other
  cleanup failure remains attached as context;
- legacy ids, versions and selectors retain their existing grammar but are
  capped at the corresponding manifest lengths, and dependency traversal is
  capped at 256 entries, preventing migration callbacks from amplifying
  memory, logs or fingerprint work without bound;
- adding a valid manifest changes policy identity from the synthetic id to the
  manifest id and must be called out in migration documentation.

Legacy support is compatibility, not a second permanent architecture. Its
deprecation/removal needs a separate decision after at least one release
window and first-party migration.

## 9. Provider registry and activation

The validated catalog owns typed provider handles and exposes immutable,
deterministically ordered lookup by kind/name.

- `PluginManager` receives only catalog-selected `NodePlugin` instances and no
  longer performs an independent ServiceLoader scan in normal runtime
  assembly. Its public legacy constructor retains direct discovery for source
  compatibility, with the same global provider bound and exact reverse-close
  ownership for failed, filtered and selected standalone instances.
- `AppChainSubsystem`, `SignerProviders` and `L1ObservationService` use the
  registry rather than calling `ServiceLoader.load()`.
- built-ins remain explicit code paths and are not synthetic plugin bundles.
- a configured lookup of a missing contribution fails with its kind/name and
  the available sorted names.
- provider construction is cached once per runtime catalog. Typed factories
  must remain stateless; their `create` methods still produce chain-scoped
  instances. One registry-wide identity ledger reserves every non-null raw product
  returned by all seven typed factory kinds: a later invocation or a different
  factory returning an already-owned instance fails activation instead of
  sharing mutable chain state. List products are preflighted and reserved
  atomically. Every otherwise-unclaimed companion in a colliding result is
  terminally reserved under the same ledger lock before reverse cleanup runs
  outside that lock. A blocked or failed close therefore cannot let another
  invocation claim that companion, and cleanup never closes the earlier
  chain's product. Factory result lists are traversed through exactly one
  iterator; every iterator operation has an independent plugin-TCCL scope. If
  traversal fails after yielding products, the already-observed fresh
  products are terminally reserved and reverse-closed, while products owned by
  an earlier invocation are neither claimed nor closed. Reservations are weak
  identity references: a product retained by a factory/runtime remains
  collision-detectable, while unreachable products from completed restart
  cycles do not accumulate for the life of the process.
- one raw provider identity may own exactly one catalog entry and may not also
  be claimed as a factory product. Concurrent activation has one owner;
  waiters observe the identical cached success or fixed activation failure,
  and arbitrary plugin `Throwable`s cannot strand an entry in `ACTIVATING`.
- the first construction verifies manifest selector name and, for
  `node-plugin`, lifecycle metadata parity.
- after activation validates a configured product identity, the runtime
  snapshots it for that product lifetime: state-machine, sequencer, executor
  and sink ids; configured observer id; and signer 32-byte public key (with a
  defensive byte copy and derived lower-case hex). Configuration binding,
  health, logging and later identity reads use the snapshot rather than
  re-entering a stateful plugin identity callback.

Activation failure follows the parent trust table:

- configured consensus or privileged-local provider construction/config/init
  failure is fatal to that chain/node startup;
- auxiliary sink factory failure is isolated and recorded for ADR-011.4 health
  reporting. Before any sink or executor identity callback, the consumer scans
  the complete factory result and claims every distinct non-null product. A
  null/duplicate/malformed middle entry or later validation failure rolls back
  every claimed unique product exactly once in reverse order, including valid
  tail entries, and publishes no partial runner/runtime. A fatal cleanup
  `Error` wins only after the remaining cleanup callbacks run. Sink cursor
  migration is also staged without writes: every legacy-cursor callback and
  read completes first, then initial cursors are committed in one atomic batch
  immediately before runner publication;
- runtime effect-executor failure follows ADR-010 parking/idempotency behavior,
  but an explicitly configured executor factory that cannot initialize fails
  startup;
- unused manifested typed providers are not constructed merely because their
  bundle is installed and selected.

## 10. Configuration namespace

The canonical external namespace is:

```text
yano.plugins.bundle."<bundle-id>".*
```

`PluginContext.config()` remains the immutable global compatibility map during
the legacy window. ADR-011.2 adds an immutable owner-stripped bundle view so a
manifested `NodePlugin` does not need to inspect another bundle's settings.
The SmallRye quoted map-key segment is parsed as one exact owner, including
dots and hyphens, rather than by dot-prefix matching. The earlier
`yano.plugins.bundle[<bundle-id>].*` spelling remains an exact, deprecated
compatibility alias. If both spellings define the same owner-stripped key with
different values, startup fails without reporting either value. Overlapping
ids, policy exclusion, and removal of a child bundle therefore cannot reassign
its stale credentials to a parent bundle. The compatibility map is deprecated
only after first-party plugins and the scaffold use the bundle view.

SmallRye environment variables use a reversible UTF-8-hex owner token:

```text
YANO_PLUGINS_BUNDLE_HEX__636F6D2E6578616D706C652E70726F647563742D70617373706F7274__ENDPOINT_URL
```

The packaged mapper retains the original environment name as well as
SmallRye's best-effort dotted name. Catalog construction decodes the original
name to one exact bundle id and publishes it only to that selected owner.
The obsolete punctuation-to-underscore owner spelling is rejected even when
only one current bundle matches: it cannot distinguish a removed `a-b` owner
from a later `a.b` owner. Environment suffix underscores map to owner-local dots
(`ENDPOINT_URL` becomes `endpoint.url`). An exact quoted property from another
source wins when a suffix itself needs punctuation that the environment format
cannot represent. Unknown, removed, denied, malformed, or ambiguous
environment owners are never prefix-reassigned.

Typed app-chain factories retain their existing per-chain, prefix-stripped
configuration (`machines.*`, `sinks.*`, `effects.executors.*`, and so on).
Secrets are not copied into inventory, fingerprints, logs, health details or
metrics labels.

## 11. Class-loader model and ownership

### 11.1 JVM v1

- packaged JVM startup with plugins enabled requires exactly one authoritative
  embedded aggregate index; library embedders choose the separately named
  compatibility mode when retaining zero-index legacy discovery;
- strict packaged JVM/native indexes reject embedded legacy providers and
  require `ARTIFACT_CLOSURE` evidence for every manifested bundle;
- the packaged node captures every regular plugin-directory JAR through a
  no-follow, size-bounded channel into a private runtime-owned snapshot
  directory before hashing, catalog inspection or class loading. The copy is
  limited to 1,024 total source-directory entries, 1 GiB per JAR, 256 JARs and
  4 GiB in aggregate per loader, and its streaming SHA-256 must equal a second
  hash of the completed snapshot. The total-entry bound includes ignored
  non-JAR entries, so a directory cannot hide an unbounded scan behind a small
  JAR count.
  Per-JAR, count and aggregate-byte policy is checked before the private
  directory is created (with the streaming per-JAR cap also rejecting source
  growth during capture). Directory identity/mtime brackets the complete capture, while each
  source file key, size and mtime is checked before and after its copy and
  again after every JAR has been captured, rejecting an ordinary mixed
  multi-JAR deployment view;
- every captured JAR is structurally scanned and type-correlated in an
  artifact-isolated validation loader before policy-filtered bytes can enter a
  shared lookup space. The final shared parent-first `URLClassLoader` contains
  selected snapshots only, in complete-SHA-256 order. Filtered artifacts remain
  in inventory but cannot shadow a selected class or resource;
- isolated provider-type traversal retains a per-artifact bound and also spends
  one deployment-wide budget shared by every captured artifact. The terminal
  selected-loader ServiceLoader traversal has its own independent global bound;
  splitting hostile metadata across many JARs cannot multiply validation work;
- snapshot leaf names are the complete SHA-256 hex plus `.jar`, so inserting or
  renaming a filtered artifact cannot change a selected provider's observable
  CodeSource/resource leaf while leaving the selected fingerprint unchanged.
  Byte-identical duplicate deployments are rejected rather than source-path
  disambiguated. Atomic replacement of an operator-owned source after capture
  cannot change catalog, lazy class or resource bytes in the running process;
- `enabled=false` keeps the handle on the immutable host/application parent and
  does not scan or activate directory bytes. Host bootstrap resources such as
  genesis files always use that host loader, never the executable plugin
  projection;
- a legacy directory provider has no declarative bundle id before its callback
  is invoked. It remains migration-compatible when no allow/deny filters are
  configured, but filtered legacy-directory deployment fails with a manifest
  migration diagnostic rather than putting ambiguously selected bytes on the
  loader;
- parent-first loading preserves one Yano API type identity;
- the parent is a trusted, clean application boundary. Operator-controlled
  directory plugin JARs must not also appear on the host classpath. Parent-first
  loading means host classes and ordinary resources may shadow selected plugin
  definitions; the directory fingerprint therefore has meaning only together
  with unchanged application-build provenance. Allow/deny URL projection is
  not isolation from a contaminated parent and is not a sandbox;
- catalog validation rejects plugin JARs containing
  `com/bloxbean/cardano/yano/api/**` classes to prevent accidental API
  shadowing even though parent-first loading would mask them;
- plugin artifacts and every artifact assigned to a strict build-time bundle
  closure may not declare a JAR-manifest `Class-Path`. All executable
  dependencies must be captured as explicit catalog artifacts; otherwise the
  JVM could resolve mutable, unsnapshotted code outside the recorded closure;
- duplicate JAR canonical paths are rejected;
- a symbolic-link plugin directory or symbolic-link directory entry is
  rejected rather than followed;
- a closeable `PluginRuntimeEnvironment` owns the loader, private artifact
  snapshots, catalog and provider registry and is passed explicitly through
  runtime assembly. It closes the `URLClassLoader` before deleting snapshots
  and also removes snapshots after capture or activation failure;
- every private snapshot holds a marker-qualified exclusive OS lease. A later
  directory-loader startup may reclaim an abnormal-exit orphan only after a
  creation-race grace period and only when it can acquire that lease; active,
  young, unmarked or unverifiable directories are never removed. Reclamation
  inspects at most 256 candidate directories per startup. A candidate must
  retain the flat runtime-owned shape (the lease plus at most 256 temporary
  five-digit or final 64-hex `.jar` snapshot files); nested, symbolic-link,
  unexpected-name or oversized candidates are skipped rather than recursively
  traversed;
- the environment closes provider instances and its loader after every
  plugin/contribution has stopped and closed, including failed assembly;
- environment close is a linear, exactly-once transition. Concurrent and later
  callers observe the same memoized terminal failure, and a loader claimed by
  an environment cannot be closed through its public handle while still owned;
- one registry-wide admission tracker covers provider construction/metadata
  and every callback on all seven returned typed SPI products. Runtime stop
  seals new callback admission only after subsystems stop accepting work,
  waits callbacks admitted before the seal, then waits explicit engine,
  effect-runtime and sink-product terminal signals before stopping
  `NodePlugin`, closing providers or closing the shared plugin loader. Restart
  reopens admission only after those barriers complete and deliberately keeps
  retained state-machine, sequencer and signer facades valid;
- ordinary callback admission is sealed before teardown, but a distinct
  host-only `NodePlugin` teardown admission class remains available so
  `stop()`/`close()` cannot be rejected by the fence they are responsible for
  draining;
- app-chain `stop()` remains bounded when plugin code ignores interruption.
  Sink delivery, engine execution and effect execution expose separate
  eventual completion barriers. RocksDB close and same-instance restart wait
  for every ledger-using engine/sink callback; plugin lifecycle/class-loader
  teardown additionally waits all typed callbacks and product closes. Truly
  bounded unload of hostile in-process code would require process isolation
  and is outside v1;
- a product-lifetime barrier becomes terminal only after its actual sink or
  executor `close()` callback returns. Thread-factory or `Thread.start()`
  failure is not termination evidence: a platform fallback owns the callback,
  or the last available caller performs it synchronously. Exact-once callback
  claims prevent a hostile/ambiguous hand-off from closing twice, and fatal
  hand-off errors are rethrown only after the truthful terminal signal;
- per-instance terminal-close accounting prevents the `NodePlugin` lifecycle
  manager and provider registry from closing the same instance twice, while
  still closing catalog-owned instances untouched by partial initialization;
- a scoped boundary binds the shared plugin loader as the thread context class
  loader around provider construction/metadata, every `NodePlugin` lifecycle
  callback, every typed app-chain factory callback and every callback on the
  returned state machine, sequencer, L1 observer, signer, effect executor and
  finalized sink. It restores the caller's TCCL in the same `finally` block on
  success, checked failure, runtime failure and fatal error;
- the same boundary covers platform-managed callbacks registered through
  `PluginContext`: event listeners and subscription filters, shared-scheduler
  tasks, and storage-filter priority/accept calls. The plugin-facing EventBus
  and scheduler views are non-owning and reject close/shutdown rather than
  terminating shared runtime resources. Listener and scheduler resources are
  accepted only during `NodePlugin.start()`/RUNNING, are stamped with that
  start generation, and are canceled at stop. Plugins recreate them on every
  start; a wrapper queued before stop is rejected if it reaches a worker after
  a later generation resumes;
- app-chain L1 subscriptions carry the same kind of monotonically unique
  resource-generation token. A callback admitted before stop retains its old
  lease until it returns; a retired callback delayed between the event-bus
  active check and listener invocation cannot enter a replacement generation
  or mix old observer/anchor products with the new engine, pool or L1 window;
- typed facades do not acquire an extra ownership role: the catalog retains
  raw provider identity for exact-once close accounting, while one
  registry-wide product reservation ledger prevents cross-chain and
  cross-factory identity reuse. Duplicates within one list share one facade so
  the existing runtime validator rejects and closes that product exactly once;
- plugin-owned maps/lists/sets returned by callbacks are defensively snapshotted
  while plugin TCCL is installed, so lazy iteration cannot escape the boundary.
  Status values are restricted to JSON-compatible null/boolean/string/finite
  primitive-number/map/list trees with no cycles or arbitrary beans: depth is
  capped at 8, every map/list at 256 entries, the whole tree at 4,096 values,
  each key/string at 4,096 characters and aggregate text at 65,536 characters;
- effect/sink factory callbacks return at most 256 products. One L1-observer
  callback returns at most 4,096 observations; each result is defensively
  copied, carries the configured observer id (at most 160 characters), callback
  slot and 32-byte block hash, has a 32-byte transaction hash and at most a
  1 MiB claim, and all claims from the callback total at most 4 MiB;
- arbitrary objects exchanged through the inter-plugin service registry and
  threads a plugin creates itself remain plugin-owned contracts; the runtime
  cannot safely infer their callback or ownership boundaries. Plugins needing
  runtime-managed TCCL propagation use `PluginContext.scheduler()`;
- install/remove/upgrade requires process restart;
- no isolation from conflicting third-party dependency versions is promised;
- development/release JVM-distribution jobs and the JVM-image release job run
  `packagedJvmPluginCatalogSmoke` before uploading a zip or preparing/publishing
  an image, so the final uber-JAR must activate the real directory fixture
  through the catalog-owned facades.

The shared-loader limitation is visible in inventory. Per-bundle child-first
loaders remain a possible future hardening only after dependency and
ServiceLoader behavior are proven with real plugins.

### 11.2 Native image

- plugin artifacts are fixed at build time; directory loading is disabled;
- native startup with plugins enabled requires exactly one authoritative
  embedded aggregate index and rejects every unindexed ServiceLoader provider;
- the application build generates the aggregate index before augmentation;
  it supplies a strict per-bundle mapping from the root artifact to the actual
  app-resolved executable dependency closure and rejects any unmapped bundle;
  manifests, index, provider classes and ServiceLoader resources must be
  included in the native image;
- the `plugin-catalog` library carries embedder-safe native metadata with exact
  resource patterns for the aggregate index and every supported plugin service
  descriptor. Its codec reflection metadata and the application's generated
  provider-constructor reflection metadata use distinct native-image
  namespaces so neither artifact can overwrite the other;
- a build/native test validates the same catalog and fingerprint inputs as JVM
  classpath mode;
- startup emits one secret-free `YANO_PLUGIN_CATALOG_PROVENANCE` record with
  the embedded index SHA-256 and selected-catalog fingerprint. The native smoke
  task regenerates the current JVM index, computes both expected values, and
  requires an exact record match; an executable built with different catalog
  inputs therefore cannot pass merely by starting successfully with otherwise
  valid diagnostics. This is catalog provenance, not a digest of unrelated
  application code;
- release distribution workflows finish the native zip before running that
  smoke task, so the executable copied into the uploaded archive is the one
  whose provenance and startup behavior were verified;
- a non-published build-only conformance bundle supplies one inert provider for
  every typed app-chain SPI plus a lifecycle probe. Unit tests resolve and
  exercise its products through the catalog-owned facades. Packaged JVM and
  native smoke tasks then auto-start an isolated one-member, no-peer app chain
  with all applicable fixture selectors and require structured running status
  for the signer, state machine, sequencer, observer, sink and effect executor. Each
  fixture factory poisons its callback TCCL before returning. Each returned
  product poisons its first callback and rejects a missing TCCL on every later
  callback. Catalog facades restore and reinstall the loader at both
  boundaries; a raw product leaks the first poison and cannot pass. The lifecycle
  query/domain checks exercise the seventh typed SPI; the lifecycle probe
  resolves no other SPI and emits only its own catalog-activation marker.
  Provider constructors still depend on retained ServiceLoader resources and
  generated reflection metadata.
  The fixture is included only with
  `-PincludeNativePluginConformanceFixture=true`, and a dedicated Linux CI job
  builds and tests that disposable image without publishing it;
- dropping a JAR next to a native binary is unsupported. If a configured
  plugin directory contains JARs, native startup emits a prominent ignored-JAR
  diagnostic after a bounded inspection of at most 256 directory entries;
- native catalog discovery must not fall back to reflection-only classpath
  scanning.

## 12. Digests, fingerprint and inventory

The manifest never supplies its trusted digest.

### 12.1 Code digest

- for a JAR, SHA-256 covers the complete artifact bytes;
- for an exploded artifact, `ARTIFACT_TREE` hashes its complete sorted input
  tree using relative path plus bytes, not only declared provider classes;
- for build-time JVM/native inclusion, `ARTIFACT_CLOSURE` hashes, per bundle,
  the complete root artifact plus the complete app-resolved executable
  dependency artifacts. Each constituent is first hashed as a JAR or canonical
  artifact tree; those `(mode,digest)` values are sorted and length-delimited
  under a versioned domain before the closure hash is calculated. Input order,
  filesystem path and filesystem modification time do not participate;
- build-time generation requires an exact mapping for every manifested bundle,
  requires the root artifact in its mapping, rejects closure artifacts outside
  the indexed runtime classpath, and rejects unmanifested legacy providers;
- the app generator deliberately reruns instead of trusting Gradle's global
  runtime-file up-to-date snapshot: dependency-edge ownership can change while
  the global artifact byte set remains identical;
- inventory records digest mode (`JAR`, `ARTIFACT_TREE`,
  `ARTIFACT_CLOSURE`, or `LEGACY_CLASS`) so operators know the evidence scope;
- unreadable declared code bytes are fatal for manifested bundles.

### 12.2 Catalog fingerprint

The selected fingerprint is SHA-256 over a canonical length-delimited encoding
of, in bundle topological order:

- manifest schema, bundle id/version and Yano API range;
- sorted dependency ids and version bounds;
- sorted contribution kind/name/provider triples and derived trust tiers;
- legacy flag, byte-evidence digest and digest mode.

No map iteration order, JSON object order, filesystem path, timestamp,
configuration value or activation status participates. Rebuilding equivalent
catalog input produces the same fingerprint. Changing selected provider code,
any artifact in its declared build-time executable closure, manifest metadata,
or policy selection changes it.

### 12.3 Inventory

The read-only inventory exposes:

- catalog fingerprint and plugin API major;
- discovered bundle id/version plus explicit `SELECTED`, `DENIED`, or
  `NOT_ALLOW_LISTED` policy status;
- legacy flag, source category (classpath/directory/native), digest and mode;
- the selected dependency-first bundle order and each bundle's declared
  dependency ids;
- contribution kind, selector, provider class and trust tier;

Only a completely compatible and structurally valid catalog is published.
Invalid catalogs fail startup with bounded operator diagnostics rather than
publishing partially valid per-bundle entries.

It does not expose local absolute paths, class-loader object strings,
configuration values, key references, credentials or stack traces.

## 13. Runtime integration and publication

Runtime assembly order becomes:

```text
own plugin class loader
  -> build/validate immutable bundle catalog
  -> initialize selected NodePlugin lifecycle hooks (ADR-011.1)
  -> construct app-chain subsystems from catalog provider registry
  -> construct sync validation with initialized customizers
  -> kernel start
```

Within kernel start, enabled app chains activate after sync is available but
before startup publication, the successful-start log, or maintenance release.
Reverse stop first enters the shutdown maintenance boundary, then stops app
chains, then sync/storage/plugins. An app-chain activation failure therefore
cannot publish a successful node start, and teardown cannot run outside the
runtime shutdown fence.

Catalog/policy failure occurs before app-chain stores, peers, effect executors,
sinks, sync or server startup. Existing construction cleanup must close any
catalog-owned provider instances and the plugin class loader on a later
assembly failure.

Runtime/library failure objects may retain raw plugin causes so transactional
cleanup and embedders can inspect them. The packaged Quarkus application is a
separate disclosure boundary: before a startup-fatal plugin failure escapes its
startup observer, it replaces the complete wrapper/cause/suppressed graph with
a cause-free `PluginStartupException`. Its message contains only a
platform-normalized failure type and, for lifecycle failures, the fixed
lifecycle phase. Quarkus can therefore render the full escaping stack without
printing a plugin message, identifier, configuration value or cleanup cause.

App-chain start is transactional as well. Opening a ledger/engine does not set
the subsystem to `running`; if observer, sink, effect or later activation
fails, the start attempt unwinds every acquired resource in reverse order.
Cleanup cannot be gated only by the public `running` flag. A test must reopen
the same RocksDB path immediately after injected activation failure.
Cleanup catches plugin `Throwable`s, continues through engine and ledger
release, and attaches cleanup failures to the primary startup failure rather
than allowing a faulty close callback to strand the database lock.

Periodic sink/effect/anchor tasks contain ordinary plugin `Throwable`s,
including assertion/linkage failures, so one bad callback does not silently
cancel every later scheduled tick. VM-fatal errors and `ThreadDeath` are
classified and rethrown before diagnostic allocation. Sink and anchor
activation/runtime/close diagnostics retain exception class names
rather than plugin messages, which may contain credentials. Anchor
confirmation records the observed L1 inclusion before resolving
callback-owned block data. A transient lookup/write failure therefore retains
the exact pending identity and later tick/force processing completes it without
requiring the L1 event to replay or resubmitting an already-confirmed
transaction. Confirmation summary plus a bounded journal commit atomically;
restart/deep-rollback recovery restores the newest surviving entry or
conservatively resets to genesis when the target predates retained history.
Script-anchor identity/bootstrap state follows the same atomic rollback rule.
Effect attempt errors retain ADR-010's node-local operational reason semantics.

Dedicated peer transport publishes its client and protocol agents as one
tuple before the connection starts, but does not call that tuple ready merely
because Yaci reports a running session (that state also covers connection
retry). Readiness begins only after protocol 100 receives `MsgInitAck`; replay
drain and readiness publication are one atomic hand-off with submission.
Disconnect clears readiness, and a connection epoch prevents a late callback
from reactivating a retired/reconnecting agent. This prevents Yaci's
post-connect agent reset from erasing messages submitted during a retry and
prevents status from advertising protocol availability too early. Diffusion
remains ADR-005's ephemeral/best-effort gossip after hand-off: this does not
claim reconnect at-least-once delivery for an agent message not yet
acknowledged by its peer.

Catalog inventory becomes visible only after complete structural validation.
Later lazy activation status is a separate mutable operations view introduced
by ADR-011.4; it cannot rewrite the immutable installed/selected fingerprint.

## 14. Migration of first-party plugins

ADR-011.2 adds manifests and catalog conformance tests for:

- built-in logging `NodePlugin`;
- app-chain stdlib state-machine providers;
- ZK state-machine providers;
- Kafka finalized-stream sink;
- Cardano effect-executor factory;
- the plugin scaffold/template.

ServiceLoader files remain mandatory. A first-party manifest/ServiceLoader
mismatch fails tests and startup. Documentation shows both files and explains
bundle-wide policy identity. Packaged JVM/native build-time inclusion rejects
unmanifested providers because it cannot assign their dependencies to a safe
closure owner. During the migration window, legacy providers remain available
only as self-contained JVM directory JARs or through explicit
`LIBRARY_COMPATIBILITY` embedding mode.
The application index and first-party bundle verification tasks also require
release-coupled manifest versions to equal the declared Gradle release version;
the independently versioned built-in logging plugin has an explicit pinned
expectation.

Published plugin examples must be deployable artifacts, not thin JARs that
silently depend on an undeclared host classpath. First-party T3 extensions
retain a thin normal artifact for explicit JVM/native build-time inclusion and
publish a reproducible `bundle` classifier for plugin-directory deployment.
They are omitted from the stock app; a build may include them before
index/reflection generation with `-PincludeFirstPartyPluginBundles=true`. The
bundle shades runtime-only dependencies, merges ServiceLoader descriptors and
excludes Yano API classes. The app build declares each known build-time bundle
project and derives its closure from the actual app resolution graph; the T3
entries also include their host-provided compile-only API dependencies. A new
manifested classpath bundle fails index generation until that explicit mapping
is added. A build-time-included bundle must not also be installed from the
directory because a duplicate bundle id is fatal. The Kafka sink, Cardano
effect executor and ZK extension bundle classifiers are scanned and launched
from an otherwise clean JVM plugin directory in integration tests.
For directory deployment, one manifested bundle is one self-contained JAR;
placing its thin artifact beside dependency JARs does not combine those files
into one catalog bundle and is unsupported.

The seven-SPI native conformance fixture lives under
`appchain/fixtures/appchain-plugin-conformance`. It is neither a supported
plugin nor a distribution input: its Maven publication is removed, its publish
tasks are disabled, and default app packaging tests require its absence. The
native build and smoke invocations must both opt in with the same
`includeNativePluginConformanceFixture` value so provenance comparison cannot
mistake a stock and fixture-bearing executable.

## 15. Security and operational rules

1. Plugin JARs are trusted in-process code; a manifest is metadata, not a
   sandbox or signature.
2. Operators control directory contents, file ownership and artifact
   provenance; allow/deny policy is an additional activation boundary.
3. Catalog capture rejects symbolic-link artifacts/entries, opens source JARs
   with no-follow semantics, and checks directory plus per-entry identity and
   mutation metadata around the capture. All subsequent inspection and loading
   uses the verified private snapshot, isolating the running catalog from
   ordinary source replacement.
   This is an integrity boundary for an operator-controlled filesystem, not a
   sandbox against a hostile same-identity process or trusted in-process plugin
   code racing or modifying private filesystem state.
4. Catalog inspection rejects JAR-manifest `Class-Path` indirection for plugin
   artifacts and strict bundle-closure inputs, so provenance and immutable
   capture cannot be bypassed through an external helper JAR.
5. Catalog parsing is bounded and runs before plugin code activation.
6. Provider/class names and digests may be public inventory; local paths and
   secrets may not.
7. Consensus-role operators compare selected fingerprints before deploying a
   plugin/config upgrade, but fingerprint equality does not prove deterministic
   state configuration or authorize an upgrade height.
8. Artifact signatures/provenance attestation remain future hardening.

## 17. Test plan

### 17.1 Manifest parsing and validation

1. Valid minimal/full v1 manifests parse to defensive immutable snapshots.
2. Oversize, malformed, duplicate-key, unknown-field and unsupported-schema
   manifests fail without dumping content.
3. Invalid ids/names/classes/SemVer/API/dependency ranges fail deterministically.
4. Duplicate bundle ids/provider declarations and duplicate `(kind,name)` keys
   within one manifest fail; duplicate names across bundles remain available
   for policy selection.
5. Reserved built-in names fail.
6. Missing/undeclared ServiceLoader entries and artifact-correlation mismatch
   fail before provider construction.

### 17.2 Policy, compatibility and graph

1. Disabled packaged/native mode requires no index and activates no provider
   code.
2. Empty allow selects all; non-empty allow selects exact bundle ids; deny wins.
3. A policy-excluded `(kind,name)` alternative is retained in inventory and
   does not conflict; selecting two alternatives fails with the same sorted
   diagnostic in packaged JVM, native and directory modes.
4. Missing non-denied allow ids fail; unknown deny ids do not.
5. Filtered/missing/version-incompatible dependencies fail and never bypass
   policy.
   A SemVer-bounded dependency on an opaque legacy `NodePlugin` version fails
   deterministically rather than being treated as compatible.
6. Stable cycle paths and lexicographic topological order are independent of
   resource/JAR/ServiceLoader order.
7. API-major mismatch fails before provider construction.

### 17.3 Provider registry and legacy

1. Every supported kind resolves through the registry with no post-catalog
   `ServiceLoader.load()` call.
2. Manifest selector mismatch fails on first provider construction.
3. Unused manifested providers are not constructed.
4. Configured consensus/privileged provider construction failure is fatal;
   auxiliary sink failure is isolated and observable.
5. Legacy providers receive stable synthetic descriptors/digests and warnings.
6. Legacy duplicate selector names remain fatal.
7. Manifested `NodePlugin` metadata parity and ADR-011.1 lifecycle order are
   enforced.
8. Normal shutdown, partial initialization failure and metadata-validation
   failure close every constructed `NodePlugin` exactly once.
9. Catalog-projected `NodePlugin` order preserves transitive dependencies that
   pass through typed-only bundles.
10. Explicitly enabled effect execution fails startup on invalid settings,
    missing executors or runtime initialization failure; it never degrades to
    an unannounced non-executor node.
11. Terminal registry close atomically rejects cleanup registrations that race
    the closing transition and waits every registration accepted beforehand.
12. Callback admission seals on stop, waits an interrupt-resistant typed
    callback before provider close, rejects late calls, and reopens retained
    state-machine/sequencer/signer facades only after cleanup quiescence.
13. Concurrent activation and environment-close callers cannot duplicate
    construction/teardown or strand waiters; all callers observe the same
    cached terminal result. Assertion errors and sneaky checked callback
    failures remain fixed, secret-safe activation diagnostics, while process-
    fatal errors escape only after cleanup state is published.
14. The bracketed configuration owner remains exact for overlapping bundle
    ids; denying or removing a child never exposes its values through a
    selected parent bundle.
15. The packaged-application startup boundary renders an adversarial plugin
    failure whose wrapper, message, identity fields, cause and suppressed
    failures all contain a sentinel secret; the escaping exception retains a
    useful normalized failure type/phase, has no cause or suppressed failures,
    and its full rendered stack contains no sentinel.

### 17.4 Digest, classloader and packaging

1. Shuffled JAR listing produces sorted loader URLs and identical fingerprint.
2. Equivalent manifest JSON field order produces the same canonical
   fingerprint.
3. Manifest/provider/root-code changes alter digest/fingerprint. A byte-only
   change in one mapped build-time dependency changes that bundle's evidence
   while an unrelated bundle closure remains stable.
4. A plugin JAR packaging Yano API classes is rejected.
5. Loader close/restoration occurs on normal shutdown and assembly failure.
6. Parent-shadowed directory provider classes and symlink artifacts are
   rejected; atomic source replacement before or after catalog publication
   leaves the captured digest, lazy class bytes and resources unchanged.
   A lower-digest denied JAR with colliding helper-class/resource names is
   type-validated and retained in inventory but absent from terminal loader
   URLs; allow- and deny-filtered launches preserve the selected fingerprint,
   helper/resource result, ordinary and supported-service resource enumeration,
   and content-addressed CodeSource leaf. Its static initializer and constructor
   are never invoked. Failure both before and after selected-loader creation
   restores TCCL and deletes every snapshot. The deployment-wide isolated
   provider budget accepts its exact limit and rejects the next provider across
   an artifact boundary without exposing a local path.
7. JVM directory, normal classpath and native build-time inclusion produce
   conformant catalogs for the same fixture.
8. Quarkus uber-JAR retains and indexes at least two distinct bundle manifests.
9. Packaged JVM and native startup auto-start a real isolated app chain through
   all seven typed fixture providers, prove facade mediation with an adversarial
   TCCL handoff, assert chain-scoped structured product status, and diagnose a
   dropped native directory JAR as unsupported.
10. The calling thread's TCCL is unchanged after successful and failed catalog
   construction, provider construction/metadata, `NodePlugin` lifecycle and
   every callback on all seven factory products (including checked callback
   failures and duplicate objects in executor/sink lists).
11. A real isolated ZK bundle creates the configured state machine through its
   default-ServiceLoader ZeroJ verifier discovery, instantiates the packaged
   verifier backends, restores the caller TCCL and leaves its closed JAR
   replaceable/deletable.
12. Event listeners/filters, scheduled and submitted tasks, and storage-filter
    callbacks receive plugin TCCL and restore the dispatch/worker TCCL even if
    plugin code changes it; plugin views cannot close the EventBus or shut down
    the shared scheduler. Init/between-cycle callback-resource registration is
    rejected; blocked EventBus/scheduler registration handoffs participate in
    the stop barrier; and a queued wrapper from an old generation is skipped
    after restart.
13. Strict build-time generation rejects missing/unknown/rootless closure
    mappings, unmanifested providers, closure paths outside the actual runtime
    classpath, and dependency artifacts carrying a nested aggregate index.
14. Reassigning the same resolved dependency artifact from one bundle graph to
    another regenerates the index and changes only the affected closure inputs;
    Gradle cannot reuse an output based solely on the unchanged global files.
15. Renaming/swapping plugin JAR filenames with unchanged bytes cannot change
    overlapping dependency resolution or the selected fingerprint.
16. Strict packaged/native startup rejects indexed legacy providers and any
    embedded bundle without complete artifact-closure evidence.
17. Catalog-library native metadata exactly retains the aggregate index and
    all supported service descriptors, does not claim the app-generated
    reflection path, and generated provider reflection contains exactly the
    indexed provider constructors with no stale or extra entry.
18. Native smoke compares the executable's embedded-index SHA-256 and catalog
    fingerprint with freshly generated packaged-JVM expectations and rejects a
    stale executable even when health and ignored-directory diagnostics pass.
19. Directory loading rejects the 1,025th total entry, a single JAR above
    1 GiB, the 257th JAR and aggregate snapshot bytes above 4 GiB before
    creating a private snapshot; streaming capture also rejects a source that
    grows through the per-JAR cap.
20. Best-effort orphan reclamation removes only a snapshot older than the
    10-minute grace period that has the exact marker, permits exclusive lease
    acquisition and passes an identity recheck. Active leased, young, unmarked
    and changed lookalikes remain untouched; only 256 candidates are inspected,
    and a nested, symbolic-link, unexpected-name or oversized candidate is
    skipped without recursive traversal.
21. Provider construction, selector, each `NodePlugin` metadata method and
    failed-candidate close each receive a fresh TCCL boundary even when the
    preceding callback replaces its thread context loader.
22. Plugin-owned status trees enforce depth, per-container, node, per-string
    and aggregate-text limits and reject cycles, non-finite numbers and
    arbitrary beans; effect/sink factory lists stop after 256 products; and one
    L1-observer callback stops after 4,096 observations, a 1 MiB claim or 4 MiB
    aggregate claims while enforcing callback id/slot/hash consistency.
    Infinite or duplicate-key iterators fail deterministically; factory
    overflow reverse-closes every newly captured unique product.
23. Native chain-scoped REST subresources retain their endpoint metadata and
   return the same configured-chain status used by the JVM conformance smoke.
24. A plugin root or any dependency assigned to its strict build-time closure
    that declares JAR-manifest `Class-Path` is rejected without exposing the
    referenced path; an unrelated application dependency remains compatible.

### 17.5 Integration and migration

1. First-party manifests match their ServiceLoader files.
   Release-coupled manifest versions also match the declared Gradle release
   version, while explicitly independent plugin versions match their pinned
   expectation.
2. Bundle allow/deny policy governs state machines, effects and sinks as well
   as `NodePlugin`.
3. A configured filtered provider is unavailable with a clear sorted
   diagnostic.
4. Existing unmanifested plugin fixture still works through JVM directory and
   explicit library-compatibility legacy modes, while strict build-time mode
   rejects it with the manifest/directory migration diagnostic.
5. Multi-chain creation reuses provider factories but creates fresh
   chain-scoped contribution instances.
6. Constructor failure releases plugin loader, providers, RocksDB and
   schedulers.
7. Injected app-chain activation failure permits immediate reopen of the same
   database path.
8. The documented Kafka and Cardano-effect bundle layouts launch with all
   third-party dependencies present.
9. Mixed-case and delimiter-edge selectors round-trip through every typed
   contribution's actual configuration surface.
10. Sink and executor results shaped like
    `[valid, malformed/null/duplicate, valid-tail]` pre-own the complete fresh
    batch, close every unique product exactly once in reverse order and publish
    no partial runner/runtime. Isolated sink and startup-fatal executor paths
    preserve their different activation policy; fatal cleanup errors retain
    precedence only after remaining closes. A later sink cursor-preparation
    failure leaves no inactive cursor, and a fatal sink close during later
    startup rollback cannot prevent immediate ledger reopen. A failed cursor
    preparation does not reserve its ids or poison a later independent sink
    scheme using the same id.
11. Two factories returning the same raw executor or sink cannot share it: the
    second activation is rejected, its otherwise-unclaimed products are
    terminally reserved and closed, and the first chain-owned product remains
    open. A third invocation racing a deliberately blocked close cannot claim
    a rejected companion or cause a double close. Fatal cleanup errors retain
    fatal precedence after the remaining cleanup callbacks run.
12. Every product identity callback is evaluated once; selector/product
    mismatch fails activation, and downstream configuration, logging, health
    and identity reads use the immutable snapshot.
13. A factory list whose iterator yields products and then fails terminally
    reserves/reverse-closes only fresh captured products, preserves fatal
    precedence, and rejects a later return of a still-retained product.
14. Interrupt-resistant sink delivery, effect execution/capability checks and
    engine state-machine execution cannot overlap product/provider/loader or
    RocksDB teardown. Public stop stays bounded, same-instance restart remains
    closed until quiescence, and no successful sink cursor write occurs after
    the shutdown boundary.
15. Repeated stop/start cycles prune completed cleanup futures and unreachable
    weak product reservations while retained core product facades continue to
    work.
16. App-chain activation occurs before node-start publication and maintenance
    release; reverse teardown enters the shutdown boundary before stopping app
    chains, and an activation failure cannot emit a successful-start signal.
17. A sync-stop/close `Throwable` or failed UTXO drain marks the runtime unsafe
    before plugin teardown, skips apply-owned plugin/EventBus/ledger resources,
    still closes schedulers/storage in unsafe mode, and rejects same-instance
    restart.
18. Paired server/transaction and producer/nonce teardown attempts both sides
    after one fails and preserve process-fatal precedence without skipping
    later kernel stages.
19. Sink/effect cleanup thread-factory and `Thread.start()` failures cannot
    complete a product-lifetime future before the actual close callback ends;
    fallback races retain exact-once close, fan out independent products and
    preserve process-fatal precedence.
20. Metadata and script-anchor submitter/signer/block-provider assertions are
    class-only health diagnostics, do not cancel later scheduled ticks and do
    not leak callback messages. A single observed confirmation whose local
    block lookup fails completes on a later tick without L1 replay or duplicate
    submission. Null block lookup fails closed; confirmation journal/summary,
    script identity and rollback updates are atomic. Restarted multi-anchor
    rollback restores the newest survivor, while rollback older than all 256
    retained entries conservatively clears every summary field to genesis.
21. Two non-process-fatal legacy cleanup `Error`s retain reverse-order winner
    precedence and both remain reachable after conversion to the safe public
    activation exception.
22. An L1 callback admitted before stop keeps its captured generation services
    and fences restart until return. A retired callback delayed before
    admission cannot acquire a replacement generation token, mutate the new
    L1 window/pool or invoke either generation's plugin products.
23. A local app message submitted while TCP is retrying, before the remote
    server exists, remains outside Yaci's resettable agent until protocol-100
    `MsgInitAck` and is then delivered from the replay queue exactly once.
    Disconnect clears readiness and fences stale negotiation callbacks. An
    adversarial non-proposer's message reaches the honest pool while its
    proposals remain rejected and cannot finalize either node.

## 18. Implementation sequence

1. Add a small `plugin-catalog` module containing manifest/inventory value
   types, strict parser, SemVer/range validation, build index generator and
   canonical fingerprint codec.
2. Add provider-kind adapters and catalog builder with policy/dependency/order
   validation and legacy descriptors.
3. Add lazy typed provider registry and integrate `NodePlugin` manager input.
4. Replace all seven app-chain ServiceLoader scans with registry lookup.
5. Harden/own the JVM plugin class loader and native resource inclusion.
6. Add first-party/scaffold manifests and migration documentation.
7. Add unit, fixture-JAR, runtime construction, JVM packaging and native
   conformance tests.
8. Run adversarial architecture/security/lifecycle review and resolve every
   blocker/high finding before marking Accepted.

## 19. Consequences

### Benefits

- one product identity governs every typed contribution;
- invalid/incompatible deployments fail before partial activation;
- policy and duplicate behavior are consistent across all SPIs;
- operators can compare deterministic code/catalog fingerprints;
- legacy providers keep working while migration is explicit and observable;
- ADR-011.3/011.4 gain a stable catalog and provider source.

### Costs and limitations

- manifests duplicate selector/class metadata already present in ServiceLoader
  and provider methods, intentionally enabling drift detection;
- structural defects in an installed selected bundle fail startup even when a
  contribution is not configured;
- legacy discovery still executes provider construction to learn metadata;
- a shared parent-first loader cannot isolate conflicting plugin dependencies;
- digests identify bytes but do not attest publisher identity or safety;
- `ARTIFACT_CLOSURE` is intentionally conservative and may change when a
  mapped dependency is not exercised by a particular contribution; it proves
  byte-set equality, not reachability or semantic determinism;
- the per-bundle closure follows the app-resolved Gradle dependency graph plus
  explicitly supplied host/compile-only roots. The shared loader cannot prove
  or fingerprint undeclared reflective access to an unrelated application
  dependency; plugins must declare every executable dependency;
- directory/compatibility legacy evidence covers only the defining class or
  containing artifact, not external dependency bytes;
- directory deployment consumes temporary disk space for its bounded private
  snapshots. Normal close removes them. After abnormal exit, a later
  directory-loader startup makes a best-effort reclamation only for snapshots
  older than the 10-minute grace period that have the exact marker, yield an
  exclusive lease and retain the same identity; cleanup failure is non-fatal;
- strict app index generation rehashes its resolved inputs on each execution
  so closure-ownership graph changes cannot be hidden by file-only up-to-date
  checks;
- native-image plugin selection remains a build-time operation.

These costs are preferable to independent, discovery-order-dependent provider
activation that cannot be audited as one plugin platform.
