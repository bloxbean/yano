## Status

## 0. In plain words

Yano already has a public `NodePlugin` SPI. The SPI itself is adequate for a
first lifecycle-correctness pass, but its manager currently initializes each
plugin while it is still discovering plugins, sorts dependencies only
afterwards, tolerates missing dependencies and cycles, and gives every plugin
an isolated service registry. A failed plugin can therefore leave a partially
initialized plugin set, and the documented inter-plugin service lookup does
not work.

This ADR fixes that narrow kernel without inventing the full ADR-011 bundle
system. Yano will first discover and snapshot the existing `NodePlugin`
metadata, apply operator policy, validate the complete selected graph, and
produce a deterministic dependency order. Only then may any lifecycle callback
receive a context. Initialization and startup are transactional at the plugin
set level: a failure rolls back already-entered plugins in reverse order. All
contexts share one owner-aware, type-checked service registry, and shutdown is
reverse ordered and idempotent.

The public `NodePlugin`, `PluginContext` and `PluginsOptions` APIs remain
unchanged. Manifests, compatibility ranges, artifact digests, class-loader
isolation and correlation of the app layer's typed SPIs are deliberately
deferred to ADR-011.2. This is the correctness foundation those features can
build on.

## 1. Context and verified gaps

The current `PluginManager` has five lifecycle defects:

1. `ServiceLoader` iteration calls `init()` immediately. A later duplicate,
   missing dependency or cycle is discovered only after earlier plugins have
   already received contexts and may have registered listeners or work.
2. Dependency sorting happens after initialization, so `dependsOn()` does not
   control initialization order. Missing dependencies and cycles are logged
   rather than rejected.
3. Each `PluginContextImpl` creates its own service map. A service registered
   by plugin A cannot be found by plugin B even though `PluginContext` documents
   inter-plugin communication.
4. `PluginsOptions` already contains global enablement plus allow and deny sets,
   but the manager does not enforce the allow/deny policy.
5. Initialization/start failures are logged and skipped. There is no rollback
   of earlier callbacks or registrations, and normal `close()` is not in
   reverse dependency order.

These are foundation issues, not bundle-manifest issues. They can and should
be fixed while retaining today's ServiceLoader files and public interfaces.

## 2. Scope

### 2.1 Goals

1. Preserve source and binary compatibility for existing `NodePlugin`
   implementations and `META-INF/services` registrations.
2. Separate discovery/metadata collection from lifecycle callbacks.
3. Apply global enablement and allow/deny policy before activation.
4. Reject duplicate ids, absent selected dependencies and dependency cycles
   before `init()`.
5. Produce one deterministic dependency order independent of JAR or
   ServiceLoader iteration order.
6. Make the existing service API genuinely shared, type-checked and
   owner-aware, with atomic duplicate-key rejection.
7. Roll back partial initialization/startup and use reverse order for normal
   stop/close.
8. Define callback idempotency and failure behavior precisely enough to test.

### 2.2 Explicit non-goals

This ADR does **not** add or decide:

- `META-INF/yano-plugin.json` or another bundle manifest;
- plugin API compatibility ranges or a version-range grammar;
- artifact digests, signatures or provenance attestation;
- per-plugin, child-first or module-layer class loaders;
- JVM/native-image packaging rules beyond today's behavior;
- correlation between a `NodePlugin` and `AppStateMachineProvider`,
  `AppEffectExecutorFactory`, `FinalizedStreamSinkFactory`,
  `SequencerModeProvider`, `L1ObserverProvider` or `SignerProviderFactory`;
- a full typed-SPI catalog, contribution conflicts or bundle inventory;
- plugin health, metrics, CLI, domain REST routes or query extensions;
- sandboxing, hot reload, runtime install/uninstall, or optional/degraded
  failure tiers.

ADR-011.2 owns manifests, compatibility, typed-SPI correlation, bundle
inventory and packaging. ADR-011.3 owns query/domain APIs, and ADR-011.4 owns
health, metrics and CLI operations. Sandboxing, hot reload and runtime
installation remain parent-roadmap non-goals. In this document, the internal
plugin snapshot is only selected `NodePlugin` instances and their existing
metadata; it is not the ADR-011.2 manifested bundle catalog.

## 3. Compatibility boundary

The following public contracts stay unchanged:

- `NodePlugin.id()`, `version()`, `dependsOn()`, `capabilities()`, `init()`,
  `start()`, `stop()` and `close()`;
- `PluginContext` methods, including string-keyed `registerService()` and
  typed `getService()`;
- the fields of `PluginsOptions`;
- ServiceLoader discovery through the existing
  `META-INF/services/com.bloxbean.cardano.yano.api.plugin.NodePlugin` entry.

Existing plugins need no manifest and no recompilation. The intentional
behavioral tightening is that an invalid or failing **selected** plugin set no
longer starts partially. Duplicate ids, missing dependencies, cycles and
lifecycle failures become startup failures instead of warnings.

The runtime may add internal descriptor, policy, registry and lifecycle-state
types. It may add a named `PluginManager` factory that accepts the existing
`PluginsOptions`; the existing map-based constructor remains unchanged. A
constructor overload is avoided because legacy source may pass a null map,
which would become ambiguous against a `PluginsOptions` overload.

## 4. Lifecycle state model

One `PluginManager` instance follows one process lifecycle:

```text
NEW
  -> DISCOVERED
  -> VALIDATED
  -> INITIALIZED
  -> STARTED
  -> STOPPED -> STARTED   (same validated instances; node lifecycle restart)
       \------> CLOSED

Any discovery, validation, init, start, or stop failure -> FAILED -> CLOSED
```

`FAILED` and `CLOSED` are terminal. A clean node stop may start the same
already-validated plugin instances again; installing, removing, enabling,
disabling or upgrading plugins still requires a new manager/process, which
matches ADR-011's restart-based deployment lifecycle.

Lifecycle methods are single-threaded manager operations. Reentrant lifecycle
transitions from a plugin callback are rejected while the manager is in a
transitional state. Services exposed by `PluginContext`, however, retain the
public thread-safety guarantee.

## 5. Discovery and immutable metadata snapshot

### 5.1 No lifecycle callback during discovery

When plugins are globally enabled, the manager first obtains all
`ServiceLoader` providers and creates an internal candidate for each one. It
does **not** call `init()` or `start()`, construct a `PluginContext`, publish
filters/customizers, or otherwise activate a candidate during this pass.

Java ServiceLoader necessarily constructs provider instances, and the current
SPI requires calls to `id()`, `version()` and `dependsOn()` to obtain metadata.
Therefore "discovery without callbacks" in this ADR means without lifecycle
callbacks; constructors and metadata access still execute trusted plugin code.
Metadata methods must be side-effect free. A future declarative manifest can
remove that limitation under ADR-011.2, but is outside this slice.

Each metadata method used by the manager is called once and snapshotted into
an immutable internal descriptor. Later phases use that snapshot rather than
calling a plugin whose metadata could change between phases.

### 5.2 Metadata validation

Before policy or graph construction, every candidate must have:

- a non-null, non-blank id;
- a non-null, non-blank version (opaque in this ADR);
- a non-null dependency set containing only non-null, non-blank ids;
- no self-dependency.

Plugin ids and dependency ids are exact, case-sensitive identifiers. Leading
or trailing whitespace is invalid rather than silently normalized. A thrown
constructor or metadata exception is a discovery failure because the manager
cannot safely identify or select that provider.

Duplicate ids in the discovered candidate set are also fatal at this stage,
even when policy would later filter that id. Allow/deny policy selects an id;
it cannot safely or predictably choose between two implementations claiming
the same identity.

## 6. Operator policy

Policy uses the existing `PluginsOptions` values:

```text
selected(id) = enabled
            && (allowList is empty || allowList contains id)
            && !denyList contains id
```

Rules:

1. `enabled=false` performs no ServiceLoader scan and invokes no plugin code.
2. An empty allow list means all discovered ids are eligible.
3. A non-empty allow list is explicit selection; every listed id that is not
   also denied must be discovered, otherwise startup fails as an
   operator/configuration error.
4. Deny wins over allow. An id in both sets is inactive and logged as denied.
5. A deny entry for an undiscovered id is harmless.
6. Dependencies are not auto-enabled. If selected plugin A depends on plugin B
   and B is absent, denied or excluded by the allow list, A has a missing
   selected dependency and validation fails.
7. A filtered plugin receives no context and no lifecycle callback.

Node/application adapters must pass the complete existing `PluginsOptions` to
the manager instead of passing only its configuration map. Exact external
configuration spelling is an adapter concern; no new public Java policy type
is required by this ADR.

## 7. Graph validation and deterministic order

Validation finishes before `init()`:

1. Confirm that discovery produced one candidate per id; duplicate discovered
   ids have already failed before policy.
2. Resolve every selected dependency against the selected id map. Any missing
   dependency is fatal and the error names both plugin and dependency.
3. Reject dependency cycles and report at least one stable cycle path.
4. Produce a topological order in which every dependency precedes its
   dependent.

Topological order must be reproducible when multiple valid choices exist:

- process dependency ids in lexicographic id order;
- choose the next zero-indegree plugin by lexicographic plugin id;
- never use ServiceLoader, JAR-directory, hash-map or set iteration order as a
  tie-breaker.

The manager publishes the immutable ordered catalog only after the full graph
validates. Header-validation customizers, storage filters and services remain
invisible to runtime consumers until initialization succeeds.

## 8. Shared service registry

All selected plugin contexts use one manager-owned service registry behind the
existing `PluginContext` API. Each context carries its owning plugin id
internally.

`registerService(key, service)`:

- rejects null or blank keys and null services;
- atomically rejects an already-owned key, including a second registration by
  the same plugin;
- records `(key, ownerPluginId, service)` so rollback can remove only the
  failing/closing owner's entries;
- treats duplicate registration during `init()` as an initialization failure.

`getService(key, type)` preserves the public behavior:

- return empty when the key is absent or the value is not an instance of the
  requested type;
- otherwise return the typed value.

A plugin that requires another plugin's service during `init()` must declare
that plugin in `dependsOn()`. Dependency-first initialization then makes the
service available deterministically. Relying on the incidental lexical order
of unrelated plugins is unsupported.

The same owner tracking applies internally to manager-owned registrations such
as storage filters. On rollback or close, registrations are removed after the
owner's `close()` callback, allowing dependents to use dependency services
while they themselves close in reverse order. Event-bus subscriptions,
scheduler tasks and arbitrary external side effects cannot be automatically
undone through today's public API; plugins remain responsible for releasing
those from `stop()`/`close()`.

Registration lifetime follows the lifecycle callback that creates it:

- a service or filter registered synchronously during `init()` is
  initialization-scoped and remains across clean stop/start cycles until
  rollback or close;
- a service or filter registered synchronously during `start()` is scoped to
  that start cycle, remains available while the plugin is running, and is
  removed immediately after the owner's `stop()` callback (even when stop
  fails), so the plugin can register the same key/filter once on restart;
- service registration remains available while the start cycle is active, but
  storage filters must be registered before `start()` returns because runtime
  installs one immutable filter snapshot immediately after all plugins start;
- contribution writes after initialization but before start, between cycles,
  during stop, or after rollback/close are rejected. Typed service reads remain
  available for dependency-safe shutdown.

The manager deactivates each context atomically before removing its owner's
registrations. A retained context therefore cannot repopulate a registry after
rollback or close. Reads remain type-safe; mutating registrations through an
inactive context fail.

This is a shared registry with typed retrieval, not the ADR-011.2 manifested
bundle catalog and not a generic replacement for Yano's typed extension SPIs.

## 9. Initialization and startup transactions

### 9.1 Initialize

After validation, the manager creates one owner-scoped context per plugin and
calls `init(context)` in topological order.

If plugin P's `init()` throws:

1. no plugin receives `start()`;
2. call `close()` best-effort on P because it may be partially initialized;
3. call `close()` on previously initialized plugins in reverse initialization
   order;
4. remove each owner's services/filters after its close attempt;
5. keep running cleanup after a cleanup callback throws;
6. clear all manager-published contribution snapshots;
7. enter terminal `FAILED` and throw a startup exception naming phase and
   plugin, attaching cleanup failures as suppressed exceptions.

### 9.2 Start

Only after **all** selected plugins initialize successfully does `startAll()`
call `start()` in topological order.

If P's `start()` throws:

1. call `stop()` best-effort on P because it may be partially started;
2. stop plugins whose `start()` completed, in reverse start order;
3. close every initialized plugin in reverse initialization order, including
   initialized plugins whose start had not yet been attempted;
4. remove owner registrations after each close attempt;
5. clear published contributions, enter `FAILED`, and propagate the primary
   failure with cleanup failures suppressed.

There is no best-effort continuation with the remaining plugins. Until an
ADR-011.2 descriptor defines optional/degraded tiers, every operator-selected
`NodePlugin` is required. This is also necessary because a current
`NodePlugin` may supply header-validation behavior; silently omitting it can be
more dangerous than refusing startup.

Runtime integration must not catch these failures merely to log and continue.
When plugins are enabled and selected, discovery, validation, init or start
failure fails node startup.

### 9.3 Runtime activation boundary

Discovery and initialization happen during runtime assembly because a selected
plugin may contribute header-validation customization needed to construct sync.
Every successfully acquired assembly resource is placed on a reverse cleanup
stack, so plugin policy/init rejection or a later constructor failure releases
plugins, listeners, stores, the event bus, RocksDB and schedulers before the
constructor rethrows.

Active plugin startup is a dedicated kernel stage immediately after the runtime
startup boundary and before transaction admission, bootstrap recovery, UTXO
background work, producers or sync. The stage starts the selected plugins and
then replaces the UTXO store's filter chain with the immutable storage-filter
snapshot, explicitly clearing any previous chain when the new snapshot is
empty. If filter installation fails, plugin start is rolled back.
`NodeStartedEvent` publication remains a later stage after sync startup.

On normal shutdown, reverse kernel order first drains sync/apply and all other
producers, then stops plugins while runtime maintenance is still held, and only
then releases the shutdown boundary. A ledger-apply worker that cannot reach a
safe stop makes the runtime terminal; plugins are not stopped underneath live
apply work and that runtime instance cannot restart.

## 10. Normal stop and close

Normal shutdown is dependency-safe:

1. `stopAll()` invokes `stop()` in reverse successful-start order.
2. `close()` first performs any required stop, then invokes `close()` on all
   initialized plugins in reverse initialization order.
3. After each owner's stop callback, remove that owner's start-cycle services
   and filters; initialization-scoped contributions remain available to
   dependents until owner close.
4. A callback exception is recorded/logged, but does not prevent cleanup of
   the remaining plugins.
5. Owner registrations are removed after each close attempt; the registry and
   published contribution snapshots are empty when manager close completes.

A stop callback failure enters terminal `FAILED`, is reported as phase `STOP`,
and prevents restart because the plugin may still be running. `close()` still
attempts every close callback and clears every registration. Runtime and fatal
errors are propagated only after the full cleanup pass, with additional
failures attached.

The manager guarantees at-most-once invocation of each lifecycle phase per
plugin per lifecycle cycle. Repeated `discoverAndInit()` or `startAll()` in
their already-completed state is a no-op. Repeated `stopAll()` and `close()`
are no-ops. `startAll()` after a clean `STOPPED` state starts the same ordered,
initialized instances again so the existing `RuntimeNode` stop/start contract
continues to work; it does not rediscover code or metadata. A call that would
move a `FAILED` or `CLOSED` manager back to an earlier phase is rejected or
remains a no-op. Individual plugins must continue to make their public
`close()` implementation idempotent as `NodePlugin` already requires.

The existing `getStorageFilters()` manager helper now returns an immutable
snapshot rather than exposing the manager's live mutable list. This helper is
runtime-internal behavior, not part of the public `NodePlugin` or
`PluginContext` compatibility boundary.

This ADR specifies order and continuation after exceptions. It does not add a
timeout, forced thread termination or sandbox for a callback that never
returns.

## 11. Failure and diagnostic contract

Failures are classified by phase:

- `DISCOVERY`: provider construction or metadata access/shape failed;
- `POLICY`: an explicit, non-denied allow-list id was not discovered;
- `VALIDATION`: duplicate id, missing selected dependency or cycle;
- `INITIALIZATION`: `init()` or registration failed;
- `START`: `start()` failed;
- `STOP` / `CLOSE`: cleanup callback failed after cleanup continued.

Startup exceptions include the stable plugin id when available and a concise
reason without dumping configuration values. Dependency and cycle messages are
deterministically ordered so tests and operator comparisons do not depend on
discovery order.

Bundle inventory is deferred to ADR-011.2 and health/operations endpoints to
ADR-011.4, but logs must at minimum report:

- selected and filtered plugin ids with the policy reason;
- the final deterministic activation order;
- primary lifecycle failures and every cleanup failure.

## 12. Decisions

| ID | Decision |
|---|---|
| D1 | Retain the existing public `NodePlugin`, `PluginContext`, `PluginsOptions` and ServiceLoader contracts. |
| D2 | Complete discovery, policy, graph validation and deterministic ordering before any lifecycle callback. |
| D3 | Enforce deny-over-allow policy; dependencies never bypass policy. |
| D4 | Reject duplicate discovered ids, missing selected dependencies and cycles before initialization. |
| D5 | Use lexicographic id tie-breaking for a deterministic topological order. |
| D6 | Back every context with one owner-aware shared registry; atomically reject duplicate service keys. |
| D7 | Treat initialization/startup as plugin-set transactions with reverse rollback. |
| D8 | Stop and close in reverse order, continue cleanup after exceptions, and invoke every phase at most once. |
| D9 | Treat every selected `NodePlugin` as required until an ADR-011.2 descriptor defines explicit optional/degraded semantics. |
| D10 | Defer manifests, compatibility, digests, class loaders, typed-SPI correlation and inventory to ADR-011.2. |
| D11 | Start plugins and freeze storage filters before bootstrap/sync; stop them after producers drain and before the maintenance boundary is released. |
| D12 | Make stop failure and unsafe apply shutdown terminal; never restart potentially live plugin work. |
| D13 | Revoke retained contexts on rollback/close and release all acquired runtime resources on assembly failure. |
| D14 | Scope start-time services and filters to one start cycle; retire them after owner stop so clean restart is duplicate-free. |

## 13. Test plan

### 13.1 Discovery, policy and graph tests

1. A duplicate discovered id fails before policy can disambiguate it and
   before any candidate receives `init()`.
2. A missing dependency fails before `init()` and names both ids.
3. A cycle fails before `init()` and reports a stable cycle path.
4. Randomized ServiceLoader/candidate iteration produces the same topological
   order across runs, including graphs with several zero-indegree choices.
5. `enabled=false` performs no provider construction or metadata/lifecycle
   callback where the adapter can avoid discovery.
6. Empty allow selects all; a non-empty allow selects only listed ids; deny
   wins; filtered plugins receive no lifecycle callback.
7. A missing explicit, non-denied allow-list id fails; a missing deny-list id
   (including an id also present in the allow list) does not; and a selected
   plugin cannot pull a denied dependency back in.
8. Metadata methods are called once and later mutations cannot change the
   internal descriptor/order.

### 13.2 Registry tests

1. A service registered by dependency A is visible with the correct type to
   dependent B during B's `init()`.
2. Missing keys and type mismatches return empty.
3. Duplicate keys fail atomically and name both existing owner and attempted
   owner without exposing the service value.
4. Concurrent duplicate registration has exactly one winner.
5. Init/start rollback and normal close remove only the closing owner's
   services and filters at the correct reverse-order point.
6. A retained context cannot register a service or filter after rollback or
   close.
7. Init-scoped contributions survive a clean stop/start; start-scoped services
   and filters are removed after stop and can be registered exactly once on
   the next start without duplicate keys or filter accumulation.
8. A restart whose new filter snapshot is empty clears the UTXO store's prior
   start-cycle filter chain rather than retaining stale filtering behavior.

### 13.3 Lifecycle and rollback tests

1. Successful callbacks occur as `init A,B,C`, `start A,B,C`,
   `stop C,B,A`, `close C,B,A` for a dependency chain A <- B <- C.
2. Init failure at C calls no `start()`, closes C then prior initialized
   plugins in reverse, and continues after a close exception.
3. Start failure at C stops the partially started C and prior started plugins
   in reverse, then closes every initialized plugin in reverse.
4. A plugin initialized after the eventual start failure point but never
   started is still closed during rollback.
5. Repeated init/start/stop/close calls do not duplicate callbacks; `FAILED`
   and `CLOSED` cannot be restarted.
6. One stop/close exception does not skip remaining cleanup, and cleanup
   failures are attached/logged without replacing the primary startup failure.
7. A stop failure is terminal and prevents restart; fatal cleanup errors are
   rethrown only after all remaining callbacks run.
8. Reentrant lifecycle calls from init/start/stop/close are rejected without
   corrupting manager state or skipping rollback.

### 13.4 Integration and compatibility tests

1. An existing ServiceLoader-only `NodePlugin` runs unchanged with no
   manifest.
2. `RuntimeNode` propagates manager discovery/init/start failures rather than
   logging and continuing with partial validation/filter state.
3. Storage filters and header-validation customizers are published only after
   successful initialization and are empty after rollback/close.
4. Existing default/empty `PluginsOptions` behavior remains: plugins enabled,
   empty allow/deny, existing config map available through `PluginContext`.
5. Plugin start and filter installation precede bootstrap and sync; reverse
   stop drains sync before plugins and releases maintenance afterwards.
6. A constructor-time plugin policy/init failure releases the RocksDB path so a
   fresh runtime can open the same database immediately.

## 14. Implemented sequence

1. Add internal immutable candidate/descriptor and lifecycle-state models.
2. Wire complete `PluginsOptions` through a named `PluginManager` factory while
   retaining current constructor entry points.
3. Split the current combined discovery/init loop into discovery, policy,
   validation/order and initialization phases.
4. Replace per-context maps with one owner-aware registry and owner-aware
   storage-filter registration.
5. Implement init/start rollback, reverse stop/close and at-most-once guards.
6. Make runtime startup propagate plugin-manager failure and publish plugin
   contributions only after successful initialization.
7. Add the unit and integration matrix in §13 before beginning ADR-011.2
   manifest and typed-SPI catalog work.

## 15. Consequences

### Benefits

- dependencies finally govern the lifecycle they claim to govern;
- invalid selections fail before plugins can partially mutate runtime state;
- inter-plugin services work through the existing API;
- startup and shutdown are deterministic and testable;
- current plugins remain usable without adopting a speculative manifest;
- ADR-011.2 and later plugin-platform features gain a sound lifecycle
  substrate.

### Costs and limitations

- deployments that currently tolerate a broken plugin by logging and
  continuing will now fail startup until the plugin is fixed or denied;
- plugin constructors and metadata methods still execute during discovery;
- the manager cannot automatically undo arbitrary event subscriptions,
  scheduled work or external side effects made by a plugin;
- a non-returning lifecycle callback can still block startup/shutdown;
- typed app-layer SPIs continue to be discovered independently until
  ADR-011.2 is designed and implemented.

These limitations are explicit and preferable to implying that the narrow
kernel already provides the full ADR-011 plugin platform.
