## 1. Decision summary

Yano will provide an `appchain-composite` library and one manifested
`AppStateMachineProvider` named `composite`. A composite profile assembles
several deterministic components behind the existing one-state-machine-per-
chain boundary and commits their ordered identity, configuration, routes,
namespaces, activation intervals, workflows, and quotas as authenticated chain
state.

V1 makes these decisions:

1. Components are assembled inside one composite bundle/provider. They are not
   independently selected across plugin bundles.
2. A profile is an immutable, canonical, versioned value. Its canonical bytes
   are written at the first app block and verified on every later transition
   and retained-state startup; its SHA-256 digest is deterministically derived
   from those committed bytes.
3. Every component receives an enforced state-key namespace and owns exact
   message topics and a query namespace. Route conflicts fail construction.
4. Components execute in profile order in one atomic block transaction and
   produce one MPF state root.
5. All effects use the one framework emitter. A consensus-state ownership link
   dispatches each CHAIN result to the exact component generation that emitted
   it. Per-component quotas compose with the framework global cap.
6. Cross-component state changes use a named, versioned workflow/coordinator
   declared in the profile. Components cannot read sibling namespaces directly.
7. Stock profile `evidence-v1` includes registry, approvals, document trail,
   and evidence publication components. Balances are opt-in, not implicit.
8. Planned activation intervals support dormant code before activation,
   component add/disable, and version replacement. Changing an already
   committed schedule is not a v1 runtime operation; it requires a new-chain
   profile or a later governed-profile ADR.

This is a composition programming model, not a second consensus engine. App
blocks, finality, MPF, effects, queries, proofs, snapshots, and Cardano anchors
remain framework-owned.

## 2. Why a dedicated abstraction is required

Yano currently selects one `AppStateMachine` per chain. Calling complete
machines in a loop is unsafe because those machines may:

- interpret every message body regardless of topic;
- use overlapping state keys;
- disagree about validation ownership;
- expose conflicting query paths;
- consume effect ordinals without a shared quota; and
- receive outcomes for effects they did not emit.

The composite must therefore own routing and isolation. A component is not an
independently running state machine and cannot bypass the composite views.

## 3. Scope and non-goals

### 3.1 In scope

- deterministic component identity, ordering, configuration, and activation;
- enforced state/message/query/effect isolation;
- root-fixed component and aggregate queries;
- exact effect-result ownership;
- declared atomic cross-component workflows;
- a first-party stock preset and no-code demo profile; and
- conformance, replay, restart, snapshot, proof, and anchor coverage.

### 3.2 Not in v1

- hot discovery of component providers from unrelated selected bundles;
- changing the committed profile schedule through an admin API;
- arbitrary sibling-state access;
- distributed transactions across chains;
- external I/O during a workflow; or
- DPP-specific schemas, actor signatures, lifecycle rules, or Cardano assets.

Those DPP semantics belong in a DPP bundle/ADR built on this foundation.

## 4. Packaging and plugin boundary

`appchain-composite-contracts` contains bounded stock-workflow wire contracts,
the public component-key/profile-commitment mapping, and the single canonical
`composite/aggregate-v1` client/runtime codec. It contributes no plugin manifest or service
provider, so clients and the standalone demo runner do not acquire node/plugin
implementations merely to submit a workflow command.

`appchain-composite` is a normal library plus an ADR-011 manifested
`app-state-machine` contribution:

```text
bundle id       com.bloxbean.cardano.yano.appchain.composite
contribution    app-state-machine/composite
provider        CompositeStateMachineProvider
```

The stock application distribution includes `appchain-composite`,
`appchain-stdlib`, and `appchain-evidence-registry` as deterministic application
capabilities. They are not copied into the demo's external plugin directory.
Kafka, S3, and IPFS remain separately staged connector bundles. This avoids a
duplicate bundle identity while keeping connector clients opt-in.

V1 deliberately does not add `app-state-component` to the plugin catalog.
Cross-bundle components would require the host to resolve component providers,
transfer products across bundle class-loader ownership, retain old provider
generations for replay and in-flight results, and coordinate teardown with the
selected composite. Treating that as ordinary `ServiceLoader` discovery would
bypass ADR-011 selection and lifecycle rules.

Custom products remain extensible: a custom manifested state-machine bundle
depends on the composite library, assembles its component/profile objects, and
exports one `AppStateMachineProvider`. A later ADR may add a host-owned
component contribution kind after its selection, dependency, lifecycle,
upgrade, and native-image behavior are fully designed.

## 5. Component contract

The library contract is conceptually:

```java
interface CompositeComponent {
    ComponentDescriptor descriptor();
    void init(AppStateReader ownState, AppChainInfo chain);
    AdmissionResult validate(AppMessage routedMessage);
    void apply(AppBlock routedBlock, AppStateWriter ownState,
               AppEffectEmitter ownedEffects);
    void onEffectResult(AppBlock block, EffectResult result,
                        AppStateWriter ownState, AppEffectEmitter ownedEffects);
    byte[] query(String localPath, byte[] params, AppQueryContext ownState);
}
```

An adapter may wrap an existing complete `AppStateMachine`, but the adapter
must declare exact routes and the composite passes only routed messages plus a
namespaced state view. The component never receives the root writer/reader.

Each descriptor contains:

```text
componentId          stable lowercase identifier
semanticVersion      stable version identifier
configurationId      canonical non-secret effective-config identifier
stateAndResultCompatibilityId
                     stable state-schema/result-callback compatibility identity
fromHeight           inclusive activation height
untilHeight          exclusive height, or 0 for unbounded
topics               sorted exact normal-message topics
queryPaths           sorted local paths, or an explicit bounded dispatcher
maxEffectsPerBlock   positive component quota, or 0 when effects are forbidden
```

Identifiers use `[a-z][a-z0-9-]{0,62}`. Semantic/configuration identifiers use
bounded printable ASCII and must not contain credentials, endpoints, or other
secrets. A component generation is identified by the tuple
`(componentId, semanticVersion, fromHeight)`.

`configurationId` may change when a replacement generation intentionally uses
different deterministic settings. It is not a schema identifier. Every
generation that reuses one `componentId` must retain the exact same
`stateAndResultCompatibilityId`, because those generations share one physical
namespace and late CHAIN results return to their exact emitter generation. A
state-layout or result-callback contract change therefore uses a new
`componentId` plus an explicit deterministic migration/workflow, or a fresh
chain; silently reusing the old namespace is rejected.

## 6. Canonical profile and chain identity

### 6.1 Profile contents

The canonical profile contains, in this exact logical order:

```text
schemaVersion = 1
profileId
profileVersion
ordered component descriptors
sorted workflow descriptors
sorted legacy query aliases
global aggregate-query limits
```

Strings are UTF-8 with unsigned length prefixes, integers are fixed-width
big-endian, lists carry explicit counts, and sets/maps are encoded in validated
lexical order. Unknown or duplicate fields are impossible in the binary
model. The profile digest is:

```text
SHA-256("yano-composite-profile-v1" || canonicalProfileBytes)
```

Canonical profile bytes are capped at 64 KiB so every public proof client can
fetch and verify the marker within the host response bound. Golden vectors
freeze the bytes and digest independently of Java collection,
locale, provider discovery, or configuration iteration order.

### 6.2 Authenticated marker

At app height 1 the composite writes one reserved application-state entry:

```text
~composite/profile/v1 -> canonicalProfileBytes
```

Before every transition it requires exact byte equality. `init()` performs the
same check when retained state already contains the marker. Therefore:

- a retained node with different effective configuration fails startup;
- a fresh mismatched follower derives a different first post-state root and
  cannot vote for or accept the proposed block; and
- a joining node replaying existing history fails before it can participate
  with a divergent state root.

Installed code may be a dormant superset. Only the committed effective profile
controls consensus behavior; equality of JAR files is neither required nor
sufficient.

## 7. Activation and upgrades

A descriptor is active when:

```text
height >= fromHeight && (untilHeight == 0 || height < untilHeight)
```

For one `componentId`, activation intervals may touch but cannot overlap, and
every replacement must retain its committed
`stateAndResultCompatibilityId`. Changing a configuration id alone does not
authorize a state schema or callback-contract change.
Message-topic ownership may move only through non-overlapping intervals. A
wire schema must not be reinterpreted at an activation boundary. Because v1
admission has no target-height context, every scheduled generation that reuses
one topic validates an admitted envelope; topic reuse therefore also requires
identical admission semantics. Any wire, authorization, or other validation
change uses a new versioned topic/query path. Admission accepts only a uniquely
declared topic; an inactive route is allowed into the pool but is a
deterministic no-op if sequenced outside its interval.

Old component generations remain installed while any supported replay or
snapshot floor, or any still-open CHAIN effect ownership link, can reference
them. V1 profiles are fixed at genesis. A later governed-profile ADR must
define how members advertise code readiness and authorize schedule mutations;
an operator config edit is never an upgrade protocol.

## 8. State isolation

Component state uses a collision-free physical key:

```text
domain || componentIdLength || componentId || localKey
```

where `domain` is the fixed binary `yano-composite-state-v1` prefix and lengths
are unsigned canonical values. Local keys are non-empty and bounded. The view:

- prefixes every `get`, `put`, and `delete`;
- clones boundary byte arrays;
- exposes the one root state hash unchanged; and
- never offers iteration or access to a raw/root writer.

Composite metadata, effect ownership, and transient quota keys use a distinct
reserved composite domain. A component cannot construct a local key that
escapes its prefix. Namespace tests use empty, maximum, binary, prefix-like,
and malicious keys.

## 9. Message routing and block order

Normal topics map to exactly one active component generation at a height.
Construction rejects duplicate ownership, ambiguous intervals, system topics,
unversioned workflow collisions, and invalid identifiers.

For each app block the composite:

1. verifies/creates the committed profile marker;
2. groups normal messages by active component while preserving their original
   relative order;
3. executes active components in committed profile order;
4. executes active declared workflow routes after components, in canonical
   workflow-descriptor order; and
5. clears transient per-block quota counters before the atomic commit.

This order is independent of plugin discovery and map iteration. Components
with isolated namespaces cannot observe grouping. Cross-component order
dependencies must be expressed as a workflow, not inferred from arrival order.

Each routed `AppBlock` is a synthetic projection. Its `messages` and
`messagesRoot` contain only the routed subset, its inherited `stateRoot` is not
a component-local post-state commitment, and its finality certificate is
empty: the original certificate authenticates the full block, not the
projection. Result callbacks receive a metadata-only projection with zero
messages and an empty certificate. Descriptors, blocks, messages, and boundary
arrays are snapshotted so a component cannot mutate sibling input or change
its committed capabilities after construction.

## 10. Queries and proofs

Direct component queries use:

```text
components/<componentId>/<localPath>
```

The active generation at `AppQueryContext.committedHeight()` receives a
namespaced context. Unsupported or inactive paths fail with the normal query
error contract.

`composite/aggregate-v1` accepts a canonical, bounded list of component
subqueries (initially at most 16 and 64 KiB aggregate parameters). It executes
them sequentially against the same supplied `AppQueryContext` and returns an
ordered canonical response. It cannot perform external I/O or observe a newer
root between subqueries.

`AggregateQueryCodecV1` and `AggregateQueryLimitsV1` live in the lightweight
contracts artifact and are the only wire implementation used by clients and
the runtime. Encoded requests are capped by the 64 KiB host request bound;
complete encoded responses are capped by the committed response limit and the
1 MiB host maximum.

The stock profile may expose documented compatibility aliases such as
`evidence/get`. An alias changes only dispatch; `AppQueryResult.stateMachineId`
remains `composite`. Proof clients translate component-local keys with the
public `CompositeCommitmentV1` codec before requesting MPF proofs. For the
generic `composite` machine they also prove the authenticated profile marker
and require an operator-pinned profile digest. The query payload, profile
proof, component proofs, height, and root must describe one snapshot.

## 11. Effects, quotas, and result dispatch

All component emitters delegate to the block's one framework emitter, so
global effect ordinals and ADR-010 hashes remain unchanged. Before delegation,
the wrapper enforces the component quota. The profile is invalid when the sum
of every component-generation quota that has started, plus currently active
workflow quotas, exceeds the configured framework `effects.max-per-block`.
Retired generations remain conservatively reserved because an open CHAIN
result may still invoke their callback and emit a continuation. This reserves
capacity deterministically and prevents an old callback or early component
from starving a later active component. V1 does not reclaim that reservation
from a fixed profile after the result window closes.

For every `ResultPolicy.CHAIN` effect the wrapper writes:

```text
~composite/effect-owner/v1/<effect-id-hash>
    -> exact component-generation identity
```

When a result is incorporated, the composite requires this link, dispatches
exactly once to that retained generation with its namespaced writer, allows
continuation emission through the same global emitter, then deletes the link.
Unknown, malformed, ambiguous, or mismatched ownership fails the proposed
transition; broadcasting a result cannot select a callback by scope prefix.

Per-block quota counters are staged under transient composite keys while result
callbacks and normal apply share the block. They are deleted before commit, so
they coordinate all callback/apply emissions without growing authenticated
state or relying on mutable Java fields across proposal retries.

## 12. Cross-component workflows

A workflow is a separate profile entry with:

```text
workflowId + semanticVersion
exact versioned topic
activation interval
ordered participant component generations
bounded command codec
deterministic preconditions and apply coordinator
effect quota allocation
```

The coordinator receives only explicitly declared participant views. It
validates every input before writing, evaluates cross-component preconditions,
and stages every mutation/effect in the same block batch. Expected business
precondition failures, exact replays, and conflicting operation ids are
deterministic per-message no-ops so an admitted message cannot poison proposer
selection. Exceptions are reserved for invariant corruption or infrastructure
failure and discard the complete block batch; external retries remain effects,
not workflow I/O.

The stock profile includes an evidence-release workflow proving the reusable
shape:

```text
registry identity exists
AND approval item is APPROVED for the submitted evidence command hash
THEN append the document-trail commitment
AND submit evidence publication atomically
```

This workflow is an example/application building block. It does not claim DPP
actor governance, regulatory validity, or real-world truth.

## 13. Stock `evidence-v1` profile

The first no-code preset contains these ordered component namespaces:

| Order | Component | Topic | Initial purpose |
|---:|---|---|---|
| 0 | `registry` | `registry.command.v1` | owned product/identity records |
| 1 | `approvals` | `approvals.command.v1` | k-of-n decisions |
| 2 | `doc-trail` | `doc-trail.command.v1` | append-only entity heads |
| 3 | `evidence` | `evidence.command.v1` | S3/IPFS publication and Kafka acknowledgement |

All begin at height 1. Evidence retains the ADR-013.1 explicit/direct result-
continuation activation. The standalone `evidence-registry` profile remains a
regression and migration fixture. Balances require a separate explicit preset
because mint/transfer semantics are not part of evidence or DPP by default.

The stock preset exposes both the coordinated `evidence.release.v1` workflow
topic and the compatible direct `evidence.command.v1` component topic. The
workflow is opt-in orchestration for the demo, not a mandatory authorization
gate: a submitter accepted on the direct topic can publish without the registry
and approval prerequisites. A DPP or regulated bundle that requires those
preconditions must omit/reject the direct route or enforce equivalent domain
authorization in its own component; the generic stock preset is not that
policy boundary.

Selection is configuration-only:

```properties
yano.app-chain.chains[0].state-machine=composite
yano.app-chain.chains[0].machines.composite.preset=evidence-v1
```

The effective component settings, including defaults and activation heights,
are represented in the committed profile configuration identities.

The stock provider uses `CompositeStateMachine.create(context, ...)`, which
always validates its reserved quotas against the actual framework
`effects.max-per-block`. Custom bundles use the overload with their own
distinct state-machine id and export that same id from their manifested
provider; the stock `composite` id is not a shared identity for arbitrary
profiles.

The no-code runner uses the lightweight contracts artifact to submit the
registry and approval prerequisites, waits for threshold finality after each,
then submits `evidence.release.v1`. The workflow performs the document-trail
append and evidence submission in one app-block transaction; the existing
storage, Kafka, state/effect proof, threshold-finality, and Cardano-anchor
audits then run unchanged.

## 14. Failure behavior

The following fail before a chain can make progress:

- duplicate profile/component/generation/workflow identifiers;
- overlapping activation intervals or topic/query routes;
- invalid/non-canonical effective configuration;
- missing referenced generations;
- quota sum above the framework cap;
- a retained profile marker mismatch;
- an unknown effect-result ownership link; or
- a workflow that requests undeclared namespace access.

Malformed ordinary messages remain admission rejections or deterministic
no-ops according to their component contract. Infrastructure exceptions are
never converted into successful partial transitions.

## 16. Acceptance criteria

This ADR moves to Accepted only when:

1. profile bytes/digest have golden vectors and every member commits the same
   marker;
2. mismatched profiles fail on retained startup and first-block/join replay;
3. route, namespace, activation, and discovery-order tests are deterministic;
4. exact component generations own and receive their CHAIN results;
5. global/per-component quotas cannot starve later components;
6. component and aggregate queries are root-fixed and their physical proof
   keys verify against the same root;
7. the stock workflow is atomic across success, precondition failure, replay,
   result failure/expiry, restart, and snapshot;
8. the stock preset is selectable without Java assembly and the standalone
   profile remains compatible;
9. Compose and normal three-node demos pass with effects, proofs, retained
   data, and Cardano script anchoring; and
10. JVM/native packaging and independent review leave no unresolved high-
    severity correctness or maintainability finding.

### 16.1 Verification evidence (2026-07-17)

- Golden profile/codec vectors, mismatch handling, activation schedules,
  discovery-order independence, namespace/route rejection, quotas, exact
  result ownership, aggregate queries, semantic state probes, replay, restart,
  and snapshot conformance passed the focused contract and implementation
  suites.
- The proof-aware evidence client verified logical-to-physical composite keys,
  the committed profile marker and digest, inclusion and exclusion proofs, and
  one-root snapshot binding.
- The stock `evidence-v1` preset was constructed through the packaged JVM
  provider and through a fresh Linux ARM64 native executable; native status
  reported the real `composite` machine with the fixed sequencer.
- Fresh and retained-restart three-member scenarios passed through both
  Compose and normal host deployment with the direct result-continuation mode,
  real first-party connectors, state/effect proofs, and script anchoring.
- Release contracts, application packaging, dependency alignment, artifact
  boundaries, and the tracked runtime regression suite passed. The one
  timing-sensitive rotation test that timed out in a repeated full run passed
  in the preceding run and immediate isolated rerun.
- Independent final reviews of the plugin/client/demo surfaces and of
  consensus/determinism both returned PASS with zero unresolved Critical,
  High, or Medium findings.

## 17. Consequences

### Positive

- Applications can combine common deterministic capabilities under one state
  root without changing app-chain consensus.
- Namespace and result ownership are enforced rather than conventional.
- Profiles are reproducible and auditable across members and restarts.
- Stock configuration supports a richer no-code evidence/DPP-shaped demo.
- Custom domain bundles reuse a focused library without a new plugin framework.

### Trade-offs

- V1 custom components are assembled into one bundle instead of independently
  hot-plugged.
- Planned schedules are immutable after genesis.
- Existing standalone proof clients need composite namespace awareness.
- A generic composite provides mechanics, not domain truth or DPP compliance.
