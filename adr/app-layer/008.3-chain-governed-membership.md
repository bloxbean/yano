## 0. In plain words (the ballot-box picture)

Today, changing the member list means every operator runs the same admin
command on their own node, by hand, in the right order — everyone separately
edits their own copy of the list and *hopes* they did it identically. One
forgotten command and nodes disagree about who's a member.

**Governed mode makes the notebook itself the ballot box.** To add Dave
(members Alice/Bob/Carol, threshold 2): Alice writes *"add Dave"* into the
chain — recorded, nothing happens (1 of 2). Bob writes the identical
*"add Dave"* — threshold reached, and the change activates automatically a
few pages later (a delay so every node sees it coming). Carol types nothing:
her node updates itself because the change is in the notebook she's already
reading. No runbooks, no ordering mistakes possible, and a member joining a
year later replays the notebook and derives the exact same member history —
including why every old signature verifies. There is no explicit "no" vote:
if not enough members echo a request within the window, it quietly expires —
silence is refusal. We chose this over building a second, dedicated voting
protocol (proposal ids, ballots, tallies): one consensus mechanism in the
codebase is one attack surface; the ledger already provides the agreement we
need.

## 1. Goal

Today membership changes are **node-local admin calls** repeated on every node
in runbook order — operators can apply them inconsistently, nodes can disagree
on the signer set at a height, and the admin endpoint is consensus-critical
with no multi-party authorization. After this ADR, membership changes are
**finalized chain transactions**: `epochAt(height)` becomes a pure function of
the chain's own history, identical on every node, replayable by catch-up and
late joiners, with threshold approval built in.

## 2. Design

### 2.1 Governance messages on a reserved topic

Membership commands are ordinary app messages on the reserved topic
`~governance/membership` (reserved `~` prefix exists since ADR-005 D7; REST
already refuses user submissions to `~` topics — a dedicated gateway/admin
path submits these):

```cddl
membership-command = [
    version : uint,               ; 1
    op      : uint,               ; 0 = add-member, 1 = remove-member, 2 = set-threshold
    subject : bstr / uint,        ; member key (op 0/1) or threshold (op 2)
    activation-lag : uint,        ; blocks between finalization and effect (>= 1)
]
```

- Envelope auth is unchanged (member-signed); only **current members** may
  submit governance commands (non-members are already rejected at admission).
- The command body is canonical CBOR; approval matching is byte-exact.

### 2.2 Approval = threshold-many identical commands

A command **activates** when `thresholdAt(h)` *distinct* members' identical
commands (same canonical bytes) have been finalized within a rolling window of
`approvalWindowBlocks` (default 600). Rationale: reuse the chain's existing
trust parameter — the set that can finalize blocks is the set that can change
membership — and avoid inventing a second voting protocol; the ledger itself
is the ballot box.

- Approvals are counted against membership **at the height of the activating
  block**; duplicates by the same member count once.
- On activation in block `h`, the framework appends a membership epoch
  `fromHeight = h + activation-lag` (existing `MemberGroup.appendEpoch`).
  The lag (default 10) gives every member time to observe the change before
  it affects cert verification and the 008.2 proposer schedule.

### 2.3 A framework-level handler, atomic with the block

The handler is **not** a user state machine — it runs inside the engine's
block-apply path for every block, on every node:

1. During `applyBlock`, messages on `~governance/membership` are interpreted
   by the framework handler (user state machines never see `~` topics —
   already the routing rule).
2. Pending-approval state (command bytes → approver set → first-seen height)
   and the resulting epoch list are written to ledger meta **inside the same
   atomic commit batch** as the block. `member_epochs` persistence moves from
   its current out-of-band `metaPutString` into `commitBlock`'s batch.
3. Because the handler is deterministic over finalized history, catch-up and
   late joiners re-derive identical epochs with no extra protocol: replaying
   blocks replays governance. (`verifyCert`/`membersAt`/`thresholdAt` are
   already height-resolved — they need no change.)

Guard rails (evaluated at activation, deterministically): a change never drops
the member count below the active threshold; `set-threshold` must satisfy
`1 ≤ k ≤ n`; under S1 fixed mode, removing the configured proposer is a no-op
activation (logged) — under 008.2 rotation the schedule simply re-derives.
An activation that violates a guard rail is void (deterministic no-op on all
nodes), never a stall.

### 2.4 Config, modes, migration, break-glass

```yaml
yano.app-chain.membership.mode: static | governed    # default: static (today's behavior)
yano.app-chain.membership.approval-window-blocks: 600
```

- **`static`** (default): exactly today — config list + admin-runbook
  rotation (E4.5). No behavior change for existing chains.
- **`governed`**: the config member list is the **genesis epoch only**; all
  later changes are chain transactions. The admin REST endpoints
  (`admin/members/add|remove`, `admin/threshold`) stop mutating locally and
  instead **submit the corresponding governance command** signed by this
  node's key — the operator UX stays identical, but one call from
  `threshold` operators replaces "run the same call on every node".
- **Break-glass**: `admin/members/reset?force=true` keeps the node-local
  override path (persisted, wins over derived epochs, loudly logged) for the
  disaster case where governance itself is wedged (e.g. lost keys below
  threshold). Documented as an explicit trust-model deviation.
- All members must agree on the mode (fail-closed like machine id / 008.2).
- Migration static→governed: coordinated restart with `mode: governed`; the
  current effective member set (config or persisted E4.5 override) becomes
  the genesis epoch.

### 2.5 Extensibility: governance modes are pluggable too

Like sequencer modes (008.2 §2.7), `static` and `governed` are two built-in
implementations behind `yano.app-chain.membership.mode`, and the framework
handler seam is an SPI so richer models (explicit yes/no ballots, separate
governance quorum, external approval systems) can ship later or be supplied
by applications as plugins/library code:

```java
public interface MembershipGovernance {
    String id();                                   // "static", "governed", custom id
    void init(GovernanceContext ctx);              // MemberGroup ops, ledger meta, settings
    /** Called for every finalized ~governance/membership message, inside the
     *  block's atomic commit — must be deterministic over finalized history. */
    void onGovernanceMessage(AppMessage msg, long height, GovernanceBatch batch);
}
// ServiceLoader: MembershipGovernanceProvider { id(); create(ctx); }
```

Unknown mode id fails startup; all members must run the same id
(fail-closed). Cert verification (`membersAt`/`thresholdAt`) stays outside
the SPI — a custom governance mode decides *how epochs are agreed*, never
*how certs are checked*.

### 2.6 What this unblocks

- **008.2 rotation** composes: `proposerFor` uses `membersAt(h)`, so governed
  changes rotate the schedule automatically.
- **Governed parameter blocks** (ADR-006 §7.4, ZK VK distribution; block
  limits) reuse the same topic/handler/approval mechanics later
  (`~governance/params`) — this ADR builds the seam generically
  (`GovernanceHandler` keyed by sub-topic) but ships only membership.
- ADR-007 P1.2 ("membership changes as finalized governance blocks,
  replayable, multi-party authorized") is fully addressed.

## 3. Test plan (definition of done)

1. **Threshold approval**: 2-of-3 chain — one member's add-command does
   nothing; the second identical command activates the epoch at `h + lag`;
   certs before/after verify against the right epochs.
2. **Determinism**: all nodes derive identical epoch lists; state roots and
   cert verification agree across nodes after multiple governed changes.
3. **Late joiner**: fresh node catches up through governed rotations and
   verifies all history (epochs derived purely from replay).
4. **Guard rails**: below-threshold removal and invalid thresholds are void
   deterministically on every node.
5. **Approval window**: a stale half-approved command expires after the
   window; re-submission restarts the count.
6. **Non-member injection**: rejected at admission and — enforcement-grade —
   an in-block governance message from a non-member fails block verification.
7. **Break-glass**: forced local reset overrides derived epochs, survives
   restart, logs the deviation; chain continues.
8. **Mode mismatch**: governed node + static node fail closed (no divergence).
9. **Rotation composition** (with 008.2): schedule follows governed changes.
10. Devnet skill extension: governed add/remove cycle on the two-node cluster.

## Delivery notes

### I2.2 — delivered 2026-07-10

- `GovernedMembership` (runtime): deterministic per-block handler — command
  codec (`[version, op, subject, activation-lag]`), pending-approval state
  keyed by canonical command bytes, threshold-identical activation, window
  expiry, guard rails voiding deterministically. Pending state
  (`gov_pending`) and `member_epochs` are written **inside the block's atomic
  commit batch** (`commitBlock` overload with meta writes); the in-memory
  group is updated during processing so same-block commands see new epochs.
- Routing split: only `~consensus/*` takes the engine fast-path;
  `~governance/*` messages are **sequenced like ordinary messages** (pooled,
  selected, finalized) but bypass state-machine `validate()` (machines must
  not veto governance; their `apply()` skips the opaque bodies
  deterministically — blob-first holds).
- Admin endpoints in governed mode SUBMIT commands (`submitGovernance`
  internal path; public submit still rejects `~` topics); `resetMembers`
  stays the break-glass local override with a loud WARN. Config:
  `membership.mode` / `membership.approval-window-blocks` via the
  pluginSettings prefix seam (zero config-record changes).
- **Governed bootstrap**: a later-added member starts with the ORIGINAL
  genesis list (it isn't in it) — the self-membership constructor check
  relaxes to a WARN in governed mode; catch-up derives its epochs.
- **Bug found by the suite run (real, not test-only)**: a proposal arriving
  at a follower whose previous commit was still in flight was dropped
  ("expected height") — and since the transport never re-delivers an acked
  message id within a session, the round stalled permanently under load.
  Proposals ahead of the local tip now take the bounded defer-retry path
  (window widened to `max(4×interval, 15s)`); catch-up cannot cover this
  case because unfinalized heights aren't fetchable.
- Tests (`GovernedMembershipIntegrationTest`, 3): threshold activation on
  both nodes + ordinary traffic after + **late joiner deriving the full
  membership history purely from replay** (fresh node, genesis config,
  becomes a member via derived epochs, equal roots); invalid-threshold guard
  rail void with the chain alive; half-approved command expiry after the
  window. Full appchain suite (23 classes) green.
- *Deferred:* the generic `MembershipGovernance` SPI (§2.5) — the built-in
  handler landed with the seam shaped for it (mode string + isolated class),
  but the public SPI/provider extraction waits for a second governance model
  (same just-in-time rule as everything else). `~governance/params` reuse
  likewise waits for the ZK VK governance work.
