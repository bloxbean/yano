# ADR-013.1: Effect Runtime Framework Closure

- **Status:** Accepted — implemented and verified 2026-07-17
- **Date:** 2026-07-17
- **Parent:** [ADR-013](013-first-party-integration-connectors-and-effect-demo.md)
- **Related:** [ADR-010](010-deterministic-effect-system.md),
  [ADR-010.1](010.1-emission-versioning.md),
  [ADR-011](011-plugin-architecture.md)

## 1. Context

ADR-013 Milestone 1 proved the first-party Kafka, S3-compatible object-store,
and Kubo/IPFS connectors through real three-member, failover, retained-restart,
Compose, and normally started host deployments. That baseline deliberately did
not change the effect framework.

Milestone 2 closes three bounded framework gaps:

1. executor ownership is currently discovered by repeatedly calling the
   predicate `supports(type)`, so duplicate ownership is resolved by iteration
   order rather than rejected;
2. an incorporated effect result can update deterministic state, but cannot
   directly emit the next deterministic effect; and
3. host operations cannot observe the already-created connector product
   without either inventing another client or exposing live plugin calls to a
   status/metrics request.

This ADR freezes those APIs, compatibility rules, activation behavior, and
lifecycle boundaries before implementation. It does not change effect record
version 1, effect ids, result messages, roots, finality gates, executor retry
semantics, or the external-worker ownership rules of ADR-010.

## 2. Decision summary

1. Add `AppEffectExecutor.effectTypes()` as a default method returning an empty
   set. A non-empty declaration is authoritative. Empty means a legacy
   predicate-only executor whose `supports(type)` is consulted only when no
   declarative owner exists.
2. Snapshot and validate declarations during Effect Runtime construction,
   before runtime ownership or an intake cursor is published. Duplicate
   declarative owners fail startup deterministically.
3. Add an emitter-capable `AppStateMachine.onEffectResult(...)` default
   overload that delegates to the existing callback. The engine invokes only
   the new overload.
4. Incorporated results are delivered in their existing deterministic order,
   followed by deterministic expiries, followed by normal block application.
   One block-scoped emitter assigns global ordinals across all three stages.
5. Direct result-driven emission is a state-machine behavior change and is
   gated by an explicit ADR-010.1 activation height committed in the chain
   profile. Absence of the activation retains the old continuation-command
   flow.
6. Add an immutable executor operations snapshot. The Effect Runtime schedules
   sampling from its lifecycle tick onto a dedicated daemon pool and caches a
   sanitized, bounded host view. HTTP and Prometheus read only that host-owned
   cache and never invoke a connector or plugin callback.

## 3. Declarative effect ownership

### 3.1 SPI

`AppEffectExecutor` gains the source- and binary-compatible method:

```java
default Set<String> effectTypes() {
    return Set.of();
}
```

The declaration belongs to the created executor product, not the factory or
bundle manifest. A factory may create a different product set from validated
configuration, so only the products actually owned by a chain can truthfully
declare active ownership. A later manifest schema may duplicate this data for
static tooling, but it cannot replace product validation.

First-party products declare exactly one frozen action type:

| Product | Type |
|---|---|
| Webhook | `webhook.post` |
| Cardano payment | `cardano.payment` |
| Kafka publish | `kafka.publish` |
| S3-compatible object put | `object.put` |
| Kubo/IPFS pin | `ipfs.pin` |

### 3.2 Validation

Construction snapshots the returned collection and its elements under the
existing plugin callback/TCCL/lifecycle boundary. The declaration must:

- be non-null and contain at most 64 unique entries;
- contain no null, blank, untrimmed, reserved-`~`, non-ASCII, or over-128-byte
  type;
- use ASCII letters or digits first and ASCII letters, digits, `.`, `_`, `-`,
  `:`, or `+` thereafter; and
- not declare the same configured type from two products.

Validation happens before `AppLedgerStore.bindFxRuntimeOwner`, cursor
initialization, workers, or runtime publication. A failed start closes every
new product through the existing reverse-order rollback path.

A conflict diagnostic lists the effect type and, for every owner, the bundle
id (or `system`/`legacy`), contribution scheme, and stable executor id. It does
not print connector configuration, endpoints, credentials, payloads, or
exception messages supplied by a plugin.

### 3.3 Routing and migration

For a type with a declared owner, routing is a constant lookup and
`supports(type)` is never called. The declaration is authoritative even if a
legacy predicate implementation would disagree.

If a type has no declared owner, the runtime evaluates only empty-declaration
legacy products in their existing stable order. This preserves old executors
without allowing them to override a declaration. Multiple legacy predicates
remain a compatibility limitation and generate a bounded warning; operators
should migrate those products to declarations. A future major version may
remove predicate routing only after a separate deprecation ADR.

`effects.executor.types` remains a local execution partition. When non-empty,
duplicate validation and routing are limited to those configured types. When
empty, every declared type is validated and routable. This does not alter
which consensus effects exist.

## 4. Emitter-capable result callback

### 4.1 SPI and compatibility

`AppStateMachine` gains:

```java
default void onEffectResult(
        AppBlock block,
        EffectResult result,
        AppStateWriter writer,
        AppEffectEmitter effects) {
    onEffectResult(block, result, writer);
}
```

The runtime, startup wrapper, and plugin facade invoke/forward only this new
overload. An existing binary or source implementation continues through the
default delegate exactly once. A new implementation may override only the new
overload. The old method remains available to direct callers and tests.

### 4.2 Block ordering and atomicity

The kernel first interprets and stages all valid incorporated result messages
in message order, then stages deterministic expiry results in expiry-index
order. It creates one rejecting-or-active `BlockEmitter` for the block and
delivers:

1. valid incorporated result callbacks in message order;
2. deterministic expiry callbacks in expiry-index order; and
3. the ordinary `apply(block, writer, effects)` callback.

The emitter ordinal starts at zero and advances across that complete order.
All callback writes, emitted records, effect/result indexes, and the resulting
roots remain in the existing atomic block batch. Existing count, payload,
result-window, and expiry limits apply to the aggregate emissions from the
whole block. A callback failure aborts the block batch.

`CONFIRMED`, `FAILED`, `CANCELLED`, and `EXPIRED` are all delivered once after
their terminal transition is accepted. The state machine decides whether an
outcome emits a follow-up; the framework adds no outcome-specific I/O.

### 4.3 Activation

The evidence registry direct-continuation behavior is controlled by the
committed setting:

```text
machines.evidence-registry.activations.direct-result-emission=<height>
```

At heights before the activation, or when the key is absent, storage results
only update deterministic business state and `evidence.notify` remains the
explicit Kafka continuation. At and after the activation, incorporation of
the second required successful storage result emits the one `kafka.publish`
effect directly. Replayed blocks use their block height, never wall-clock or
local configuration timing.

The effective machine configuration digest/profile must include this setting
under the existing chain-profile rules. Activation tests cover A-k, A-1, A,
A+1, A+k, multi-result blocks, restart, and snapshot replay. The effect record
format remains version 1 because the format does not change.

## 5. Executor observability bridge

### 5.1 Product snapshot

`AppEffectExecutor` gains a default immutable operational snapshot whose
default is `UNKNOWN` with zero counters. A snapshot contains only:

- bounded readiness: `READY`, `DEGRADED`, `UNAVAILABLE`, or `UNKNOWN`;
- monotonic attempts, successes, retryable failures, and terminal failures;
- bounded in-flight count;
- last-success and last-failure monotonic ages as normalized buckets; and
- a bounded framework-defined failure code, never an exception message.

It contains no endpoint, topic/bucket/path, credential, payload, external
reference, arbitrary label, client object, mutation method, or connector
extension map.

### 5.2 Sampling ownership

The lifecycle-owned executor instance maintains its local counters and
returns a memory-only snapshot. The Effect Runtime initiates sampling from the
existing tick but invokes the admitted plugin callback on a dedicated daemon
pool so a blocked observation cannot stop effect scheduling. It writes an
immutable sanitized copy to a host-owned cache.

Status, health, metrics, and dashboard code read only that cache. They never
call `operationalSnapshot()`, `supports()`, `execute()`, a connector client, or
an external service. Scrape storms therefore cannot amplify connector traffic
or plugin callbacks. Stale samples degrade to a framework-owned stale state.

The cache is generation-scoped, bounded by the validated executor inventory,
and discarded on replacement, partial startup rollback, or close. Shutdown
seals sampling before product close and cannot resurrect a prior generation.
The contract uses Java data only and requires no reflection or dynamic proxy,
so it remains native-image compatible.

### 5.3 Frozen operational bounds and surfaces

The initial implementation fixes these framework-owned bounds rather than
adding connector-specific tuning:

- at most 256 created executor products per chain;
- at most one observation callback in flight per product;
- sampling initiated no more often than once per second;
- a 100 ms callback budget, after which the cached view reports
  `TIMED_OUT`/`CALLBACK_TIMEOUT` without blocking a request thread; and
- a five-second freshness window, after which the cached view reports
  `STALE` until a valid observation is published.

The cache exposes only stable executor identity/source fields, declared effect
types, sample state, readiness, the four monotonic counters, in-flight count,
normalized success/failure age buckets, and one framework failure code. The
status API and app-chain dashboard render that fixed view. Prometheus registers
startup-bounded readiness, in-flight, age-bucket, and monotonic counter series
using only chain, executor, slot, fixed readiness, and fixed event tags.

All shipped executors (`webhook.post`, `cardano.payment`, `kafka.publish`,
`object.put`, and `ipfs.pin`) use the common tracker. Legacy/custom products
remain compatible through the default all-zero `UNKNOWN` snapshot.

## 6. Alternatives rejected

- **Factory-only or manifest-only declarations:** cannot describe the active
  products created from chain configuration.
- **Keep first-match predicate routing:** classpath/discovery order is not an
  ownership rule and silently masks conflicting connectors.
- **Call both result callbacks:** duplicates state transitions and effects.
- **Create an internal continuation command:** preserves an unnecessary extra
  block/command once the deterministic callback can emit directly.
- **Probe connectors from HTTP/Prometheus:** couples availability and load to
  scrape traffic and risks duplicate clients, credential exposure, and
  shutdown races.
- **Put executor observations in consensus state:** attempts differ by member
  and must remain node-local under ADR-010.

## 7. Verification and rollout

Implementation proceeds in parent-ADR phases 2.1 through 2.4. Acceptance
requires:

1. duplicate declaration, malformed declaration, first-party declaration,
   legacy fallback, plugin TCCL/timeout, rollback-close, JVM catalog, and native
   catalog tests;
2. exactly-one callback tests for legacy and new state machines plus global
   ordinal/cap/atomicity and every terminal outcome;
3. evidence old-profile and activated-profile A±k replay with byte-identical
   roots across members, restart, and snapshot;
4. cached-observability stale/blocked/scrape-storm/redaction/cardinality/
   replacement/shutdown tests; and
5. the full ADR-013 connector fault, failover, deployment-parity, packaging,
   native, cleanup, and three-member anchored scenario suites.

The ADR moves to Accepted only after all of the above pass and the parent
Milestone 2 acceptance checklist has no unresolved correctness or high-severity
finding.

## 8. Implementation and acceptance evidence

Milestone 2 is implemented and Accepted. The delivered runtime now:

- validates bounded declarative effect-type ownership before runtime
  publication, rejects duplicate owners, and preserves predicate-only legacy
  executors as the documented fallback;
- invokes exactly one emitter-capable result callback with one block-wide
  ordinal sequence shared by incorporated results, expiries, and ordinary
  application;
- preserves the old explicit `evidence.notify` profile while an activated
  profile can emit `kafka.publish` directly after both storage results are
  incorporated; and
- samples executor-owned operational snapshots into a bounded runtime cache
  which is consumed by status, Prometheus, and the dashboard without invoking
  connectors from request or scrape threads.

Acceptance covered malformed and duplicate declarations, lifecycle rollback,
legacy compatibility, callback ordering and atomicity, every terminal result,
activation-boundary replay, snapshot/restart behavior, stale and blocked
observations, redaction, bounded metric labels, and shutdown/replacement
races. JVM/plugin packaging and native-catalog contracts remained green.

The complete old-profile scenario passed the three-member executor-failover
and retained-restart test. The activated direct-continuation scenario passed
the Compose-versus-normal-deployment parity test, including byte-identical
external artifacts and semantics, effect/result proofs, plugin/status/UI
inventory, executor metrics, and retained replay. Live testing also found and
fixed delayed metric registration: Prometheus now freezes its executor meter
inventory when the runtime-owned descriptor snapshot first becomes available,
rather than assuming that inventory exists at the host startup event.

No effect-record format, effect id, finality, retry, or result-message change
was required; emission remains ADR-010.1 version 1.
