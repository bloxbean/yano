# ADR-010.1: Emission Versioning — Replay-Stable Effect Upgrades

## Status
Proposed — gates the FX-M1 SPI freeze (ADR-010 §15, hard requirement #2)

## Date
2026-07-13

## Parent
`adr/app-layer/010-deterministic-effect-system.md` — pre-implementation design
item #1. Precedents: ADR-008.3 (activation-lag membership epochs), Temporal
`getVersion` markers, ADR-005 §4.4 (byte-for-byte root verification).

---

## 0. In plain words

Once effects ship, *what a machine emits* becomes part of the replayed
history, exactly like what it writes to state: every block's ordered effect
list is hashed into `~fx/root/<height>`, which lives under the state root that
every follower re-derives and verifies byte-for-byte. If version 2 of a
machine emits even slightly differently than version 1 did **for an old
block**, a node catching up with v2 code recomputes a different root for that
old block — and can *never* sync. Every careless emission change is a
permanent fork.

The fix is the same one Temporal and the chain's own governed membership use:
new behavior activates at a **declared point in history** (a block height),
old behavior is kept for heights below it, and replay picks the branch by the
block's height — which is deterministic on every node, forever.

## 1. Problem statement

Three concrete failure scenarios (from the ADR-010 adversarial review, ranked
its #1 risk):

- **S1 — catch-up fork.** Machine v2 changes an emission condition. A fresh
  follower (or a node rebuilding a corrupted outbox) replays history with v2
  code: block 500's effect list differs from what the network certified ⇒
  `~fx/root/500` differs ⇒ state root differs ⇒ the certified block is
  rejected. The node is permanently unable to join.
- **S2 — silent outbox divergence.** Even where the root check would not fire
  (a chain that never enabled the `effectsRoot` leaf — not possible under
  ADR-010, but the failure class exists), a replayed node would derive a
  *different pending set* and execute effects the network never authorized.
- **S3 — result-handler drift.** `onEffectResult()` writes app state
  deterministically from sequenced results. Changing that logic changes
  historical state transitions on replay — the same fork, arriving through
  the result path instead of the emission path.

Scope note: this discipline is not new — **any** change to `apply()` state
writes already has this property (ADR-005 §4.4). Effects merely make the
requirement unavoidable and give us the occasion to formalize the mechanism.
Everything below applies equally to `apply()` state-write changes,
`apply()` emission changes, and `onEffectResult()` changes. Collectively:
**transition-logic changes**.

## 2. Design

### 2.1 Activation heights in chain settings

A transition-logic change is named and activated at an explicit height,
declared in chain configuration:

```properties
# consensus-affecting — MUST be identical on every member (same class as
# state-machine id, threshold, effects.max-per-block)
yano.app-chain.machines.<machine-id>.activations.<change-name>=<height>
# e.g.
yano.app-chain.machines.approvals.activations.payment-effects=125000
yano.app-chain.machines.approvals.activations.payment-v2-split=310000
```

Machines read these via the existing `AppStateMachineContext.settings()`
sub-map — no new plumbing. Change names are free-form, permanent identifiers
(never reused, never renamed once any member has them configured).

### 2.2 The `ActivationSchedule` helper (core-api)

```java
// core-api: com.bloxbean.cardano.yano.api.appchain.effects
public final class ActivationSchedule {

    /** Parse every "activations.<name>=<height>" entry from a machine's settings sub-map. */
    public static ActivationSchedule from(Map<String, String> settings, String machineId) { ... }

    /**
     * True when the named change is active at this height. A change that is
     * absent from configuration is NEVER active — deploying a jar whose new
     * branches are all gated is therefore safe before the config lands.
     */
    public boolean isActive(String changeName, long height) { ... }

    /** All configured names → heights, for status/diagnostics surfaces. */
    public Map<String, Long> entries() { ... }
}
```

Usage inside a machine:

```java
private ActivationSchedule activations;   // built once in init()/provider

@Override
public void apply(AppBlock block, AppStateWriter writer, AppEffectEmitter effects) {
    ...
    if (approved) {
        if (activations.isActive("payment-v2-split", block.height())) {
            emitSplitPayments(effects, item);          // new behavior
        } else if (activations.isActive("payment-effects", block.height())) {
            emitSinglePayment(effects, item);          // v1 behavior, kept verbatim
        }
        // below the first activation: no emission — the chain's pre-effects era
    }
}
```

**The missing-key default is the load-bearing safety property**: absent ⇒
inactive ⇒ old behavior. A jar rollout and a config rollout can therefore
happen in either order *per node* as long as the config lands on all members
before the activation height arrives (§4 runbook).

### 2.3 What the framework enforces (and what it cannot)

The framework needs **no new consensus machinery**: a violated rule surfaces
as a divergent `~fx/root/H` (or app-state) leaf ⇒ divergent state root ⇒
certified block rejected by the *existing* byte-for-byte check. What 010.1
adds instead:

1. **Convention + helper** (§2.1/2.2) so the safe pattern is the easy pattern.
2. **Conformance replay-matrix** (§3) so violations are caught on the
   developer's machine, not in production catch-up.
3. **Startup diagnostic**: the subsystem logs each machine's activation
   entries at chain start, and `status()` exposes them — operators can
   eyeball-compare members before an activation height arrives. (A future
   membership-handshake cross-check of consensus-affecting settings is noted
   in ADR-010 §11 risk 5 and stays out of scope here.)
4. **Framework-level changes follow the same rule**: the FxKernel's canonical
   `EffectRecord` encoding and `effectsRoot` computation are versioned by the
   record's leading `v` field; any layout/algorithm bump ships gated on a
   framework activation key (`yano.app-chain.effects.activations.<name>`) with
   identical semantics. The kernel is "just another machine" for versioning
   purposes.

What it cannot enforce: a developer editing old branch code in place. That is
what the replay-matrix (§3) and review discipline are for; the ADR-010 MUST
list already requires conformance coverage before deploying machine changes.

### 2.4 Executor-side versioning is a different (non-consensus) problem

Emission determinism concerns *what is recorded on-chain*. The executor's
interpretation of a payload (effect **type** schema) is off-chain and may
evolve independently — an old pending effect must remain executable after an
executor upgrade. Convention: type names may carry a schema suffix
(`cardano.payment.v2`) when a payload format changes incompatibly; executors
declare support per exact type string, so old pending `cardano.payment`
effects keep routing to the executor that still understands them. No
consensus impact; no activation heights needed.

## 3. Conformance replay-matrix (harness extension)

`StateMachineConformance` (008.1 I1.6) gains an upgrade mode that encodes S1
directly:

```java
StateMachineConformance.upgrade(oldProvider, newProvider)
        .settings(commonSettings)                        // includes activations.* for BOTH
        .activationAt("payment-v2-split", 60)            // the change under test
        .blocks(100).messagesPerBlock(5).seed(42)
        .bodyGenerator(...)
        .assertReplayStable();
```

Semantics:

1. **Baseline run**: corpus applied end-to-end with `oldProvider` (activation
   key absent) — roots and per-block effect lists recorded.
2. **Upgrade-replay run**: the SAME corpus applied end-to-end with
   `newProvider` and the activation configured at height A. Assert:
   - roots and effect lists at every height `< A` are **byte-identical** to
     the baseline (replay stability — the S1 test);
   - heights `≥ A` are internally deterministic across N repeated runs
     (the existing determinism property, applied to the new branch).
3. **Mixed-restart run**: kill-and-reopen at `A±k` (reusing the existing
   restart machinery) — upgrade boundaries must survive crash replay.

The existing single-version `assertDeterministic()` additionally starts
asserting identical **effect lists** (not just roots) in every run, so
emission nondeterminism is diagnosed as such rather than as an opaque root
mismatch.

## 4. Rollout runbook (operator-facing)

For a transition-logic change on a live chain:

1. Pick the change name and an activation height comfortably in the future
   (rule of thumb: ≥ 1 day of blocks past the slowest member's expected
   restart completion).
2. Deploy the new machine jar to members in any order, at any pace — all new
   branches are inactive without the config key.
3. Set the identical `activations.<name>=<height>` on **every** member
   (config + restart, or the future governed path §5). Verify via `status()`
   that all members report the same entries **before** the height arrives.
4. The height passes; the new branch activates simultaneously everywhere, in
   consensus.
5. Never edit or remove the old branch while any node may still replay
   heights below A — concretely, an old branch may be deleted only when the
   chain's **snapshot floor** (the lowest height a joining node would ever
   replay from, per the snapshot onboarding + body-pruning posture) is ≥ A.
   Chains that onboard exclusively from recent snapshots can retire old
   branches aggressively; archival chains keep them.

Failure containment: if a member misses step 3, it diverges at height A — a
**halt** (its blocks are rejected / it rejects certified blocks), not silent
corruption, and the fix is config + restart + catch-up. This is the same
failure class and remedy as any consensus-config mismatch.

## 5. Future: governed activation (deferred)

Config-distributed activation heights carry a residual mismatch risk (§4
step 3 is manual). The eventual fix is the 008.3 pattern: a
`~governance/activation` sequenced message (change name + height + k-of-n
echo + activation lag) making the schedule itself part of chain history —
replayable, mismatch-proof, and auditable. Deferred until governed membership
mode has production mileage; the `ActivationSchedule` API is deliberately
shaped so its backing source can switch from settings to chain state without
touching machine code.

## 6. Decisions

| # | Decision |
|---|---|
| 010.1-D1 | Transition-logic changes (emission, state writes, result handling) activate at explicit heights; replay selects the branch by `block.height()` |
| 010.1-D2 | Activation heights live in chain settings under `machines.<id>.activations.<name>` (consensus-affecting); missing key ⇒ inactive ⇒ old behavior |
| 010.1-D3 | `ActivationSchedule` helper in core-api effects package; machines never parse settings ad hoc |
| 010.1-D4 | Conformance harness gains `upgrade()` replay-matrix (baseline vs upgrade-replay vs mixed-restart) and asserts effect lists, not only roots |
| 010.1-D5 | FxKernel record/effectsRoot changes use the same mechanism under `effects.activations.<name>` |
| 010.1-D6 | Old branches retire only when the snapshot floor passes the activation height |
| 010.1-D7 | Effect-type payload schema evolution is executor-side, via type-string versioning — no consensus involvement |
| 010.1-D8 | Governed activation via `~governance/activation` is the designed future, not v1 |
