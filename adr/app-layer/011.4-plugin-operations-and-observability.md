## Proposal-time context

ADR-011.1 through ADR-011.3 provide deterministic bundle discovery, lifecycle,
typed activation, committed queries, and constrained domain APIs. The runtime
could identify which code was installed, selected, and fingerprinted, but an
operator could not answer these node-local questions through one stable
surface:

- Which selected bundle and contribution is active, stopped, degraded, or
  failed on this node?
- Are a plugin's external dependencies healthy?
- Which bounded plugin metrics should be exported through the host's existing
  Micrometer/Prometheus integration?
- Can an artifact be inspected and validated without loading provider code?

`PluginCatalogView` is intentionally immutable. Its selected-catalog
fingerprint is deployment evidence and may be compared across consensus nodes.
Lifecycle, health, counters, and failures are non-deterministic node-local
state and must never be added to that fingerprinted model.

The application already had SmallRye Health, Micrometer, and native-safe static
status pages. Plugin code must not receive those framework objects. Calling a
plugin directly from an HTTP health request or Prometheus scrape would let one
stuck callback exhaust request threads and keep a plugin class loader alive
outside the ADR-011 lifecycle fence.

## Goals

- Add a separate immutable-snapshot operations view without changing catalog
  identity or consensus behavior.
- Let manifested bundles contribute bounded health and metric snapshots under
  the existing catalog, TCCL, lifecycle, and packaging rules.
- Isolate observability failures as `AUXILIARY_LOCAL`; optional telemetry must
  not prevent node startup or make main readiness fail.
- Serve HTTP health, metrics, and dashboard reads entirely from a host cache.
- Bound callback concurrency, time, schema size, series cardinality, text, and
  response size.
- Provide deterministic offline `validate` and `inspect` commands that scan
  artifacts without loading provider classes.
- Work identically for directory JVM, packaged JVM, and build-time native
  catalogs, subject to native images not loading new JARs after build.

## Non-goals

- Replacing Java `ServiceLoader` or the ADR-011 catalog with another plugin
  framework.
- Making plugin health part of consensus, the catalog fingerprint, liveness,
  or readiness.
- Exposing Micrometer, MicroProfile Health, JAX-RS, CDI, Quarkus, a raw
  scheduler, or HTTP objects to plugins.
- Arbitrary plugin-defined CLI commands, dashboards, metric exporters, labels,
  or management actions.
- Hot reload, hard CPU/heap isolation, signed artifacts, or remote plugin
  installation.
- Treating a health response or metric as an authenticated state proof.

## 1. Static catalog and dynamic operations stay separate

`PluginCatalogView` and `PluginBundleInfo` remain unchanged. A new
`PluginOperationsView` returns an atomically published,
defensively copied `PluginOperationsSnapshot`. Reading that snapshot never
loads a provider, invokes plugin code, waits for a plugin lock, or changes
runtime state.

The snapshot contains:

- the catalog fingerprint it describes;
- a monotonically increasing node-local generation and capture timestamp;
- bounded aggregate counts;
- deterministic bundle and contribution runtime entries;
- bounded, canonical per-check health entries for selected health sources;
- cached custom metric series.

Every contribution also carries an explicit `lifecycleObserved` flag. The
operations registry currently observes `node-plugin`, `domain-api`, `health`
and `metrics` callbacks. The other six typed app-chain factory contributions
remain catalog-visible but publish `lifecycleObserved=false`, `VALIDATED` and
`UNKNOWN`; the platform must not infer an active instance merely because a
provider was selected. Per-chain lifecycle observation for those factories is
deferred until their owning managers expose one common attribution contract.
In API v1, an optional instance entry is only a scope-qualified projection of
its contribution and must exactly match that contribution's lifecycle, health,
failure, and stale state. A future contract that aggregates divergent
multi-instance states requires an explicit plugin API-level advance and one
shared contribution-aggregation policy before it can be published.

Runtime lifecycle states are:

```
NOT_SELECTED -> VALIDATED -> ACTIVATING -> ACTIVE -> STOPPED -> CLOSED
                                  \-> FAILED
```

Health states are `UNKNOWN`, `UP`, `DEGRADED`, and `DOWN`. A failure contains
only a platform-owned enum code and observation timestamp. Public snapshots do
not contain exception messages, stack traces, configuration, paths,
class-loader text, or caller data.

The catalog fingerprint is copied into the dynamic snapshot as a join key; no
dynamic field is an input to its calculation. Two nodes can therefore have the
same fingerprint and state root while reporting different local health.

Bundle lifecycle, health, failure, metrics-stale state, and the corresponding
host totals are derived from contribution runtime entries by one pure core
aggregation policy. The runtime uses that policy to publish each bundle, and
`PluginOperationsSnapshot` applies the same policy when validating any
independently supplied view. A summary that contradicts its contribution
detail is rejected before an HTTP, health, metrics, or dashboard adapter can
observe it. `REQUIRED` and `CONSENSUS` lifecycle failures remain decisive over
local contribution failures; callback counts and transition timestamps stay
host observations and are not reconstructed by this policy.

## 2. Manifested auxiliary telemetry SPIs

Two manifest-required ServiceLoader kinds are added:

| Manifest kind | SPI | Purpose | Trust tier |
|---|---|---|---|
| `health` | `PluginHealthProvider` | fixed health checks plus cached reports | `AUXILIARY_LOCAL` |
| `metrics` | `PluginMetricsProvider` | fixed metric descriptors plus cached values | `AUXILIARY_LOCAL` |

A bundle may declare at most one provider of each kind. The manifest
contribution name and provider `id()` must equal the owning bundle id. Neither
kind has a synthetic legacy adapter: unmanifested providers are rejected.

This keeps optional telemetry separate from `NodePlugin`, whose lifecycle is
`REQUIRED`, and allows a typed-only bundle to expose operations without adding
a general node plugin. The existing catalog remains the only discovery and
selection mechanism.

### 2.1 Health contract

Conceptually:

```java
interface PluginHealthProvider {
    String id();
    PluginHealthSource create(PluginHealthContext context);
}

interface PluginHealthSource extends AutoCloseable {
    List<PluginHealthCheckDescriptor> checks();
    PluginHealthSnapshot snapshot();
}
```

The source declares its complete check schema once. A snapshot must contain
exactly one report for every descriptor. Plugin-supplied runtime messages and
detail maps are not accepted in v1; each report contains only its declared
check id and a health enum. This avoids making exception text or external data
part of an operator API.

Limits are fixed and fail closed:

- at most 16 checks per bundle and 512 host-wide;
- check ids match `[a-z][a-z0-9_.-]{0,63}`;
- static ASCII descriptions are at most 256 characters;
- duplicate, missing, extra, or null identities are invalid; publication is
  canonicalized by descriptor id.

The v1 context exposes the owning bundle id and an empty configuration map.
A typed ordinary-value/secret-reference contract is required before any
bundle configuration is passed to an observability source.

### 2.2 Metric contract

Conceptually:

```java
interface PluginMetricsProvider {
    String id();
    PluginMetricsSource create(PluginMetricsContext context);
}

interface PluginMetricsSource extends AutoCloseable {
    List<PluginMetricDescriptor> descriptors();
    PluginMetricSnapshot snapshot();
}
```

Supported metric types are:

- `GAUGE`: one finite value;
- `COUNTER`: a finite, non-negative, generation-monotonic total;
- `TIMER`: a non-negative, generation-monotonic count and total nanoseconds.

The descriptor schema freezes on activation. Snapshot keys must exactly equal
the descriptor ids. Plugins cannot contribute labels in v1; the descriptor id
is the bounded dimension and the host adds only its owning bundle tag.
Workload values can never create descriptors or labels.

Limits are:

- at most 64 series per bundle and 4,096 custom series host-wide;
- ids, names, and units use bounded printable ASCII grammars and descriptions
  are at most 256 characters;
- the host owns the `plugin` tag and `yano.plugin.*` metric namespace;
- NaN, infinities, negative counters/timers, regressions, duplicate ids,
  missing values, or schema changes invalidate the whole source snapshot.

Versions, digests, provider classes, paths, exception types/messages,
configuration, route ids, request values, and health reason text are never
metric labels.

## 3. Sampling, caching, and lifecycle

The runtime owns one `PluginOperationsRegistry`. It is created from the
validated catalog and owns all health/metric source products, the sampler,
cached snapshots, and transition counters.

Sampling rules:

- plugin callbacks run only on a dedicated four-worker auxiliary pool with a
  bounded queue of 256;
- one source has at most one admitted or running sample;
- the default interval is five seconds and the deadline is one second from
  admission, including queue time;
- queued work that expires or is cancelled is removed and never invokes the
  source;
- a running timeout interrupts caller wait but retains the catalog generation,
  callback admission, product, provider, and class-loader lease until the
  callback actually exits;
- a source that ignores interruption cannot consume HTTP, consensus, app-chain,
  or Prometheus threads, though trusted in-process code can still consume one
  bounded auxiliary worker;
- source callbacks and product close run with the owning plugin loader as TCCL
  and restore the caller's TCCL and interrupt state;
- process-fatal JVM failures retain process-fatal precedence.

Telemetry activation uses the same bounded auxiliary executor and never runs
provider lookup, provider identity, source creation or schema publication on
the startup caller. One generation-wide one-second admission deadline covers
all activation candidates. At most four candidates are submitted at once;
later catalog ordinals enter only as workers become available. A hostile early
ordinal can therefore make the unentered tail time out, but cannot turn startup
into one sequential deadline per plugin. Results commit strictly in catalog
ordinal order, so host-wide check/series capacity winners and published
failures are independent of thread scheduling. Installed sources close in the
reverse of that committed order.

The last valid sample remains available after an error and is marked stale.
Before the first valid sample, health is `UNKNOWN` and metrics are absent. A
timeout, invalid snapshot, or ordinary callback failure records only a stable
host failure code. A later valid sample clears the transient code and stale
flag. An explicit platform lifecycle failure cannot be masked by plugin health.

The operations snapshot publishes every activation-frozen health descriptor in
`bundleId`, then check-id order. Before the first valid sample, each entry is
`UNKNOWN`; it is not stale until a sampling failure or timeout is observed.
After a valid sample, each entry carries its cached status;
if a later callback times out, fails, or returns an invalid whole-source
snapshot, the last-good per-check statuses remain and every entry for that
source is marked stale. Bundle health is the worst cached check status, subject
to lifecycle failure taking precedence. Detailed checks are exposed only in
the privileged bundle-detail response and dashboard; they do not create
per-check Prometheus series or unbounded health-group data.

Auxiliary provider/source creation failure is isolated to that contribution,
marks it failed/degraded, and does not abort otherwise valid node startup.
Catalog/schema/service-correlation defects still fail before activation under
ADR-011.2.

Lifecycle order is:

1. validate and publish the immutable catalog;
2. initialize required `NodePlugin` and application contributions;
3. construct telemetry products in dependency order;
4. start ordinary runtime contributions, then start sampling last;
5. on stop, seal sampling first, cancel queued samples, and await only work
   that actually entered plugin code;
6. close telemetry products in reverse dependency order before provider and
   class-loader close.

The concrete runtime close sequence is domain APIs, `NodePlugin` manager,
operations registry, then plugin environment/class loader. Telemetry admission
is sealed before domain and ordinary plugin admission. A timed-out callback's
generation and loader lease remain owned until it exits. Because plugin code is
trusted in-process code, a source callback or `close()` implementation that
ignores interruption may delay graceful shutdown indefinitely; a process
supervisor must retain a force-termination policy for that case.

Restart creates a new sampling generation. Completion from an older generation
may release its retained lease but cannot publish into the replacement.

## 4. Platform-owned operations data and metrics

The operations registry also records bounded host-owned transitions without a
custom provider:

- selected/validated/active/stopped/failed contribution counts;
- NodePlugin lifecycle callback outcomes;
- provider/source activation and sampling outcomes;
- active sampling callbacks and stale sources;
- domain API admitted, active, queued, rejected, timed-out, succeeded, and
  failed request totals by bundle and fixed outcome.

Platform counters use catalog bundle ids and fixed enums only. They do not use
paths, routes, exception data, requests, or plugin-supplied strings as labels.
Instrumentation is observational and may not change callback ordering,
failure precedence, admission, or consensus results.

## 5. Host adapters

### 5.1 Operations REST API

The host owns a read-only namespace distinct from ADR-011.3 domain routes.
It is mounted below the artifact's API prefix (shown with the default):

```
GET /api/v1/plugin-operations
GET /api/v1/plugin-operations/bundles?after=<bundle-id>&limit=<1..100>
GET /api/v1/plugin-operations/bundles/{bundle-id}
```

The summary is bounded. Bundle results are deterministic and cursor-paginated;
the default page size is 50 and maximum is 100. Responses join the cached
dynamic entry with catalog-approved inventory fields and enforce a final
encoded response limit.

These are privileged operator endpoints even though they use GET. Until Yano
has a general management-auth realm, they reuse the existing full app-chain
API-key credential and fail closed:

- authentication disabled or no unscoped full key configured: `503`;
- missing or invalid key: `401`;
- topic-scoped key: `403`;
- unscoped full key: `200`.

Encoded, matrix, doubled, dot-segment, and trailing-slash aliases are rejected
before dispatch. Full inventory is not public merely because it is
secret-free.

### 5.2 Health

A separate SmallRye health group is exposed at:

```
/q/health/group/plugins
```

It reads the cache only and reports bounded aggregate counts plus at most 16
failing bundle ids. It may be DOWN for operator alerting when a selected
auxiliary contribution is degraded, down, failed, or stale. It is deliberately
not registered as liveness/readiness and not added to `NodeKernel`: optional
plugin degradation cannot make `/q/health/ready` fail. Required or consensus
plugin failures keep their existing owner semantics.

### 5.3 Micrometer/Prometheus

The application maps cached host values to Micrometer. A scrape never calls a
provider or source. Standard series use only bounded catalog/fixed-enum tags;
custom descriptors map under fixed host metric families with exactly
`plugin=<bundle-id>` and `metric=<descriptor-id>` tags. Descriptor name,
description, and base unit remain bounded REST/dashboard metadata and do not
become metric-family metadata or labels. The same cache feeds REST, health,
Prometheus, and the dashboard.

Counter and timer exports retain a monotonic host offset when a new runtime
generation resets the plugin's raw total. If a custom series is absent from an
authoritative operations snapshot, its Micrometer meter is unregistered rather
than exported forever; retained host state allows a later reappearance to
continue monotonically. A source explicitly retained as stale by the runtime
remains present with its last-good value. The host caps custom and operation
meter identities independently, so request paths and error strings cannot grow
the registry.

If an independently supplied operations view changes the metric type for an
existing `(bundle-id, descriptor-id)` across generations, the adapter removes
the old meter and state before registering the new type. Type-specific
continuity resets rather than silently exporting an old value under a new REST
descriptor. The runtime itself rejects such schema drift inside one registry
lifetime.

`/q/metrics` is a management surface and is not protected by the app-chain API
key. Deployments must bind/protect the Quarkus management listener
appropriately; ADR-011.4 does not imply otherwise.

### 5.4 Health dashboard

A native-safe, read-only page is served from `/ui/plugins/`. It consumes the
protected operations JSON rather than parsing Prometheus text, but displays the
same cached metric data that the Micrometer bridge exports. It includes catalog
identity, state/health totals, stale/failure indicators, search, contribution
inventory, and metric values.

The page has no remote assets or third-party scripts. Plugin-derived values are
inserted with DOM `textContent`, never `innerHTML`. The API key is accepted
interactively and retained only in memory or `sessionStorage`, never a URL,
log, or rendered node. One cancellable refresh may be outstanding at a time.
The public static shell reveals no inventory before successful authentication.
The dashboard never accepts an API prefix from a query parameter, fragment, or
other caller-controlled input. Before enabling credential entry, it fetches the
artifact-generated `/ui/plugins/api-prefix.json` document, validates one
canonical same-origin path, and binds stored credentials to that exact prefix.
Missing or invalid discovery data fails closed before a key is read, stored, or
sent. The prefix document, shell, and all assets receive CSP with
`frame-ancestors 'none'`, `DENY`, `nosniff`, `no-referrer`, and `no-store`
response headers.

The API prefix is an immutable artifact contract. Its sole build input is the
strict Gradle property `-PyanoApiPrefix=<path>`, defaulting to `/api/v1`. The
value is at most 256 characters and is either `/` or a canonical absolute path
of unescaped `[A-Za-z0-9._~-]+` segments, with no empty, `.` or `..` segment
and no trailing slash. Generation writes all of the following from that one
value:

- literal `quarkus.resteasy.path`, `yano.api-prefix`, and reserved
  `quarkus.http.root-path=/` build configuration;
- the raw `META-INF/yano-api-prefix-v1` classpath marker;
- immutable `/ui/plugins/api-prefix.json` dashboard discovery data.

Runtime-style system properties or environment variables for
`yano.api-prefix`, `quarkus.resteasy.path`, or `quarkus.http.root-path` are
rejected when supplied to the build, so Quarkus augmentation cannot capture an
untracked value. At process start, a high-priority guard compares effective
`yano.api-prefix` and `quarkus.resteasy.path` with the raw artifact marker and
independently requires `quarkus.http.root-path=/`. Any launch-time drift aborts
before node or plugin initialization. Reserving the HTTP root at `/` prevents a
second prefix from composing with the RESTEasy path.

Discovery and every key-bearing request reject redirects. The canonical root
API prefix `/` is supported without producing a doubled slash. Dashboard list
reads use pages of 100 and retain at most 500 bundles, bounding an automatic
refresh to five paginated list requests; the protected REST API remains
available for all 4,096 catalog entries and the UI explicitly reports when its
display cap is reached.

## 6. Catalog inspection CLI

The host supplies two offline commands around the same strict artifact scanner
and shared static validation logic used by catalog construction:

```
yano-plugins validate [policy options] <artifact>...
yano-plugins inspect [--format table|json] [policy options] <artifact>...
```

The CLI validates manifest/service correlation, duplicate identities,
compatibility ranges, dependency presence/version constraints, cycles, and
selection order. Output is deterministic. Diagnostics use safe basenames, not
absolute paths, and never print manifest bodies or configuration.

The CLI scans resources only. It must not load a provider class, run a static
initializer, call a constructor, or claim that structural validation proves
activation safety. Malformed catalogs return a stable nonzero exit code;
usage and I/O failures have distinct codes.

Artifact classification, manifest/service parsing and digest calculation use
one private immutable snapshot captured before classification. Ordinary-file
metadata is never accepted as evidence for bytes reopened later. Replacement,
same-size/timestamp ABA, file-key changes and symlink swaps either produce one
coherent old/new snapshot or fail safely; they cannot mix validation evidence
from one artifact with the digest of another.

Snapshot capture accepts at most one GiB for a JAR or for the aggregate regular
files in an exploded artifact. The overflow-safe limit is checked from the
initial no-follow metadata before creating the private copy and again while
streaming bytes, so a sparse or concurrently growing input cannot consume
unbounded temporary storage.

`PluginIndexGenerator` remains the build-index command. Offline dropped-JAR
inspection is a JVM tool. A native node cannot add a JAR after build; native
operators use the live operations endpoint for its embedded catalog.
Plugin-defined CLI commands remain deferred.

## 7. Failure and security policy

- All plugin exception text is untrusted and redacted from public responses,
  health, metrics, and default logs.
- Operations failure codes and timestamps are host generated.
- Telemetry callbacks are trusted in-process code, not a sandbox; hard
  isolation requires an out-of-process collector.
- A denied or unselected telemetry provider is never constructed.
- Inventory, health, or metrics never contain configuration values, secrets,
  authentication data, request bodies, external references, or filesystem
  paths.
- Cardinality limits are validated before publication; input is rejected, not
  truncated into a potentially colliding identity.
- Static health descriptions and metric metadata are bounded but still
  considered operator-visible plugin data.

## 8. Decisions

| ID | Decision |
|---|---|
| D1 | Dynamic operations state is separate from `PluginCatalogView` and its fingerprint. |
| D2 | Health and metrics use two manifest-only typed SPIs under the existing catalog. |
| D3 | Both telemetry kinds are `AUXILIARY_LOCAL`; ordinary failure is isolated. |
| D4 | HTTP, health, metrics, and UI read host-cached snapshots and never invoke plugin code. |
| D5 | Sampling is bounded, single-flight per source, deadline-aware, TCCL-fenced, and lifecycle-owned. |
| D6 | Metric schemas freeze at activation, custom labels are absent in v1, and series have hard per-bundle/host cardinality caps. |
| D7 | Plugin health is an operator health group, not node liveness/readiness or consensus. |
| D8 | Detailed operations REST is privileged and uses a namespace distinct from domain APIs. |
| D9 | The dashboard is a host-owned static read-only client of the protected operations view. |
| D10 | Catalog validate/inspect commands are host-owned and never load provider code. |
| D11 | Arbitrary plugin CLI, management actions, and alternative effect transports remain deferred. |
| D12 | Runtime lifecycle claims are explicit coverage: unobserved typed contributions stay catalog-valid/health-unknown rather than being reported active. |
| D13 | Auxiliary activation commits in catalog order under one bounded generation deadline; execution completion order is not observable policy. |
| D14 | Per-check health is a bounded privileged cached view; errors retain stale last-good values and never add per-check metric series. |
| D15 | API-v1 instance entries must exactly match their contribution summary; divergent multi-instance aggregation requires an API-level advance. |
| D16 | The dashboard discovers one immutable artifact API prefix generated with the REST route and raw marker from strict `-PyanoApiPrefix`; runtime drift in either prefix property or the reserved HTTP root is startup-fatal before plugin initialization. |

## 9. Verification gates

### Core and catalog

- exact-limit and limit-plus-one checks for descriptors, checks, text,
  series, one-GiB artifact snapshots, and defensive copies;
- manifest/name/provider correlation and at-most-one-kind rules;
- no legacy synthesis and no construction for denied/unselected providers;
- scanner/index/native service correlation for both new kinds;
- CLI deterministic table/JSON, stable exit codes, dependency/cycle/API-policy
  validation, symlink/change-under-scan defenses, safe paths, and proof that
  constructors/static initializers never run.
- offline/runtime policy parity for all ten contribution kinds, bounded
  dependency ranges, topology, allow/deny precedence, reserved selectors,
  duplicate conflicts, all digest modes, and fingerprint equality.

### Runtime

- atomically coherent immutable snapshots under concurrent samples and state
  transitions;
- catalog fingerprint unchanged while dynamic generations and health change;
- isolated auxiliary construction/sample failures;
- deadline includes queue time; queued cancellation; per-source no-overlap;
- last-good cache and stale behavior for timeout/failure/invalid snapshots;
- canonical per-check publication, `UNKNOWN` before first-good, aggregate
  consistency, and stale last-good retention;
- API-v1 instance/contribution lifecycle, health, failure, and stale-state
  coherence, while observed contributions without an instance remain valid;
- counter/timer monotonicity and generation reset;
- TCCL/interrupt restoration, process-fatal precedence, and callback lease
  retention until a timed-out callback really exits;
- seal/restart races, stale-generation rejection, reverse close order, and
  reentrant teardown rejection;
- deterministic catalog-order activation commit and generation-wide admission
  timeout of an unentered tail;
- host and per-bundle capacity isolation;
- no secret sentinel in a snapshot, diagnostic, or metric identity.

### Application and packaging

- privileged auth matrix and canonical-path alias rejection;
- pagination/order/response bounds and absent-bundle behavior;
- plugin health group reads cache only and cannot change readiness;
- exact Micrometer names/types/fixed `plugin`/`metric` tags/cardinality and
  proof a scrape invokes no plugin code or creates per-check series;
- metric-type replacement, root/custom API prefixes, redirect rejection, and
  the bounded dashboard refresh/display cap;
- thousands of paths/errors do not increase meter count;
- dashboard serving, no remote assets, no unsafe plugin interpolation, and no
  inventory before authentication;
- scaffold and conformance fixture cover both SPIs;
- packaged JVM and native smokes cover catalog, operations REST, health group,
  metrics, and dashboard assets.

### Final ADR-011 closure

- full repository tests;
- two nodes load the same catalog fingerprint and finalize the same app-chain
  roots while one node reports deliberately different local plugin health;
- query/domain APIs remain usable while one telemetry source hangs;
- independent architecture, security, operability, and maintainability review
  resolves every blocker/high finding.

## 11. Consequences and deferred work

Operators gain one bounded, consistent view of catalog identity and node-local
plugin behavior, plus metrics and a practical dashboard. Plugin authors gain
framework-neutral SPIs that survive JVM/native packaging without depending on
Quarkus.

The cost is another auxiliary execution lane, two typed provider contracts,
more catalog switch exhaustiveness, and explicit schema/cardinality rules. A
trusted plugin can still ignore interruption and occupy a bounded sampler
worker; only process isolation can remove that risk.

Deferred work includes signed deployment attestations, out-of-process
telemetry sources, histograms/exemplars, arbitrary labels, plugin-defined CLI
commands, write-capable operator actions, hot reload, schema/OpenAPI
contribution, and alternative effect transports. It also includes per-chain
lifecycle attribution for the six typed app-chain factory contributions and,
if policy behavior expands further, extraction of one pure catalog policy
kernel shared directly by offline inspection and runtime construction. The v1
parity matrix is the lower-risk guard against drift until that extraction can
be done without destabilizing the accepted catalog path.
