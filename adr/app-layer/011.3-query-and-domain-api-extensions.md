# ADR-011.3: Committed Queries and Constrained Domain APIs

## Status

Accepted — implemented and verified

## Date

2026-07-14

## Proposal-time context

ADR-011 establishes a manifested, typed plugin architecture. ADR-011.1 closes
the lifecycle kernel and ADR-011.2 implements the catalog, compatibility,
selection, provenance, class-loader and packaging boundary. The next missing
platform capability is a safe read surface for application state.

`AppStateMachine.query(path, params)` existed, but no runtime or HTTP path
called it. It also received no committed-state reader, so an implementation
could not reliably bind a response to the state root reported to the caller.
Applications therefore either exposed ad-hoc APIs outside the framework or
inspected raw keys, neither of which was a stable domain contract.

A second request is plugin-contributed domain APIs, such as a product-passport
view. Letting dropped JARs contribute arbitrary Quarkus/JAX-RS resources would
break route ownership, centralized authorization, native-image closure and
plugin lifecycle fencing. A host-owned dispatcher with typed contributions was
required instead.

This ADR resolves both concerns in two ordered gates:

1. a generic, root-fixed state-machine query;
2. a constrained, manifested `DomainApiProvider` using that query surface.

Cross-machine materialized views and arbitrary command APIs remain deferred.

## Goals

- Return every generic query with the exact committed height and state root it
  observed.
- Keep query and domain callbacks outside deterministic block execution.
- Bound admission, request size, response size and caller wait time.
- Keep plugin code under the ADR-011 lifecycle, TCCL and class-loader fences
  even after an HTTP timeout.
- Give domain APIs one manifest-owned namespace with deterministic collision
  checks and centralized authorization.
- Use the same descriptor/index model for JVM directory, packaged JVM and
  build-time native plugins.
- Preserve source and binary linkage for state machines implementing the
  existing two-argument query hook without presenting that legacy hook as a
  root-attested runtime query.

## Non-goals

- Query execution is not consensus and is not replicated.
- A response is not automatically a proof. The returned state root lets a
  domain-specific response include or reference proofs where appropriate.
- In-process Java plugins are trusted code, not a CPU, heap or syscall sandbox.
- v1 does not expose a Quarkus router, CDI container, JAX-RS registration,
  arbitrary response headers, WebSockets or streaming handlers to plugins.
- v1 does not support hot reload.
- Product-passport aggregation is not a framework primitive in this ADR.

## 1. Generic committed-state query

### 1.1 Public contract

The state-machine SPI gains a committed-context overload. The context exposes:

- the captured committed height;
- a root-fixed `AppStateReader`;
- the committed state root through that reader.

The existing `query(String, byte[])` method remains in the SPI for source and
binary linkage, but the bounded runtime and HTTP route never invoke it and
never attach a committed root to its result. The committed-context overload
defaults to `AppQueryException(UNSUPPORTED)`; a machine that implements only
the legacy hook is therefore reported as `UNSUPPORTED` by the runtime. This is
intentional: the legacy hook receives no root-fixed reader and could consult a
retained reader or mutable machine-local state, so delegating to it would make
the returned height/root claim untrue.

Machines that want the runtime query surface implement the committed-context
form and read application state only through the supplied reader. The legacy
method may remain useful to direct library callers, but such a call is outside
the root-attested contract of this ADR.

The gateway returns an immutable result:

```
AppQueryResult(
    chainId,
    stateMachineId,
    committedHeight,
    stateRoot,
    payload
)
```

All byte arrays are copied on construction and access.

### 1.2 Root-fixed capture

The runtime captures one committed view before invoking plugin code. Height and
root come from the same atomic RocksDB snapshot. Every `get(key)` is evaluated
against that captured root, never the moving tip. The reader expires when the
callback returns and rejects later use by plugin-created threads.

MPF nodes are immutable and currently retained; block-body retention does not
remove nodes required by a query admitted against a committed root. A future
MPF-node pruning policy must introduce reader pinning before it may prune a root
visible to an admitted query.

For an empty chain the committed view is height `0` and the canonical zero
root.

### 1.3 Concurrency and lifecycle

Each chain owns a dedicated bounded query lane:

- one active callback;
- queue capacity `16`;
- caller deadline `2 seconds`;
- request payload at most `64 KiB`;
- response payload at most `1 MiB`;
- normalized path at most `256` characters.

The two-second deadline starts when the dispatch call enters, before admission,
so all admission and queue time counts toward it. A request that expires or is
cancelled while queued is removed and
must never invoke plugin code later. A callback that has already started is
interrupted on timeout, but it remains the only kind of timed-out work that
retains generation and catalog callback leases until actual exit.

The lane never uses the app-chain engine's serialized transition executor, so
a slow query cannot delay block application, voting or finality. Query code may
run concurrently with `apply`; implementations must therefore avoid mutable
machine-local read models and use the supplied committed reader.

Java cannot safely kill an interrupt-ignoring callback. A deadline bounds the
caller's wait and requests interruption, but the worker continues to hold both
the app-chain generation-use lease and the catalog callback lease until plugin
code actually exits. Shutdown and class-loader close cannot race surviving
plugin code. A stuck trusted plugin can consequently delay restart or shutdown;
that is an explicit limitation of in-process plugins.

Queue exhaustion is reported as `BUSY`, deadline expiry as `TIMEOUT`, a stopped
chain as `UNAVAILABLE`, unsupported machine paths as `UNSUPPORTED`, invalid
input as `INVALID_REQUEST`, an excessive transport request as
`REQUEST_TOO_LARGE`, excessive output as `RESULT_TOO_LARGE`, and other plugin
failures as `FAILED`. The core exception codes are transport-neutral.

### 1.4 HTTP representation

The generic route is chain-scoped only. It is mounted below the artifact's
build-time API prefix (the example uses the default `/api/v1`):

```
POST /api/v1/app-chain/chains/{chainId}/query/{path}
Content-Type: application/json

{"paramsHex":"..."}
```

`paramsHex` is optional and defaults to the empty byte string. The response is:

```json
{
  "chainId": "supply-chain",
  "stateMachineId": "product-registry",
  "committedHeight": 42,
  "stateRoot": "...",
  "payloadHex": "..."
}
```

This is a strict transport envelope, not a consensus-canonical JSON encoding.
The adapter rejects unknown fields and, before JSON deserialization, caps the
raw entity at `132 KiB` and `paramsHex` at 131,072 ASCII characters. Those
transport limits complement the `64 KiB` decoded-parameter limit. Input hex is
canonical lowercase and output hex is always normalized lowercase;
clients must not hash the JSON representation as a protocol object.

The HTTP adapter receives the raw encoded path and accepts only the canonical
unreserved ASCII form. It rejects every percent escape, matrix parameter,
double-slash and trailing-slash alias—including encoded separators and dot
segments—rather than trusting the framework-normalized route. The same shared
query-path validator is used by the Java gateway and `DomainQueryService`.
The grammar is a relative path of 1-128 non-empty normalized ASCII URI
segments, at most 256 characters, with no NUL, control, percent, slash alias,
`.` or `..` segment. Non-canonical encoded aliases are rejected before plugin
invocation.

Transport mapping is deterministic:

| Query code | HTTP |
|---|---:|
| `INVALID_REQUEST` | 400 |
| `REQUEST_TOO_LARGE` | 413 |
| `UNSUPPORTED` | 404 |
| `BUSY` | 429 |
| `RESULT_TOO_LARGE` | 502 |
| `UNAVAILABLE` | 503 |
| `TIMEOUT` | 504 |
| `FAILED` | 500 with a redacted diagnostic |

## 2. Semantic authorization

HTTP method alone does not determine privilege: the generic query uses POST to
carry bounded parameters but is read-only. The app-chain request filter moves
to semantic classes:

- `READ`: status, proofs, history and generic queries;
- `SUBMIT`: app-message submission, still subject to topic scope;
- `PRIVILEGED`: admin, effect claim/report/requeue/cancel and privileged domain
  handlers;
- `INTERNAL`: never reachable from the public HTTP adapter.

`READ` and `SUBMIT` retain the existing opt-in app-chain authentication policy.
`PRIVILEGED` is fail-closed unless at least one unscoped full key is configured,
independently of broad READ/SUBMIT authentication. A statically declared
privileged app-chain operation then returns a host-owned `503`
configuration-unavailable response;
a privileged domain route is omitted from the public route table. Disabling
broad auth must never turn either form into an anonymous endpoint.

With keys configured and broad auth disabled, `READ` and `SUBMIT` remain
public while `PRIVILEGED` requires an unscoped full key. Setting
`yano.app-chain.api.auth.enabled=true` retains whole-surface authentication.

When app-chain API authentication is enabled:

- a missing or invalid key receives `401`;
- a topic-scoped key may invoke `READ` routes and may `SUBMIT` only to its
  configured topics;
- only an unscoped full key may invoke `PRIVILEGED` routes;
- `INTERNAL` routes are absent from public routing.

Existing `READ` and `SUBMIT` behavior is preserved; privileged behavior is
deliberately hardened. Authorization is attached to the matched operation
rather than inferred solely from its verb or URL spelling. Every host operation
and validated plugin route must have one semantic class; an absent or unknown
classification fails closed. `INTERNAL` is a reserved inventory value in v1:
it is not published through either the HTTP adapter or the public gateway and
therefore appears as `404`/absent at the HTTP boundary. A later trusted
host-only dispatch port requires a separate, explicit capability design.

## 3. Domain API contribution

### 3.1 Catalog identity

ADR-011.2 gains the manifested contribution kind `domain-api` backed by
`DomainApiProvider`.

Rules for schema v1:

- a bundle declares at most one `domain-api` contribution;
- its contribution name and `DomainApiProvider.id()` equal the containing
  bundle id;
- it is `PRIVILEGED_LOCAL` regardless of individual route access;
- there is no synthetic legacy form;
- allow/deny policy is applied before provider construction;
- provider, route and collision validation completes before publication.

The bundle id, not provider-supplied route data, owns the public namespace.

### 3.2 Host-owned routing

Public routes live below the same artifact prefix (shown with the default):

```
/api/v1/plugins/{bundleId}/...
```

The host owns the only JAX-RS resource. A provider creates one `DomainApi`
product containing immutable route metadata and a handler callback. Plugins do
not receive or return JAX-RS, CDI, Vert.x or Quarkus objects.

The route grammar is deliberately small:

- methods: `GET` and `POST`;
- relative templates only, with no leading/trailing slash;
- literal ASCII segments or a whole-segment `{parameter}`;
- at most 8 segments and 256 characters;
- no duplicate parameter names, wildcard/catch-all, dot segment, encoded slash,
  NUL or empty segment;
- at most 64 routes per domain API;
- `routeId` values are unique within a domain API;
- exact and structural method/template collisions are fatal before
  publication (`a/{x}` and `a/{y}` are the same structural route).

Literal routes outrank parameter routes. Remaining ambiguity at a segment is a
validation error rather than discovery-order routing.

The dispatcher uses the same raw canonical-path rule as the generic query
adapter: percent escapes, matrix parameters, double slashes and trailing-slash
aliases are rejected, and matching occurs only on the validated unreserved
ASCII form. The framework's possibly normalized path is never treated as proof
that the original request was canonical.

`INTERNAL` route metadata may be declared so manifests and route inventories
can evolve without renumbering the access model, but it is not dispatchable in
v1. The HTTP adapter and public gateway both reject it. This prevents a
publicly injectable gateway from becoming an accidental privilege-escalation
path; a future trusted host-only port must be designed separately.

### 3.3 Request and response boundary

A domain handler receives only immutable platform values:

- method and normalized relative path;
- validated path-parameter map;
- bounded query-parameter map;
- copied request bytes;
- no raw headers, cookies, security identity, servlet request or mutable host
  service.

The provider context exposes an immutable bundle-scoped configuration view and
a query-only service. The production v1 runtime always supplies an empty
configuration map: the catalog namespace may contain credentials and no typed
public-value or opaque secret-reference/capability contract exists yet. Tests
may construct a non-empty context, but plugins must not depend on that data in
production. A later non-empty view may contain ordinary non-secret values and
explicit opaque secret references only; direct credentials and secret material
are never copied into the context.
Configuration key names that may appear in diagnostics use a normalized safe
grammar or are escaped.
The query service can list hosted chain ids and call the generic committed
query; it cannot submit messages, administer a chain, claim effects or access
raw runtime internals.

v1 POST request bodies are capped at `64 KiB`; GET entities are rejected before
route lookup; responses are capped at `1 MiB`, with raw HTTP entity limits
applied before materialization. Response status is restricted to `200`, `400`,
`404`, `409`, `410`, or `422`; authentication, admission, redirect,
no-content/partial-content, and server statuses remain host-owned. Media type
is JSON or octet-stream, and plugins cannot contribute arbitrary headers.
Bodies remain opaque bytes. If a handler
selects JSON, the host parses and validates the bytes before advertising
`application/json`; invalid JSON fails the request. Bounded JSON schema and
OpenAPI contribution are explicitly deferred rather than inferred from the
media type. A handler may deliberately throw `DomainApiException` to select a
stable transport reason (including when translating `AppQueryException`); the
host preserves the code but always substitutes its own safe message. All other
handler failures become `FAILED`. Failures are redacted at the HTTP boundary
while retaining bundle/route attribution in host logs. Standard attributed
counters and health aggregation are supplied by ADR-011.4.

### 3.4 Execution and cleanup

Domain handlers use a host-owned bounded execution lane and the same callback
tracker/TCCL boundary as other typed products. Provider construction, route
enumeration, invocation and product close all run with the owning plugin class
loader as TCCL and restore the caller's TCCL even on failure.

Admission is isolated per bundle: each selected domain API has at most one
active callback, a queue capacity of 16 and a two-second caller deadline,
backed by a host-wide bounded worker/queue budget. One bundle cannot consume
another bundle's per-bundle admission capacity or an unbounded number of host
threads. The deadline starts when dispatch enters, before waiting for either
host-wide or per-bundle admission. Queued requests that
expire or are cancelled are removed and never invoke the handler; already
running callbacks follow the same interrupt-but-retain-leases rule as generic
queries. Shutdown rejects new work, cancels queued work, and drains only
callbacks that actually started.

The registry registers its terminal completion with the catalog environment.
Shutdown stops public admission, drains admitted callbacks, closes products in
reverse deterministic bundle order and only then permits provider/class-loader
close. Reentrant lifecycle teardown initiated from a handler is rejected.

Domain handlers are off-consensus. A `READ` designation controls authorization;
it does not make untrusted handler code safe or deterministic.

## 4. Native and packaged operation

The contribution participates in the same manifest, service-file correlation,
aggregate index, artifact digest and catalog fingerprint as ADR-011.2.

- Directory JARs work only on the JVM and only when policy selects the whole
  validated artifact.
- Packaged JVM distributions include provider classes and service metadata in
  the generated aggregate catalog.
- Native images include selected providers at build time and register their
  provider constructors through generated native metadata.
- A native binary cannot discover a newly dropped domain API JAR.

The host dispatcher itself is statically known, so native mode never performs
dynamic JAX-RS resource discovery.

## 5. Failure policy

Catalog/schema, selector, route-grammar, collision, provider-construction or
route-enumeration failures are startup-fatal for a selected domain API. No
partial route table is published.

At runtime, an individual handler failure fails that request without stopping
consensus or another plugin's routes. Attributed counters and repeated-failure
health policy are supplied by ADR-011.4.

Diagnostics exposed over HTTP never include plugin exception messages, stack
traces, configuration values or artifact paths. Operator logs retain exception
type and host-owned bundle/route identity; secret-bearing messages remain
redacted by default.

## 6. Verification gates

### Gate A — committed query

- default unsupported behavior and old-hook compatibility;
- proof that the runtime never invokes or root-attests the legacy hook;
- one atomic height/root under concurrent commits;
- multiple reads remain bound to the captured root;
- request/result/reader defensive-copy and expiry behavior;
- path, request and response bounds;
- busy, timeout and unavailable mappings;
- a timed-out callback retains generation and plugin lifetime until exit;
- stop/restart interaction and TCCL restoration on success/failure;
- REST encoding, status mapping and semantic READ authorization;
- raw entity limits, raw canonical-path validation, and shared path grammar.

### Gate B — domain API

- manifest/parser/index/service-file correlation and selector/bundle matching;
- no unmanifested legacy synthesis;
- denied providers are never constructed;
- route grammar, route budget, deterministic ordering and collision rejection;
- unique `routeId` values and structural-template collision rejection;
- READ/full-key/privileged/internal authorization behavior;
- fail-closed privileged behavior when a full key is absent;
- namespace, traversal, encoded-separator and response-bound attacks;
- handler TCCL, lifecycle drain, timeout and reverse close order;
- bundle-scoped configuration and query-only context;
- packaged JVM and native-image conformance.

### Parent ADR-011 closure (re-run after ADR-011.4)

- full repository tests;
- plugin scaffold/catalog verification;
- packaged JVM smoke;
- native build and plugin smoke;
- two-node app-chain query/domain regression;
- independent architecture, security and maintainability review.

Gate A, Gate B, the focused module suites, the complete runtime suite, the
author scaffold/conformance checks and the packaged JVM smoke passed before
this ADR was accepted. The parent two-node regression subsequently passed with
query and domain APIs remaining available while one node's deliberately hung
telemetry source was isolated. Both nodes finalized identical ordered-log and
KV roots and reported the same selected catalog fingerprint. ADR-011.4 records
the exact cluster and passed post-prefix packaging/full-suite evidence, so this
sub-ADR no longer carries a separate closure gate.

## 7. Decisions

| ID | Decision |
|---|---|
| D1 | A generic query is bound to one captured committed height/root. |
| D2 | Query and domain callbacks never run on the consensus transition executor. |
| D3 | A timeout bounds caller wait but cannot release lifecycle fences around still-running Java code. |
| D4 | The generic REST surface uses a strict JSON envelope with normalized lowercase output hex; the JSON itself is not a consensus-canonical encoding. |
| D5 | Authorization is semantic; a POST query is `READ`. |
| D6 | Domain APIs use one manifested provider per bundle and a host-owned dispatcher. |
| D7 | Domain contribution identity equals its containing bundle id and has no legacy adapter. |
| D8 | Plugins never receive arbitrary HTTP framework objects or register JAX-RS/CDI resources. |
| D9 | Domain contexts expose query-only chain access. |
| D10 | Product-passport aggregation remains an application/plugin concern until the base contracts prove stable. |
| D11 | `INTERNAL` is reserved inventory in v1 and is not dispatchable through the public gateway or HTTP. |
| D12 | The production v1 domain context exposes an empty configuration map until a typed safe-value/secret-reference contract exists. |

## 8. Consequences

Applications gain a stable way to publish domain reads without moving those
reads into consensus or exposing storage layout. Responses are auditable
against a named committed root, though not automatically proven.

The bounded lane protects consensus and ordinary request capacity, but cannot
provide hard CPU or heap isolation for trusted in-process code. Deployments
requiring that boundary must use an out-of-process API service and the generic
query endpoint.

The constrained dispatcher deliberately offers less convenience than arbitrary
JAX-RS plugins. In return, route ownership, authentication, native packaging,
failure attribution and shutdown remain platform-controlled and testable.

## 9. Deferred work

- signed artifact/deployment attestations;
- proof-carrying query response conventions;
- bounded JSON-schema and OpenAPI contribution metadata;
- materialized cross-machine read models and product-passport aggregation;
- streaming, WebSocket and subscription domain routes;
- out-of-process domain handler protocol;
- configurable per-plugin quotas after ADR-011.4 provides operational data;
- arbitrary CLI commands or write APIs.
